# 测试管理平台-项目唯一负责人后端接口与表结构改造方案

> 版本：v0.1（创建于 2026-04-05）
> 项目：Aisight（后端 TestPilot / 前端 TestFront）
> 适用对象：后端、前端、测试、联调
> 关联文档：
> - `测试管理平台-项目唯一负责人需求方案-20260405.md`
> - `测试管理平台-需求文档-20260308.md`
> - `测试管理平台-后端开发规范-20260405.md`
> 文档定位：将“项目唯一负责人”需求落到后端数据模型、迁移方案、接口契约与服务逻辑，作为研发实现基线

---

## 0. 设计目标

本方案解决以下问题：

1. 项目列表负责人必须返回真实数据，而不是前端 mock。
2. 每个项目任一时刻只允许存在 1 个负责人。
3. 创建项目与编辑项目都支持负责人指定或转交。
4. 历史多 `owner` 成员数据需要收敛为单一负责人。
5. 后端应成为负责人唯一口径来源，前端只做展示与操作发起。

---

## 1. 当前实现现状

基于当前代码，项目负责人相关现状如下：

1. `projects` 主表尚未维护 `owner_id` 字段，项目实体也未返回负责人摘要。
2. `/projects` 列表接口仅返回项目基础信息、成员数、用例数，不返回负责人信息。
3. 创建项目时仅会把创建人写入 `project_members.role = owner`。
4. 添加项目成员接口允许直接写入 `role=owner`，因此同一项目可能出现多个 owner。
5. 项目成员表中的 `owner` 更像“成员角色标签”，还不足以支撑“唯一负责人”业务语义。

因此，本次改造需要从“成员角色驱动负责人”升级为“项目主表驱动负责人”。

---

## 2. 总体设计原则

1. 项目负责人以 `projects.owner_id` 为唯一真源。
2. `project_members` 保留项目访问范围与协作关系语义，不再承担唯一负责人主语义。
3. 负责人变更入口统一收敛到项目创建、项目编辑，或后续独立的“转交负责人”接口。
4. 项目列表与项目详情如需展示负责人，均通过主表 join 用户表获取摘要信息。
5. 历史兼容期允许成员表仍保留 `owner/member` 值，但业务判定不再以 `project_members.role='owner'` 为准。

---

## 3. 数据模型改造

## 3.1 `projects` 表新增字段

建议为 `projects` 主表新增：

- `owner_id BIGINT UNSIGNED NOT NULL`

说明：

1. `owner_id` 指向当前项目唯一负责人用户 ID。
2. 建议建立普通索引 `idx_projects_owner_id`，便于后续按负责人筛选项目。
3. 是否增加外键约束可按当前项目统一策略决定；若现有库普遍不加外键，则保持一致，仅做应用层校验。

建议 SQL 草案：

```sql
ALTER TABLE projects
  ADD COLUMN owner_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '项目唯一负责人ID' AFTER avatar,
  ADD INDEX idx_projects_owner_id (owner_id);
```

---

## 3.2 `Project` 模型字段调整

`internal/model/models.go` 中的 `Project` 建议补充：

- 入库字段
  - `OwnerID uint`
- 虚拟返回字段
  - `OwnerName string`
  - `OwnerAvatar string`

建议返回口径：

```json
{
  "id": 1,
  "name": "AiSight Demo",
  "owner_id": 12,
  "owner_name": "李莎莎",
  "owner_avatar": "/uploads/avatars/u12.png"
}
```

---

## 3.3 `project_members` 表角色语义调整

本次改造后建议保留 `project_members.role` 字段，但调整语义边界：

1. `project_members` 的核心职责是维护“项目成员资格”。
2. 负责人判定不再通过扫描 `project_members.role='owner'` 得出。
3. 为兼容旧数据和最小改动，短期仍允许保留 `owner/member` 枚举值。
4. 中长期建议收敛为纯成员关系，或者将 `owner` 仅保留为冗余展示字段，不再作为业务主判断条件。

建议本期先采用“兼容保留”策略，避免一次性改动过大。

---

## 4. 历史数据迁移方案

## 4.1 迁移目标

将历史项目中可能存在的多个 `owner` 成员归并为：

- `projects.owner_id` 中唯一 1 个负责人
- `project_members` 中其余历史 owner 全部降级为普通成员

## 4.2 归并规则

按以下顺序确定每个项目的唯一负责人：

1. 优先取该项目最早创建的 `project_members.role='owner'` 成员；
2. 若无 owner 成员，则取该项目最早创建的成员；
3. 若项目连成员都没有，则回退为项目创建人；若当前无创建人字段可用，则需先补种子数据修复脚本。

## 4.3 迁移后修正动作

对每个项目执行：

1. 回填 `projects.owner_id`
2. 保留唯一负责人的成员记录
3. 其余 `role='owner'` 记录批量更新为 `role='member'`

建议迁移脚本分两步：

1. DDL：给 `projects` 增加 `owner_id`
2. DML：按归并规则回填并清洗历史数据

## 5. 仓储层改造

## 5.1 项目列表查询

当前 `List / ListByUserID / ListByUserIDIncludeArchived` 只查询项目基础字段。

改造后建议统一追加：

```sql
LEFT JOIN users owner_user ON owner_user.id = projects.owner_id
```

并返回：

- `projects.owner_id`
- `owner_user.name AS owner_name`
- `owner_user.avatar AS owner_avatar`

这样项目列表接口可以一次查齐负责人摘要，避免前端逐项目再拉成员列表推导负责人。

## 5.2 按项目查询

`FindByID` 建议补充支持负责人摘要预加载或新增 `FindDetailByID` 查询，用于项目详情、编辑回显等场景。

## 5.3 成员相关仓储接口

建议新增以下仓储能力：

- `FindOwnerMember(ctx, projectID)`：查询当前负责人对应成员记录（可选）
- `UpsertMember(ctx, projectID, userID)`：确保负责人用户在成员表中存在
- `DemoteOtherOwners(ctx, projectID, keepUserID)`：将除当前负责人外的历史 owner 记录降级为 member

若不想增加过多 repository 方法，也可在 service 内通过现有接口组合实现，但建议至少沉淀一个“负责人切换”原子操作。

## 6. Service 层改造

## 6.1 创建项目

当前 `ProjectService.Create` 入参仅有：

- `name`
- `description`
- `avatar`

建议改造为支持：

- `ownerID *uint`

业务规则：

1. 若 `ownerID` 为空，默认使用 `creatorID`
2. 校验目标负责人用户存在且为有效用户
3. 创建项目时直接写入 `projects.owner_id`
4. 确保负责人在 `project_members` 中存在
5. 若沿用成员角色冗余，则负责人成员可写为 `owner`，其他成员为 `member`

## 6.2 更新项目

当前 `ProjectService.Update` 仅支持名称、描述、头像。

建议增加：

- `ownerID *uint`

当 `ownerID` 有值且发生变化时：

1. 校验目标用户存在
2. 校验项目未归档
3. 自动补充目标用户的项目成员关系
4. 更新 `projects.owner_id`
5. 将原负责人降级为普通成员
6. 将新负责人在成员表中标记为 owner（若本期保留该冗余语义）
7. 清理其他遗留 owner，保证冗余数据也只剩 1 条

建议该逻辑放在事务中执行，确保项目主表与成员表状态一致。

## 6.3 移除成员

当前 `RemoveMember` 仅限制受保护全局角色不可移除。

改造后需增加：

1. 若 `userID == project.owner_id`，直接拒绝移除
2. 返回明确业务错误：“当前负责人不可直接移除，请先转交负责人”

## 6.4 添加成员

当前 `AddMember` 支持直接传 `role=owner|member`。

建议本期调整为：

1. 若沿用现有接口兼容前端，则后端临时拒绝 `role=owner`，提示通过项目编辑修改负责人
2. 或将 `role` 请求收窄为仅允许 `member`

推荐方案是第二种，即从接口契约层彻底禁止新增成员时直接指定 owner。

## 7. API 契约改造

## 7.1 `GET /projects`

返回对象增加：

- `owner_id`
- `owner_name`
- `owner_avatar`

示例：

```json
[
  {
    "id": 1,
    "name": "AiSight Demo",
    "description": "默认演示项目",
    "avatar": "/uploads/projects/demo.png",
    "status": "active",
    "owner_id": 12,
    "owner_name": "李莎莎",
    "owner_avatar": "/uploads/avatars/u12.png",
    "member_count": 8,
    "testcase_count": 126
  }
]
```

## 7.2 `POST /projects`

请求增加可选字段：

```json
{
  "name": "支付回归项目",
  "description": "支付链路自动化回归",
  "owner_id": 12
}
```

规则：

1. `owner_id` 可选
2. 未传则默认创建人为负责人
3. 响应返回中带回负责人摘要字段

## 7.3 `PUT /projects/{projectID}`

请求增加可选字段：

```json
{
  "name": "支付回归项目",
  "description": "支付链路自动化回归（一期）",
  "owner_id": 21
}
```

规则：

1. `owner_id` 不传则不变更负责人
2. `owner_id` 变化即视为负责人转交
3. 响应中返回最新负责人摘要字段

## 7.4 `POST /projects/{projectID}/members`

建议改造后仅允许：

```json
{
  "user_id": 21,
  "role": "member"
}
```

或进一步简化为直接去掉 `role` 字段，由后端固定落 `member`。

## 7.5 `DELETE /projects/{projectID}/members/{userID}`

新增规则：

- 若删除对象为当前负责人，返回 400/409，并附带明确错误信息

## 8. 错误处理建议

按后端开发规范，新增业务错误时应使用 `internal/service/errors.go` 中统一的 6 位数字错误码。

本需求建议新增以下业务错误语义：

1. 指定负责人不存在
2. 指定负责人已禁用
3. 当前负责人不可直接移除
4. 不允许通过成员接口直接设置负责人

具体码值由后端统一错误码表分配，本文不直接占号。

---

## 9. 前端联调影响

前端需要同步调整：

1. 项目列表页不再使用 mock 负责人
2. `Project` 类型增加 `owner_id / owner_name / owner_avatar`
3. 新建/编辑项目弹窗增加负责人选择器
4. 成员管理页对当前负责人显示“负责人”标识
5. 当前负责人行的“移除”按钮需禁用并提示先转交

---

## 10. 测试建议

建议至少覆盖以下测试：

1. 创建项目未传 `owner_id` 时，默认创建人为负责人
2. 创建项目传 `owner_id` 时，列表可正确返回负责人摘要
3. 修改负责人后，项目列表返回已切换
4. 修改负责人时，目标用户不在成员中也能自动补成员
5. 删除当前负责人时报错
6. 添加成员接口传 `role=owner` 被拒绝
7. 历史多 owner 数据迁移后，每个项目仅剩唯一负责人

---

## 11. 实施顺序建议

建议按以下顺序推进：

1. 新增需求文档与后端改造方案文档
2. 增加数据库字段与历史迁移脚本
3. 调整模型、仓储层、Service 层
4. 调整项目列表、创建、更新、成员删除接口
5. 增补 API 测试
6. 最后改前端项目管理页面

---

## 12. 当前结论

自本方案生效起，项目负责人后端实现应遵循以下基线：

1. `projects.owner_id` 是负责人唯一真源
2. `/projects` 必须返回负责人摘要字段
3. 项目创建和编辑是负责人唯一正式入口
4. 当前负责人不可直接删除
5. 历史多 owner 数据必须收敛为单一负责人
