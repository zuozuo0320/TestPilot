# 测试管理平台-测试智编模块 MySQL 表结构 SQL 草案

> 版本：v0.1（创建于 2026-03-28）
> 关联文档：`测试管理平台-测试智编模块数据与接口设计-20260328.md`
> 文档定位：用于后端建表、数据层开发、DDL 评审与数据库初始化

## 1. 说明

本文档基于当前已确认的数据模型，输出首期可落地的 MySQL DDL 草案。

建议前提：
- MySQL 版本：`8.0+`
- 字符集：`utf8mb4`
- 存储引擎：`InnoDB`
- 主键方案：平台统一 `bigint` ID（如雪花 ID）
- 时间字段：统一使用 `datetime(3)`

设计原则：
- 对项目、用例、用户等外部域对象先走逻辑关联，不强制建立物理外键
- 对测试智编模块内部对象，也优先使用逻辑约束 + 索引，减少跨表强耦合
- 大文件不入库，仅存元数据和存储地址
- 所有版本记录只新增不覆盖

## 2. 建表顺序建议

建议建表顺序：
1. `ai_script_task`
2. `ai_script_task_case_rel`
3. `ai_script_recording_session`
4. `ai_script_version`
5. `ai_script_validation`
6. `ai_script_trace`
7. `ai_script_evidence`
8. `ai_script_operation_log`

```sql
SET NAMES utf8mb4;

CREATE TABLE IF NOT EXISTS `ai_script_task` (

  `framework_type` VARCHAR(32) NOT NULL DEFAULT 'Playwright' COMMENT '输出框架类型，首期固定Playwright，脚本语言固定TypeScript',
  `latest_recording_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '最近一次录制会话ID',
  `current_script_version_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '当前脚本版本ID',
  `latest_validation_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '最近一次回放验证ID',
  `latest_validation_status` VARCHAR(32) DEFAULT NULL COMMENT '最近一次回放验证状态',
  `latest_execute_at` DATETIME(3) DEFAULT NULL COMMENT '最近一次执行生成时间',
  `latest_confirmed_at` DATETIME(3) DEFAULT NULL COMMENT '最近一次确认时间',
  `latest_confirmed_by` BIGINT UNSIGNED DEFAULT NULL COMMENT '最近一次确认人',
  `discard_reason` VARCHAR(255) DEFAULT NULL COMMENT '废弃原因',
  `created_by` BIGINT UNSIGNED NOT NULL COMMENT '创建人',
  `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
  `updated_by` BIGINT UNSIGNED NOT NULL COMMENT '更新人',
  `updated_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
  `deleted_flag` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '逻辑删除标识：0否1是',

  KEY `idx_ai_script_task_project_status` (`project_id`, `task_status`),

  `source_recording_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '来源录制会话ID',

  `is_current_flag` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否当前版本：0否1是',

  `updated_by` BIGINT UNSIGNED NOT NULL COMMENT '更新人',
  `updated_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
  `deleted_flag` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '逻辑删除标识：0否1是',

  KEY `idx_script_current` (`task_id`, `is_current_flag`),

  KEY `idx_script_based_on_version` (`based_on_version_id`)

  `id` BIGINT UNSIGNED NOT NULL COMMENT '主键',

CREATE TABLE IF NOT EXISTS `ai_script_validation` (

  `task_id` BIGINT UNSIGNED NOT NULL COMMENT '任务ID',
```

## 5. 建表补充说明

### 5.1 关于外键

当前草案未增加物理外键，原因是：
- `project_id`、`case_id`、`operator_id` 等属于外部业务域对象
- 平台若已有跨模块逻辑关联规范，建议沿用
- 执行链路里大量长任务、回调、补偿逻辑，物理外键可能增加迁移和联调成本

若后续数据库规范要求增加外键，可在内部表之间选择性增加，例如：
- `ai_script_version.task_id -> ai_script_task.id`
- `ai_script_validation.script_version_id -> ai_script_version.id`

### 5.2 关于 JSON 字段

以下字段建议保留为 `json`：
- `env_config_json`
- `locator_candidates_json`
- `raw_payload_json`
- `assertion_summary_json`
- `before_snapshot_json`
- `after_snapshot_json`

理由：
- 首期业务结构仍可能继续微调
- 便于保留执行上下文和扩展信息
- 后续若字段稳定，再考虑拆列优化查询

### 5.3 关于脚本正文与日志内容

建议：
- 脚本正文保留在 `ai_script_version.script_content`
- 首期脚本正文约定为 `Playwright TypeScript` 源码，导出文件后缀默认 `.spec.ts`
- 大日志原文尽量放对象存储或文件存储
- 数据库存储日志元数据和摘要，不建议无限制存放超大文本
