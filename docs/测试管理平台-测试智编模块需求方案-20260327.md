# 测试管理平台-测试智编模块需求方案（讨论稿）

> 版本：v0.1（创建于 2026-03-27）
> 项目：Aisight（后端 TestPilot / 前端 TestFront）
> 适用对象：产品、研发、测试、运维
> 文档定位：讨论稿，仅用于需求收敛，不代表最终开发方案

## 1. 模块目标

在测试管理平台内新增“测试智编”能力，用于把用户在真实页面上的业务场景探索过程，沉淀为可维护的自动化测试脚本。

本模块采用两阶段方案：
- `browser-use` 负责探索页面、执行场景、记录操作轨迹
- `Playwright` 负责承载首期最终测试脚本、稳定回放、接入测试体系；`Cypress` 后续按需求扩展

模块目标不是直接替代现有自动化测试框架，而是降低测试脚本的编写门槛，提升从业务场景到自动化脚本的转换效率。

## 2. 模块定位

### 2.1 定位说明

本模块是测试管理平台中的独立功能模块，用于：
- 基于真实页面交互过程生成自动化脚本初稿
- 将业务人员、测试人员的场景操作沉淀为可复用测试资产
- 为后续自动化执行、回归测试、CI 集成提供脚本来源

### 2.2 与现有能力关系

- 本模块不替代“用例管理”，而是与“用例管理”配套
- 测试用例描述的是业务验证目标；测试脚本描述的是自动化执行实现
- 测试智编任务必须关联至少 1 条测试用例，且支持同时关联多条测试用例
- 本模块生成的是“脚本草稿/脚本资产”，不是最终的自动化执行引擎

### 2.3 核心原则

1. 先采集真实操作轨迹，再生成脚本
2. 先生成脚本草稿，再进入人工确认或平台内二次编辑
3. 操作轨迹、脚本草稿、正式脚本三个层次需清晰分离
4. AI 生成的脚本默认不视为“已验证通过”，必须具备确认状态

## 3. 业务范围

### 3.1 In Scope（范围内）

1. 创建智能脚本生成任务
2. 配置目标站点、场景描述、账号信息、执行环境
3. 使用 browser-use 执行页面探索与场景操作
4. 记录结构化操作轨迹
5. 首期基于操作轨迹生成 Playwright 脚本；Cypress 后续按需求扩展
6. 首期支持脚本生成后的回放验证，且仅支持手动触发
7. 将脚本与一条或多条测试用例建立关联
8. 保存脚本版本、生成时间、生成人、生成方式
9. 预览脚本、复制脚本、导出脚本
10. 记录生成日志、失败原因、关键截图与页面证据

### 3.2 Out of Scope（暂不纳入）

1. 平台内完整自动修复引擎
2. 平台内完整可视化脚本编辑器
3. 全量替代人工编写 Playwright/Cypress 测试
4. 所有网站场景下 100% 稳定生成可执行脚本
5. 自动生成高质量业务断言闭环而无需人工确认

## 4. 核心业务模型

### 4.1 对象定义

| 对象 | 说明 |
|------|------|
| 智能脚本生成任务 | 一次场景探索与脚本生成过程 |
| 操作轨迹 | browser-use 执行过程中的结构化动作记录 |
| 脚本草稿 | 基于操作轨迹生成的 Playwright 代码初稿（首期）；Cypress 为后续扩展方向 |
| 正式脚本 | 经人工确认后沉淀的脚本版本 |
| 生成证据 | 包含截图、页面信息、日志、错误信息、关键 DOM/定位器候选 |

### 4.2 生命周期建议

智能脚本生成任务建议包含以下状态：
- `待配置`
- `待执行`
- `执行中`
- `生成成功`
- `生成失败`
- `待确认`
- `待重新验证`
- `已确认`
- `已废弃`

脚本草稿建议包含以下状态：
- `草稿`
- `待确认`
- `已确认`
- `已废弃`

## 5. 主流程

### 5.1 标准流程

1. 用户选择项目与目标测试用例（至少 1 条，可多选）
2. 用户创建“智能脚本生成任务”
3. 用户填写场景描述、起始地址、账号信息、附加说明
4. 平台调用 browser-use 执行场景探索
5. 系统记录结构化操作轨迹与执行证据
6. 系统根据轨迹生成 Playwright 脚本草稿（首期唯一输出）
7. 用户查看脚本草稿、轨迹、截图、日志
8. 用户可手动点击“验证”，触发脚本生成后的回放验证
9. 若回放验证失败，用户可手动调试脚本并重新触发验证
10. 回放验证通过后，用户确认脚本草稿，沉淀为正式脚本版本
11. 正式脚本与已关联测试用例保持关联，用于后续维护与执行

### 5.2 失败流程

1. 若探索阶段失败，任务状态置为 `生成失败`
2. 系统保留失败日志、错误原因、最后页面截图、已采集轨迹
3. 用户可基于原任务重新生成，或复制配置后新建任务
4. 若回放验证失败，脚本不可确认，用户可手动调试脚本并重新验证

## 6. 关键业务规则

### 6.1 总体规则

1. 模块输出的核心结果是“自动化脚本草稿”，不是直接保证可用于生产回归的最终脚本
2. 脚本生成必须基于结构化操作轨迹，不允许只基于自然语言场景描述直接落最终脚本
3. 一次生成任务必须绑定所属项目
4. 生成任务必须关联至少 1 条测试用例，且支持关联多条测试用例
5. 同一测试用例允许存在多个脚本版本或多个框架脚本

### 6.2 探索与轨迹采集规则

1. browser-use 负责页面探索、执行场景、记录操作轨迹
2. 操作轨迹至少应记录：
   - 页面访问 URL
   - 动作类型（跳转、点击、输入、选择、上传、滚动、等待、断言候选等）
   - 目标元素信息
   - 候选定位器信息
   - 动作输入值
   - 动作执行结果
   - 时间顺序
3. 关键步骤需保留截图或页面证据
4. 失败步骤需保留错误信息与失败前最后状态
5. 敏感信息在轨迹、日志、脚本展示中应支持脱敏

### 6.3 脚本生成规则

1. 首期默认且唯一输出 `Playwright`；`Cypress` 暂不支持，后续按需求扩展
2. 首期无需在任务创建时选择目标输出框架，统一输出 `Playwright`
3. 后续若扩展多框架输出，再开放目标输出框架选择能力
4. 生成脚本应尽量输出可维护代码，而非简单动作堆叠
5. 生成脚本时应优先使用更稳定的定位方式，候选优先级建议为：
   - `data-testid`
   - 语义化 role/label
   - 稳定文本
   - CSS/XPath 兜底
6. 若生成过程中无法获得可靠定位器，应在脚本或结果中明确标注风险
7. 生成脚本应包含必要注释，标识关键业务步骤与人工确认点
8. 首期纳入断言生成能力，生成结果应包含基础断言内容或断言建议
9. 首期支持的断言类型建议包括：
   - 页面到达类断言：URL、页面标题、主标题、面包屑等
   - 元素可见性断言：元素出现、消失、显示、隐藏
   - 元素状态断言：可点击、禁用、选中、勾选、展开、折叠等
   - 文本结果断言：成功提示、失败提示、状态文本、详情文本等，首期优先支持“包含匹配”
   - 表单结果断言：输入值、已选值、回填值、提交后展示值等
   - 列表/表格结果断言：存在某条数据、某状态标签、某关键字段
10. 首期每条生成脚本至少应包含 1 个最终结果断言，不允许只有操作步骤而没有校验结果
11. 首期允许在关键步骤插入少量过程断言，建议控制在 1-3 个，避免脚本过于脆弱
12. 高置信度断言可直接生成到脚本中；低置信度断言应以“断言建议 / 待确认项”形式输出
13. 断言生成优先依据建议为：测试用例预期结果 > 页面明确反馈 > 稳定 UI 状态 > 弱文本线索
14. 首期暂不纳入数据库断言、接口级断言、视觉对比断言、性能耗时断言、跨系统一致性断言、复杂动态统计断言

### 6.4 测试脚本定位规则

1. `Playwright` 负责承载首期最终测试脚本、稳定回放、接入测试体系；`Cypress` 为后续扩展方向
2. 平台内生成的脚本默认归类为“脚本草稿”
3. 脚本草稿确认后，方可标记为“正式脚本”
4. 正式脚本应支持版本化管理
5. 脚本修改不影响原始操作轨迹留存
6. 首期允许在平台内对脚本草稿进行有限编辑，但不提供完整在线编辑器能力
7. 平台内编辑后的脚本必须保存为新版本，不覆盖原始 AI 生成稿
8. 已确认脚本若再次编辑，应生成新版本并回到 `待确认`、`待重新验证` 状态

### 6.5 与测试用例的关联规则

1. 一个测试智编任务必须关联至少 1 条测试用例，且支持同时关联多条测试用例
2. 一条测试用例可关联多个测试智编任务与多个脚本版本
3. 每个脚本需明确：
   - 所属项目
   - 关联用例
   - 输出框架
   - 版本号
   - 生成来源（AI 生成 / 人工修改 / 混合）
   - 生成人
   - 生成时间
4. 删除测试用例时，默认不物理删除其关联脚本，脚本进入“孤立脚本待处理”状态，具体策略待后续确认

### 6.6 人工确认规则

1. AI 生成结果必须支持人工确认
2. 未确认脚本不得默认纳入正式回归资产
3. 人工确认前，脚本状态为 `待确认`
4. 人工确认后，脚本状态变为 `已确认`
5. 若脚本确认后再次编辑，应形成新版本，不覆盖旧版本
6. `admin` / `manager` / `reviewer` 可执行脚本确认；`tester` 可提交脚本待确认，但不可执行最终确认
7. 回放验证未通过的脚本不可执行最终确认


### 6.7 回放验证规则

1. 首期必须支持脚本生成后的回放验证
2. 回放验证仅支持用户手动点击“验证”触发，首期不自动触发
3. 回放验证结果需展示失败步骤、报错信息、截图、日志、断言结果
4. 回放验证失败后，允许用户手动调试脚本并再次触发验证
5. 仅回放验证通过的脚本允许进入确认
6. 以下场景必须重新触发回放验证：脚本重新生成、脚本人工编辑并保存为新版本、步骤/定位器/断言/等待逻辑等关键内容变更、起始地址/环境/账号/浏览器等执行配置变更、关联用例的前置条件/步骤/预期结果发生且影响当前脚本的变更
7. 以下场景无需重新触发回放验证：仅修改任务名称、脚本名称、备注、标签，仅执行预览、复制、导出，仅修改不影响脚本内容和执行配置的展示类信息
8. 命中必须重新触发回放验证的场景时，原验证结果立即失效，状态变为 `待重新验证`
9. 已确认脚本若命中上述变更场景，应生成新版本并回到 `待确认`、`待重新验证` 状态

## 9. 角色与权限建议（讨论版）

建议权限边界如下：
- `admin`：可查看、创建、删除、确认所有脚本生成任务
- `manager`：可查看、创建、确认本项目脚本生成任务
- `tester`：可查看、创建本项目脚本生成任务，可提交脚本待确认，但不可执行最终确认
- `reviewer`：可查看脚本结果，可执行脚本确认
- `developer`：默认只读，可按后续策略决定是否允许导出脚本
- `readonly`：仅查看，不允许创建或确认

> 上述权限为建议口径，需后续与你继续确认

## 10. 非功能要求建议

1. 任务执行过程需具备超时控制
2. 日志、截图、轨迹需可追溯
3. 敏感信息需脱敏存储与展示
4. 失败任务需可复盘
5. 脚本生成结果需支持复制与导出
6. 后续若接入执行能力，应与当前需求解耦设计

## 11. 当前已确认结论

1. 本能力作为测试管理平台中的独立模块/功能点集成
2. 总体技术路线采用两阶段方案：
   - `browser-use` 负责探索页面、执行场景、记录操作轨迹
   - `Playwright` 负责承载首期最终测试脚本、稳定回放、接入测试体系；`Cypress` 后续按需求扩展
3. 已确认：测试智编必须绑定测试用例，且支持同时绑定多条测试用例
4. 已确认：首期允许平台内对脚本草稿进行有限编辑，但不提供完整在线编辑器能力
5. 已确认：`admin` / `manager` / `reviewer` 可确认脚本，`tester` 仅可提交待确认
6. 已确认：首期默认且唯一输出 Playwright，Cypress 暂不支持，后续按需求扩展
