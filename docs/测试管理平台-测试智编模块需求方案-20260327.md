# 测试管理平台-测试智编模块需求方案（讨论稿）

> 版本：v0.1（创建于 2026-03-27）
> 项目：Aisight（后端 TestPilot / 前端 TestFront）
> 适用对象：产品、研发、测试、运维
> 文档定位：需求方案基线文档；其中“2026-04-01 当前实现对齐说明”用于同步当前代码已落地逻辑
> 2026-04-01 实现对齐说明：当前代码已进入“多项目 Playwright 工程化 V1”阶段。若本文旧表述与当前实现冲突，以本说明及 `playwright-architecture-design.md` 为准。

## 0. 当前实现对齐说明（2026-04-01）

当前测试智编模块已落地并需要统一对齐的实现口径如下：

- 主路径是 `录制增强模式`，执行链路为：`Playwright 原始录制稿 -> step_model -> LLM 工程化重构 -> 项目级 Playwright 验证`
- 平台运行是主路径，但生成结果必须具备脱离平台直接执行的能力；当前执行载体为项目工作区下的标准 `Playwright TypeScript` 工程代码
- 系统已升级为多项目版本，所有生成、验证、导出都必须先绑定 `project_id / project_key`
- `LLM` 在录制增强模式下只做 `POM` 结构化组织，不允许改写、简化、替换原始录制稿中的 locator
- 对于录制稿中的复杂链式入口 locator，必须显式保留，不允许泛化成 `openTaskListByIndex(index)`、`viewTaskButtons.nth(index)` 这一类通用 helper
- 若录制稿中没有出现 `waitForURL` 或 URL 断言，则 `LLM` 不允许凭空发明固定 URL ready 条件
- `spec` 层只允许表达业务意图，不允许直接写 locator，也不允许引用未在目标 Page Object 中定义的成员
- `base.fixture.ts` 由程序根据 `page-registry.json` 全量重建，`LLM` 不直接编辑 fixture
- 共享页面对象当前固定白名单为：`NavigationPage / DialogPage / ToastPage / LoginPage`
- 所有生成类、action 方法以及关键业务逻辑都必须补充中文注释；这是硬约束，不是建议项
- Windows 环境下 AST 合并链路必须固定 `UTF-8`，避免中文 locator 与注释在增量合并时变成乱码

## 1. 模块目标

在测试管理平台内新增“测试智编”能力，用于把用户在真实页面上的业务场景探索过程，沉淀为可维护的自动化测试脚本。

本模块首期采用“双模式生成 + 统一验证确认”方案：
- `录制增强模式` 作为默认主模式：用户通过 `Playwright` 录制获得基础元素定位与测试执行逻辑，系统沉淀原始录制稿、结构化步骤模型，再由 AI 按团队标准框架重构为最终项目级脚本资产（`spec / pages / registry / fixture`）
- `AI直生模式` 作为辅助模式：基于自然语言场景描述，由 `browser-use + LLM` 直接生成脚本草稿
- `Playwright` 负责承载首期最终测试脚本、稳定回放、接入测试体系；首期生成代码语言固定为 `TypeScript`，`Cypress` 后续按需求扩展

模块目标不是直接替代现有自动化测试框架，而是降低测试脚本的编写门槛，提升从业务场景到自动化脚本的转换效率。

## 2. 模块定位

### 2.1 定位说明

本模块是测试管理平台中的独立功能模块，用于：
- 基于真实页面交互过程生成自动化脚本初稿
- 将业务人员、测试人员的场景操作沉淀为可复用测试资产
- 为后续自动化执行、回归测试、CI 集成提供脚本来源

### 2.2 与现有能力关系

- 本模块不替代“用例管理”，而是与“用例管理”配套
- 测试用例描述的是业务验证目标；测试脚本描述的是自动化执行实现
- 测试智编任务必须关联至少 1 条测试用例，且支持同时关联多条测试用例
- 本模块生成的是“脚本草稿/脚本资产”，不是最终的自动化执行引擎

### 2.3 核心原则

1. 先采集真实操作轨迹，再生成脚本
2. 先生成脚本草稿，再进入人工确认或平台内二次编辑
3. 操作轨迹、脚本草稿、正式脚本三个层次需清晰分离
4. AI 生成的脚本默认不视为“已验证通过”，必须具备确认状态
5. 首期默认主路径为 `录制增强模式`，`AI直生模式` 作为辅助入口保留

## 3. 业务范围

### 3.1 In Scope（范围内）

1. 创建智能脚本生成任务
2. 配置目标站点、场景描述、账号信息、执行环境
3. 支持选择生成模式：`录制增强模式（默认）` / `AI直生模式`
4. `录制增强模式` 支持使用 `Playwright` 录制基础操作、定位信息与执行逻辑
5. `AI直生模式` 支持使用 `browser-use` 执行页面探索与场景操作
6. 记录结构化操作轨迹、原始录制稿、结构化步骤模型
7. 首期基于轨迹或录制结果生成 `Playwright TypeScript` 脚本；`Cypress` 后续按需求扩展
8. 首期支持脚本生成后的回放验证，且仅支持手动触发
9. 将脚本与一条或多条测试用例建立关联
10. 保存脚本版本、生成时间、生成人、生成方式
11. 预览脚本、复制脚本、导出脚本
12. 记录生成日志、失败原因、关键截图与页面证据
13. 支持多项目独立工作区、独立 registry、独立 auth_states 与独立 Playwright 配置
14. 平台内执行为主路径，同时保留项目级代码脱离平台直接执行的能力

### 3.2 Out of Scope（暂不纳入）

1. 平台内完整自动修复引擎
2. 平台内完整可视化脚本编辑器
3. 全量替代人工编写 Playwright/Cypress 测试
4. 所有网站场景下 100% 稳定生成可执行脚本
5. 自动生成高质量业务断言闭环而无需人工确认
6. 账号口令、验证码、MFA 等复杂认证场景的边界收敛与专项能力，本期暂不处理，后续单独讨论

## 4. 核心业务模型

### 4.1 对象定义

| 对象 | 说明 |
|------|------|
| 智能脚本生成任务 | 一次场景探索与脚本生成过程 |
| 操作轨迹 | browser-use 执行过程中的结构化动作记录 |
| 原始录制稿 | `Playwright` 录制得到的基础脚本、locator 与执行步骤原稿 |
| 结构化步骤模型 | 从录制稿或探索轨迹中提取出的稳定步骤序列、定位信息、断言线索 |
| 脚本草稿 | 基于操作轨迹生成的 `Playwright TypeScript` 代码初稿（首期）；`Cypress` 为后续扩展方向 |
| 正式脚本 | 经人工确认后沉淀的脚本版本 |
| 生成证据 | 包含截图、页面信息、日志、错误信息、关键 DOM/定位器候选 |

### 4.2 生命周期建议

智能脚本生成任务建议包含以下状态：
- `待配置`
- `待执行`
- `执行中`
- `生成成功`
- `生成失败`
- `待确认`
- `待重新验证`
- `已确认`
- `已废弃`

脚本草稿建议包含以下状态：
- `草稿`
- `待确认`
- `待重新验证`
- `已确认`
- `已废弃`

## 5. 主流程

### 5.1 默认主流程：录制增强模式

1. 用户选择项目与目标测试用例（至少 1 条，可多选）
2. 用户创建“智能脚本生成任务”
3. 用户填写场景描述、起始地址、账号信息、附加说明，生成模式默认选择 `录制增强模式`
4. 平台启动 `Playwright` 录制，引导用户完成真实业务操作
5. 系统保存原始录制稿、关键 locator、操作步骤和执行证据
6. 系统将录制结果转换为结构化步骤模型
7. AI 基于结构化步骤模型，按团队标准框架重构 `Playwright TypeScript` 脚本草稿
   当前实现补充：录制增强模式下，AI 实际输出的是多文件工程化结果，包含 `spec`、Page Object 增量更新建议、registry 更新建议和人工审核提示
8. 用户查看标准化脚本草稿、原始录制稿、轨迹、截图、日志
9. 用户可手动点击“验证”，触发脚本生成后的回放验证
10. 若回放验证失败，用户可手动调试脚本并重新触发验证
11. 回放验证通过后，用户确认脚本草稿，沉淀为正式脚本版本
12. 正式脚本与已关联测试用例保持关联，用于后续维护与执行

### 5.2 辅助流程：AI直生模式

1. 用户选择项目与目标测试用例（至少 1 条，可多选）
2. 用户创建“智能脚本生成任务”
3. 用户在生成模式中切换为 `AI直生模式`
4. 用户填写场景描述、起始地址、账号信息、附加说明
5. 平台调用 `browser-use + LLM` 执行页面探索
6. 系统记录结构化操作轨迹与执行证据
7. 系统根据轨迹直接生成 `Playwright TypeScript` 脚本草稿
8. 后续验证、确认、入库流程与录制增强模式保持一致

### 5.3 失败流程

1. 若录制、探索、AI 重构任一阶段失败，任务状态置为 `生成失败`
2. 系统保留失败日志、错误原因、最后页面截图、已采集轨迹或录制稿
3. 用户可基于原任务重新生成，或复制配置后新建任务
4. 若回放验证失败，脚本不可确认，用户可手动调试脚本并重新验证

## 6. 关键业务规则

### 6.1 总体规则

1. 模块输出的核心结果是“自动化脚本草稿”，不是直接保证可用于生产回归的最终脚本
2. 脚本生成必须基于结构化操作轨迹或结构化步骤模型，不允许 AI 在无可追溯输入的情况下直接落最终脚本
3. 一次生成任务必须绑定所属项目
4. 生成任务必须关联至少 1 条测试用例，且支持关联多条测试用例
5. 同一测试用例允许存在多个脚本版本或多个框架脚本
6. `录制增强模式` 为首期默认模式，`AI直生模式` 为补充模式

### 6.2 探索与轨迹采集规则

1. browser-use 负责页面探索、执行场景、记录操作轨迹
2. 操作轨迹至少应记录：
   - 页面访问 URL
   - 动作类型（跳转、点击、输入、选择、上传、滚动、等待、断言候选等）
   - 目标元素信息
   - 候选定位器信息
   - 动作输入值
   - 动作执行结果
   - 时间顺序
3. 关键步骤需保留截图或页面证据
4. 失败步骤需保留错误信息与失败前最后状态
5. 敏感信息在轨迹、日志、脚本展示中应支持脱敏

### 6.3 录制增强规则

1. 首期默认生成模式为 `录制增强模式`
2. `Playwright` 录制负责生成基础元素定位、操作步骤与执行逻辑原稿
3. 系统需保留原始录制稿，作为后续脚本重构、审计与问题回溯依据
4. 系统需从录制稿中提取结构化步骤模型，至少包含：步骤顺序、动作类型、目标 locator、输入值、页面 URL、断言线索
5. AI 对录制结果的处理定位为“按标准框架重构”，不允许擅自改变关键步骤顺序和核心 locator
6. 原始录制稿中的 locator 是录制增强模式下的唯一定位事实来源；AI 只允许做结构化组织，不允许按经验优化 locator
7. 若录制稿中存在复杂链式入口 locator，必须显式保留完整链路，不允许改造成泛化索引 helper
8. 若 AI 重构失败，允许用户直接查看原始录制稿，并支持后续再次重构
9. 录制增强模式下，回放验证与确认规则与其他模式完全一致

### 6.4 脚本生成规则

1. 首期默认且唯一输出 `Playwright TypeScript`；`Cypress` 暂不支持，后续按需求扩展
2. 首期无需在任务创建时选择目标输出框架或语言，统一输出 `Playwright TypeScript`
3. 后续若扩展多框架输出，再开放目标输出框架选择能力
4. 录制增强模式下，生成结果应输出为可维护的项目级工程代码，而不是单文件动作堆叠
5. 录制增强模式下必须严格保留原始录制稿中的 locator；AI 不允许改写、简化、替换或补发明新的 locator
6. 若生成过程中无法确认页面归属、复杂入口 locator 归属或方法签名，应进入 `manual_review`，而不是自行猜测改写
7. `spec` 层不得直接写 locator，只允许调用 fixture 与 Page Object 方法
8. `Page Object` 层负责 locator 封装、业务 action、必要等待与断言；共享层仅允许固定白名单页面对象
9. `base.fixture.ts` 由程序根据 registry 全量重建，不允许 LLM 直接自由修改
10. 生成脚本必须包含中文类注释、中文 action 方法注释，以及关键逻辑中文说明
11. 首期纳入断言生成能力，但仅允许补充高置信度断言；低置信度内容应作为风险提示或人工确认项输出
12. 首期支持的断言类型建议包括：
   - 页面到达类断言：URL、页面标题、主标题、面包屑等
   - 元素可见性断言：元素出现、消失、显示、隐藏
   - 元素状态断言：可点击、禁用、选中、勾选、展开、折叠等
   - 文本结果断言：成功提示、失败提示、状态文本、详情文本等，首期优先支持“包含匹配”
   - 表单结果断言：输入值、已选值、回填值、提交后展示值等
   - 列表/表格结果断言：存在某条数据、某状态标签、某关键字段
13. 只有录制稿中明确出现 `waitForURL` 或 URL 断言时，才允许保留 URL 级等待；否则应优先使用业务控件判定页面 ready
14. 首期允许在关键步骤插入少量过程断言，避免脚本过于脆弱
15. 断言生成优先依据建议为：测试用例预期结果 > 页面明确反馈 > 稳定 UI 状态 > 弱文本线索
16. 首期暂不纳入数据库断言、接口级断言、视觉对比断言、性能耗时断言、跨系统一致性断言、复杂动态统计断言

### 6.5 测试脚本定位规则

1. `Playwright` 负责承载首期最终测试脚本、稳定回放、接入测试体系；首期脚本语言固定为 `TypeScript`；`Cypress` 为后续扩展方向
2. 平台内运行是主路径，但生成结果必须支持在项目工作区中独立执行
3. 当前正式产物是多文件项目代码：`tests/**/*.spec.ts`、`pages/**/*.ts`、`fixtures/*.ts`、`registry/page-registry.json`
4. 平台内生成的脚本默认归类为“脚本草稿”
5. 脚本草稿确认后，方可标记为“正式脚本”
6. 正式脚本应支持版本化管理
7. 脚本修改不影响原始操作轨迹留存
8. 首期允许在平台内对脚本草稿进行有限编辑，但不提供完整在线编辑器能力
9. 平台内编辑后的脚本必须保存为新版本，不覆盖原始生成稿
10. 已确认脚本若再次编辑，应生成新版本并回到 `待确认`、`待重新验证` 状态

### 6.6 与测试用例的关联规则

1. 一个测试智编任务必须关联至少 1 条测试用例，且支持同时关联多条测试用例
2. 一条测试用例可关联多个测试智编任务与多个脚本版本
3. 每个脚本需明确：
   - 所属项目
   - 关联用例
   - 输出框架
   - 版本号
   - 生成来源（AI 直生 / Playwright 录制 / 录制增强重构 / 人工修改 / 混合）
   - 生成人
   - 生成时间
4. 删除测试用例时，默认不物理删除其关联脚本，脚本进入“孤立脚本待处理”状态，具体策略待后续确认

### 6.7 人工确认规则

1. 系统生成结果必须支持人工确认
2. 未确认脚本不得默认纳入正式回归资产
3. 人工确认前，脚本状态为 `待确认`
4. 人工确认后，脚本状态变为 `已确认`
5. 若脚本确认后再次编辑，应形成新版本，不覆盖旧版本
6. `admin` / `manager` / `reviewer` 可执行脚本确认；`tester` 可提交脚本待确认，但不可执行最终确认
7. 回放验证未通过的脚本不可执行最终确认


### 6.8 回放验证规则

1. 首期必须支持脚本生成后的回放验证
2. 回放验证仅支持用户手动点击“验证”触发，首期不自动触发
3. 回放验证结果需展示失败步骤、报错信息、截图、日志、断言结果
4. 回放验证失败后，允许用户手动调试脚本并再次触发验证
5. 仅回放验证通过的脚本允许进入确认
6. 以下场景必须重新触发回放验证：脚本重新生成、脚本人工编辑并保存为新版本、步骤/定位器/断言/等待逻辑等关键内容变更、起始地址/环境/账号/浏览器等执行配置变更、关联用例的前置条件/步骤/预期结果发生且影响当前脚本的变更
7. 以下场景无需重新触发回放验证：仅修改任务名称、脚本名称、备注、标签，仅执行预览、复制、导出，仅修改不影响脚本内容和执行配置的展示类信息
8. 命中必须重新触发回放验证的场景时，原验证结果立即失效，状态变为 `待重新验证`
9. 已确认脚本若命中上述变更场景，应生成新版本并回到 `待确认`、`待重新验证` 状态

### 6.9 任务列表批量治理规则

1. 任务列表必须支持`本页全选`、`跨页保留勾选`、`按当前筛选结果全选`
2. 批量选择模型统一采用两种模式：
   - `IDS`：前端显式传递已勾选任务 `task_ids`
   - `FILTER_ALL`：前端传递当前筛选快照 `filter_snapshot`，服务端按快照解析命中集合，并排除 `excluded_task_ids`
3. 用户在“按筛选结果全选”模式下取消单条勾选时，不应退回普通勾选模式，而应将该任务写入 `excluded_task_ids`
4. 用户切换筛选条件、重置筛选条件或主动点击“清空勾选”时，必须清空当前任务选择状态，避免误操作历史选择集合
5. 批量废弃与批量删除必须复用统一选择模型与统一结果结构，不允许前后端分别维护两套批量协议
6. 批量废弃仅允许 `admin / manager` 执行，且必须填写废弃原因
7. `RUNNING` 状态任务不允许废弃；已废弃任务不允许重复废弃；批量废弃需按任务逐条校验并返回跳过原因统计
8. 批量删除仅允许删除 `DISCARDED` 状态任务；命中其他状态时必须按“跳过”处理，而不是整批失败
9. 批量接口必须支持“部分成功”，返回结果至少包含：`matched / success / skipped / failed / skip_reasons / failed_reasons`
10. 任务列表页面需明确展示当前已选数量、是否处于“按筛选结果全选”模式，以及当前排除数量

## 9. 角色与权限建议（讨论版）

建议权限边界如下：
- `admin`：可查看、创建、删除、确认所有脚本生成任务
- `manager`：可查看、创建、确认本项目脚本生成任务
- `tester`：可查看、创建本项目脚本生成任务，可提交脚本待确认，但不可执行最终确认
- `reviewer`：可查看脚本结果，可执行脚本确认
- `developer`：默认只读，不允许导出脚本
- `readonly`：仅查看，不允许创建或确认

> 上述权限已在权限矩阵文档中完成定稿，后续如需调整需单独评审

## 11. 当前已确认结论

1. 本能力作为测试管理平台中的独立模块/功能点集成
2. 总体技术路线采用“双模式生成 + 统一验证确认”方案：
   - 默认主路径：`Playwright` 录制生成原始录制稿与步骤逻辑，AI 按团队标准框架重构为最终脚本
   - 辅助路径：`browser-use + LLM` 根据自然语言场景直接生成脚本草稿
   - `Playwright` 负责承载首期最终测试脚本、稳定回放、接入测试体系；首期代码语言固定 `TypeScript`；`Cypress` 后续按需求扩展
3. 已确认：测试智编必须绑定测试用例，且支持同时绑定多条测试用例
4. 已确认：首期允许平台内对脚本草稿进行有限编辑，但不提供完整在线编辑器能力
5. 已确认：`admin` / `manager` / `reviewer` 可确认脚本，`tester` 仅可提交待确认
6. 已确认：首期默认主模式为 `录制增强模式`，`AI直生模式` 为辅助入口
7. 已确认：首期默认且唯一输出 `Playwright TypeScript`，`Cypress` 暂不支持，后续按需求扩展
8. 已确认：录制结果需保留原始录制稿和结构化步骤模型，作为 AI 重构与问题回溯依据
9. 已确认：AI 在录制增强模式下主要负责按标准框架重构，不应擅自改变关键执行逻辑与核心 locator
10. 已确认：断言生成策略纳入首期范围
11. 已确认：首期支持页面到达、元素可见性、元素状态、文本结果、表单结果、列表/表格结果 6 类基础断言
12. 已确认：每条生成脚本至少包含 1 个最终结果断言；低置信度断言以“断言建议 / 待确认项”形式输出
13. 已确认：首期必须支持脚本生成后的回放验证，且仅支持手动点击“验证”触发
14. 已确认：回放验证结果需展示失败步骤、报错信息、截图、日志、断言结果
15. 已确认：回放验证未通过的脚本不可确认，可手动调试并在验证通过后确认
16. 已确认：脚本重新生成、人工编辑为新版本、关键内容变更、执行配置变更、关联用例关键变更时必须重新触发回放验证
17. 已确认：仅修改名称、备注、标签、展示类信息，或执行预览/复制/导出时，不触发重新验证
18. 已确认：命中重验场景时原验证结果立即失效，状态变为 `待重新验证`；已确认脚本需生成新版本并回到 `待确认`、`待重新验证`

21. 已确认：任务列表批量治理是正式需求，必须支持跨页全选、按筛选结果全选、批量废弃、批量删除
