## 一、概述

### 1.1 背景

测试管理平台目前的用例录入完全依赖人工：测试工程师在阅读需求文档（PRD、用户故事、接口文档等）后，手动拆解、抽象、归纳为测试用例。这条链路存在三个问题：

1. **效率瓶颈**：一份 5000 字的功能需求，人工拆解为 20–40 条用例通常需要 1–2 小时；遇到大型迭代需求文档时，用例覆盖容易出现遗漏。
2. **风格不一致**：不同测试工程师产出的用例在标题结构、步骤粒度、断言密度上差异较大，给后续评审带来额外成本。
3. **需求不可溯源**：现有 `requirements` 表只承载 `title + content` 纯文本，**没有记录原始上传的需求文件**，也无法追溯一条用例从哪份需求来。

平台已具备的基础设施：

- **AI 模型配置模块** (`ai_model_configs`)：admin 已在系统管理中维护启用的 LLM，激活后会同步到 executor，提供统一的 LLM 调用通道。
- **Executor** (Python + FastAPI)：已承担 AI 脚本生成 / 重构 / 模型连通性测试职责，可继续承担"需求 → 用例"的提示词工程与解析职责。
- **测试用例 + 模块树**：用例的 `draft` 状态、模块树挂载、版本管理已成熟，AI 产物可直接落入。
- **评审模块**：本需求**不与评审流程耦合**，AI 产物以 draft 落库后由用户决定后续是否提审。

本次需求目标：在系统中新增「需求智生」一级菜单，支持用户上传/录入需求文档，调用平台配置的 LLM 通过预设 Skill 分析需求并产出测试用例草稿，用户预览勾选后批量入库到指定项目模块。所有需求文档与 AI 生成任务**可溯源、可回看、可重跑**。

### 1.2 核心设计决策

| 决策项 | 结论 |
|--------|------|
| 需求承载 | **新建独立 `requirement_docs` 表**，与旧 `requirements` 解耦；旧表保留兼容现有 `requirement_testcases` 关联，不动现有逻辑 |
| 支持格式 | Word (.docx) / PDF (.pdf) / Markdown (.md) / 纯文本 (.txt) / 页面直接粘贴文本，共 5 种入口 |
| 文档解析 | **executor 侧统一处理**：python-docx / pdfplumber 在 Python 端复用更稳定，Go 后端仅做转发 |
| Skill 模型 | **代码内置预设 + DB 可覆写**：内置一组系统 Skill（如"功能测试用例"），admin 可在系统管理新增/覆写自定义 Skill |
| 入口位置 | 侧边栏新增一级菜单「需求智生」（`/requirement-gen`），独立于用例列表；用例列表不增加入口避免污染 |
| AI 执行模式 | **异步任务 + 轮询**：复用 AI 脚本任务的状态机模式（DRAFT→RUNNING→SUCCESS/FAILED），前端轮询 1 次/2s |
| 用例落库 | **预览 → 用户勾选 → 批量入库 draft**：AI 产物不自动写入用例库，必须用户主动确认；与评审流程完全解耦 |
| 溯源 | 每条 AI 生成的用例**强制关联** `requirement_doc_id + gen_task_id + skill_id`，可在用例详情查看来源 |
| 项目隔离 | 需求文档、生成任务、Skill 自定义副本均按项目隔离（Skill 系统预设全局共享，项目可覆写） |
| LLM 调用 | **复用现有 ai_model_configs 激活模型**，executor 内通过现有通道调用；不在 Go 侧引入新的 LLM SDK |

### 1.3 与现有模块的边界

| 关注点 | 本模块 | 既有模块 |
|---|---|---|
| 文档存储 | `requirement_docs` 表 + uploads/requirement_docs/ | 用例附件 / 评审附件各自独立 |
| 旧 `requirements` 表 | 不读不写不引用 | 沿用现有 requirement_testcases 关联 |
| 用例落库 | 写入 `testcases`，status=draft，携带 source_* 字段 | 用例列表正常显示，与手动创建无差异 |
| 评审 | 不耦合：AI 产物不会自动进入评审 | 用户后续手动提审 |
| LLM 调用 | 仅消费 `ai_model_configs.is_active=true` 的配置 | 配置维护仍在系统管理 → AI 模型 |

## 二、功能需求

### 2.1 需求智生主页（`/requirement-gen`）

页面分三个 tab：**需求文档** / **生成任务** / **Skill 管理**。

#### FR-REQGEN-01 需求文档列表（默认 tab）

- 顶部：项目选择器（复用侧边栏）+ 「上传需求」按钮 + 「粘贴文本」按钮 + 搜索框
- 列表字段：

| 列 | 说明 |
|----|------|
| 标题 | 文档标题（用户填或自动取文件名） |
| 来源类型 | upload_file / paste_text，对应 icon |
| 文件格式 | docx / pdf / md / txt / text（粘贴） |
| 解析状态 | not_parsed / parsing / parsed / parse_failed |
| 字数 | 解析后的纯文本字数 |
| 关联任务数 | 已基于此文档发起的生成任务数 |
| 关联用例数 | 已从此文档落库的用例数 |
| 上传人 | 头像 + 名称 |
| 上传时间 | 时间戳 |
| 操作 | 查看 / 发起生成 / 删除 |

- 「查看」抽屉：左侧元信息 + 右侧解析后的纯文本预览（带原始格式标记），可下载原文件
- 「发起生成」：跳转到 2.2 生成任务创建抽屉
- 「删除」：软删除；若已有关联用例则二次确认（删除后用例上的来源链接显示为"原始需求已删除"）

#### FR-REQGEN-02 上传需求

抽屉表单：

| 字段 | 必填 | 说明 |
|---|---|---|
| 文档标题 | ✓ | 默认取文件名（去扩展名），可改 |
| 项目 | ✓ | 默认当前项目，可改 |
| 文件 | ✓（上传模式） | 支持 .docx/.pdf/.md/.txt，单文件最大 10 MB |
| 备注 | ✗ | 业务背景说明，可选 |

- 上传成功立即异步触发解析（调用 executor `/requirement/parse`）
- 解析失败标记 `parse_failed`，前端可重试

#### FR-REQGEN-03 粘贴文本

抽屉表单：

| 字段 | 必填 | 说明 |
|---|---|---|
| 文档标题 | ✓ | 用户输入 |
| 项目 | ✓ | 默认当前项目 |
| 内容 | ✓ | 富文本/纯文本编辑器，最长 50 000 字 |
| 备注 | ✗ | 可选 |

- 粘贴文本无需 executor 解析，提交即 `parsed`

#### FR-REQGEN-04 生成任务列表（第二 tab）

- 列表字段：任务名 / 来源文档 / 使用 Skill / 状态 / 生成数 / 已采纳数 / 执行人 / 创建时间 / 操作
- 状态：`PENDING`（待执行）→ `RUNNING`（生成中）→ `SUCCESS` / `FAILED` → `PARTIAL_ADOPTED`（部分采纳）→ `FULLY_CLOSED`（已完成预览或被丢弃）
- 操作：查看产物 / 重跑（基于同需求重新生成） / 丢弃

#### FR-REQGEN-05 Skill 管理（第三 tab，仅 admin 可见）

- 列表展示所有 Skill：内置（不可删除，可覆写）+ 自定义
- 字段：Skill 名 / 适用场景 / 输出粒度 / 是否系统内置 / 创建人 / 操作
- 操作：新建 / 编辑 / 覆写预设（生成项目副本） / 删除（仅自定义）
- 编辑表单：name / scope（functional/api/compat/security/...） / 适用场景描述 / prompt_template（多行文本，支持 `{{requirement_text}}` `{{project_context}}` 占位符） / output_schema（JSON Schema 选模板，详见 §3.4）

### 2.2 发起生成任务

从需求文档列表「发起生成」入口打开抽屉：

| 字段 | 必填 | 说明 |
|---|---|---|
| 任务名 | ✓ | 默认"基于《{doc_title}》生成 - {时间}" |
| 来源文档 | ✓（预填） | 不可改 |
| Skill | ✓ | 下拉选择本项目可用 Skill |
| 目标模块 | ✗ | 预选落库模块，不选则置于"未分类" |
| 用例级别默认值 | ✗ | P0/P1/P2/P3，默认 P2 |
| 生成数量上限 | ✓ | 默认 30，最大 50（控本） |
| 额外提示 | ✗ | 用户补充上下文，如"侧重边界值" |

提交后：

1. 创建 `requirement_gen_tasks` 记录，状态 = PENDING
2. 异步调用 executor `/requirement/generate-cases`（携带文档解析文本 + Skill 提示词 + 用户补充上下文 + 激活模型配置）
3. 状态转为 RUNNING，前端轮询
4. executor 完成后，回写产物到 `requirement_gen_results`（每条一行），状态 = SUCCESS / FAILED

### 2.3 预览与勾选入库

任务状态 = SUCCESS 后，点击「查看产物」打开全屏抽屉：

- 顶部：任务元信息（来源文档、Skill、生成数）+ 「全选」「反选」「批量调整级别/模块」按钮
- 列表：每行一条 AI 产物用例

| 列 | 说明 |
|---|---|
| ☑ | 复选框 |
| # | 序号 |
| 标题 | 可点击展开 |
| 级别 | 行内可改 |
| 步骤数 | 摘要 |
| AI 置信度 | LLM 自评（可选，0–1，仅展示） |
| 标签 | AI 建议标签（来自项目标签库） |
| 操作 | 编辑 / 删除（仅本预览页） |

- 点击行展开完整用例（标题/前置/步骤/后置/备注/标签）行内编辑
- 底部：「入库选中（N）」 + 「全部丢弃」
- 入库时：
  - 选中条目以 status=draft 写入 `testcases`
  - 自动填充 `source_requirement_doc_id` / `source_gen_task_id` / `source_skill_id` / `source_ai_generated=true`
  - 已入库条目在产物列表标记"已入库"灰显，不可再次入库
  - 任务状态根据剩余可采纳数转为 PARTIAL_ADOPTED 或 FULLY_CLOSED

### 2.4 用例详情的来源溯源

用例详情页（已有页面）右侧信息栏新增「来源」区块：

- 来源类型：人工创建 / AI 生成
- AI 生成时展示：
  - 来源需求文档（链接到文档查看页）
  - 生成任务（链接到任务详情）
  - 使用的 Skill
  - 生成时间
  - 「查看原始产物」按钮（弹出 LLM 原始返回 JSON，便于审计）

## 三、数据模型

### 3.1 RequirementDoc 需求文档表

```sql
CREATE TABLE requirement_docs (
  id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  project_id      BIGINT UNSIGNED NOT NULL,
  title           VARCHAR(200) NOT NULL,
  source_type     VARCHAR(20) NOT NULL,           -- upload_file / paste_text
  file_format     VARCHAR(20) NOT NULL,           -- docx / pdf / md / txt / text
  file_path       VARCHAR(500) DEFAULT '',        -- 原始文件相对路径（粘贴文本为空）
  file_size       BIGINT DEFAULT 0,
  raw_content     LONGTEXT,                       -- 解析后纯文本（也存粘贴文本原文）
  word_count      INT DEFAULT 0,
  parse_status    VARCHAR(20) NOT NULL DEFAULT 'not_parsed',
                                                  -- not_parsed / parsing / parsed / parse_failed
  parse_error     VARCHAR(1000) DEFAULT '',
  truncated       TINYINT(1) NOT NULL DEFAULT 0,  -- 是否因超长被截断
  original_word_count INT DEFAULT 0,              -- 截断前原始字数（未截断时与 word_count 一致）
  remark          VARCHAR(500) DEFAULT '',
  created_by      BIGINT UNSIGNED NOT NULL,
  created_at      DATETIME NOT NULL,
  updated_at      DATETIME NOT NULL,
  deleted_at      DATETIME,                       -- 软删
  INDEX idx_rd_project (project_id),
  INDEX idx_rd_creator (created_by),
  INDEX idx_rd_parse_status (parse_status)
);
```

### 3.2 RequirementGenTask 生成任务表

```sql
CREATE TABLE requirement_gen_tasks (
  id                    BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  project_id            BIGINT UNSIGNED NOT NULL,
  requirement_doc_id    BIGINT UNSIGNED NOT NULL,
  skill_id              BIGINT UNSIGNED NOT NULL,
  ai_model_config_id    BIGINT UNSIGNED NOT NULL,   -- 任务发起时刻的激活模型快照
  ai_model_snapshot     VARCHAR(200),                -- provider/model_id 文本快照（审计）
  task_name             VARCHAR(200) NOT NULL,
  target_module_id      BIGINT UNSIGNED DEFAULT 0,
  default_level         VARCHAR(10) DEFAULT 'P2',
  max_cases             INT NOT NULL DEFAULT 30,
  extra_prompt          TEXT,                        -- 用户补充上下文
  parent_task_id        BIGINT UNSIGNED DEFAULT 0,   -- rerun 来源任务 ID，0 表示原生任务
  status                VARCHAR(20) NOT NULL DEFAULT 'PENDING',
                                                     -- PENDING / RUNNING / SUCCESS / FAILED
                                                     -- / PARTIAL_ADOPTED / FULLY_CLOSED
  generated_count       INT DEFAULT 0,
  adopted_count         INT DEFAULT 0,
  fail_reason           VARCHAR(1000),
  started_at            DATETIME,
  finished_at           DATETIME,
  duration_ms           BIGINT,
  prompt_tokens         INT DEFAULT 0,               -- LLM 用量审计
  completion_tokens     INT DEFAULT 0,
  created_by            BIGINT UNSIGNED NOT NULL,
  created_at            DATETIME NOT NULL,
  updated_at            DATETIME NOT NULL,
  INDEX idx_rgt_project_status (project_id, status),
  INDEX idx_rgt_doc (requirement_doc_id),
  INDEX idx_rgt_skill (skill_id),
  INDEX idx_rgt_creator (created_by)
);
```

### 3.3 RequirementGenResult AI 产物表（预览池）

```sql
CREATE TABLE requirement_gen_results (
  id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  task_id         BIGINT UNSIGNED NOT NULL,
  project_id      BIGINT UNSIGNED NOT NULL,
  seq_no          INT NOT NULL,                   -- 在任务内的序号
  title           VARCHAR(200) NOT NULL,
  level           VARCHAR(10) DEFAULT 'P2',
  precondition    TEXT,
  steps           LONGTEXT,                       -- 与 testcase.steps 同结构（JSON or Markdown）
  postcondition   TEXT,
  remark          TEXT,
  tags_suggested  VARCHAR(500),                   -- AI 建议标签，逗号分隔
  ai_confidence   DECIMAL(3,2),                   -- 0.00–1.00，LLM 自评
  raw_json        LONGTEXT,                       -- LLM 原始输出片段（审计）
  adopted         TINYINT(1) NOT NULL DEFAULT 0,  -- 是否已入库
  adopted_case_id BIGINT UNSIGNED DEFAULT 0,
  adopted_at      DATETIME,
  adopted_by      BIGINT UNSIGNED DEFAULT 0,
  discarded       TINYINT(1) NOT NULL DEFAULT 0,
  created_at      DATETIME NOT NULL,
  UNIQUE KEY uk_task_seq (task_id, seq_no),
  INDEX idx_rgr_task (task_id),
  INDEX idx_rgr_adopted (adopted)
);
```

### 3.4 AISkill Skill 模板表

```sql
CREATE TABLE ai_skills (
  id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  project_id      BIGINT UNSIGNED DEFAULT 0,      -- 0 表示系统内置（全局）
  skill_key       VARCHAR(50) NOT NULL,           -- 唯一标识，如 'functional_testcase'
  name            VARCHAR(100) NOT NULL,
  scope           VARCHAR(20) NOT NULL,           -- functional / api / compat / security / custom
  description     VARCHAR(500),
  prompt_template LONGTEXT NOT NULL,              -- 含 {{requirement_text}} {{project_context}} {{few_shot_examples}} 占位
                                                  -- few_shot_examples 当前版本始终渲染为空串，预留 v1.1 接入
  output_schema   VARCHAR(50) NOT NULL DEFAULT 'standard_case_v1',
                                                  -- 预设 JSON Schema 名称
  is_system       TINYINT(1) NOT NULL DEFAULT 0,  -- 内置不可删
  is_active       TINYINT(1) NOT NULL DEFAULT 1,
  sort_order      INT DEFAULT 0,
  created_by      BIGINT UNSIGNED DEFAULT 0,
  created_at      DATETIME NOT NULL,
  updated_at      DATETIME NOT NULL,
  UNIQUE KEY uk_skill_project_key (project_id, skill_key),
  INDEX idx_skill_active (is_active)
);
```

**内置 Skill（MVP 至少提供这两个）**

| skill_key | name | scope | 适用场景 |
|---|---|---|---|
| functional_testcase | 通用功能测试用例 | functional | 业务功能验证，正向+异常+边界 |
| api_testcase | 接口测试用例 | api | 接口契约校验、参数边界、状态码 |

后续可扩展：compat_testcase（兼容性矩阵）、security_testcase（安全要点）、ui_ux_testcase 等。

**output_schema 预设值**

- `standard_case_v1`：title / level / precondition / steps[] / postcondition / remark / tags[]
- 后续可扩展 `api_case_v1`（含 method/url/headers/body/expected）等

### 3.5 TestCase 字段扩展

在 `testcases` 表追加三个可空溯源字段（AutoMigrate 增量）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `source_ai_generated` | BOOL | 是否 AI 生成（默认 false） |
| `source_requirement_doc_id` | BIGINT | 来源需求文档（可空） |
| `source_gen_task_id` | BIGINT | 来源生成任务（可空） |
| `source_skill_id` | BIGINT | 使用的 Skill（可空） |

老用例字段保持 NULL/0，**不影响现有查询**。用例详情接口在返回时 JOIN 文档/任务/Skill 表回填 name 字段。

## 四、接口设计

所有接口位于 `/api/v1` 下，需认证。错误码新增段：`13xxxx` 需求智生模块。

### 4.1 需求文档

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/projects/:projectID/requirement-docs/upload` | 上传文件（multipart） |
| POST | `/projects/:projectID/requirement-docs/paste` | 提交粘贴文本 |
| GET  | `/projects/:projectID/requirement-docs` | 列表（分页 + 状态筛选 + 关键字） |
| GET  | `/projects/:projectID/requirement-docs/:docID` | 详情（含解析后内容） |
| POST | `/projects/:projectID/requirement-docs/:docID/reparse` | 重新解析 |
| GET  | `/projects/:projectID/requirement-docs/:docID/download` | 下载原文件 |
| DELETE | `/projects/:projectID/requirement-docs/:docID` | 软删 |

### 4.2 生成任务

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/projects/:projectID/requirement-gen/tasks` | 创建任务（立即异步触发） |
| GET  | `/projects/:projectID/requirement-gen/tasks` | 任务列表 |
| GET  | `/projects/:projectID/requirement-gen/tasks/:taskID` | 任务详情 + 状态 |
| GET  | `/projects/:projectID/requirement-gen/tasks/:taskID/results` | 产物列表 |
| POST | `/projects/:projectID/requirement-gen/tasks/:taskID/adopt` | 批量采纳（入库为用例） |
| POST | `/projects/:projectID/requirement-gen/tasks/:taskID/discard-all` | 全部丢弃 |
| POST | `/projects/:projectID/requirement-gen/tasks/:taskID/rerun` | 重跑 |

`POST /adopt` 请求体：

```json
{
  "result_ids": [12, 14, 17],
  "target_module_id": 8,
  "default_level_override": "P1"   // 可选
}
```

响应：

```json
{
  "adopted_count": 3,
  "adopted_case_ids": [201, 202, 203],
  "skipped": []
}
```

### 4.3 Skill 管理（仅 admin）

| 方法 | 路径 | 说明 |
|---|---|---|
| GET    | `/ai-skills` | 全量列表（系统内置 + 全部项目自定义） |
| GET    | `/projects/:projectID/ai-skills` | 本项目可用列表（系统内置 + 本项目副本，按 skill_key 去重，副本优先） |
| POST   | `/ai-skills` | 新建（指定 project_id=0 即系统内置） |
| PUT    | `/ai-skills/:skillID` | 更新 |
| POST   | `/ai-skills/:skillID/clone-to-project` | 从内置克隆出项目副本 |
| DELETE | `/ai-skills/:skillID` | 删除（系统内置禁止） |

### 4.4 Executor 新增接口

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/requirement/parse` | 解析文件 → 纯文本 |

## 六、状态机与流程

### 6.1 RequirementDoc 解析状态

```
not_parsed ──(异步触发)──→ parsing ──→ parsed
                              │
                              └─(失败)→ parse_failed ──(重试)──→ parsing
```

粘贴文本直接落 `parsed`，跳过 `parsing`。

### 6.2 RequirementGenTask 任务状态

```
PENDING ──(executor 接单)──→ RUNNING ──→ SUCCESS ──(用户采纳部分)──→ PARTIAL_ADOPTED
                                |                                         |
                                └─→ FAILED                                 └─(剩余全丢弃)→ FULLY_CLOSED
```

- SUCCESS 状态下用户可多次采纳，直到全部产物被处理完
- 全部产物 adopted=1 或 discarded=1 时，自动转 FULLY_CLOSED

### 6.3 端到端流程

```
[用户]                      [Go 后端]                    [Executor]                 [LLM]
  │  上传需求 →                 │                              │                         │
  │                              │── /requirement/parse ──→     │                         │
  │                              │←─ {text, word_count} ────── │                         │
  │  发起生成 →                 │                              │                         │
  │                              │── /requirement/generate-cases →│                       │
  │  ←── taskID, status=RUNNING  │                              │── /v1/chat/completions →│
  │  (前端轮询 2s)               │                              │←─ JSON 用例数组 ───── │
  │                              │←─ {status, cases[], tokens}──│                         │
  │  status=SUCCESS               │                              │                         │
  │  查看产物 → 勾选 → 采纳 →    │ 写入 testcases (draft)        │                         │
```

## 十、开放问题决议

本节固化 v1.0 讨论稿留下的 5 个开放问题的最终结论。每条决议都有对应的方案落点，未来调整须连带更新前文相关章节。

### 10.1 LLM 调用失败的重试策略

**决议：仅对网络层错误自动重试 1 次，业务层错误一律不重试。**

| 错误类别 | 示例 | 重试策略 |
|---|---|---|
| 网络层 | HTTP 5xx / connection reset / timeout / DNS 解析失败 | executor 内自动重试 1 次（间隔 2s），仍失败则任务 FAILED |
| 业务层 | HTTP 4xx（401/429/400） / LLM 输出非合法 JSON / Schema 校验失败 / 内容安全策略拦截 | 不重试，立即 FAILED，错误信息回写 `fail_reason` |

**落点**：
- §七 非功能要求新增"LLM 失败重试"行
- executor `/requirement/generate-cases` 内部实现该策略，Go 侧仅消费最终结果
- `requirement_gen_tasks.fail_reason` 明确包含错误类别前缀，例如 `[NETWORK] connection reset` / `[BUSINESS] schema_violation: missing field 'steps'`

### 10.2 多文件合并喂给 AI

**决议：本期不支持，1 个需求文档 = 1 个生成任务源。**

理由：
- 多文件合并会让 token 用量难以预估，容易突破上下文窗口
- "合并"语义不清晰（拼接？分段总结后合并？），产品边界模糊
- 溯源会从 `1:N` 退化为 `M:N`，需额外关联表，复杂度收益不匹配

**用户替代路径**：
- 通过「粘贴文本」入口手工拼接多份内容形成单一文档
- 多份独立需求建议拆为多个任务（同一需求文档列表可批量发起）

**落点**：
- §一 1.2 表中"需求承载"行结论保持不变
- §四 4.1 接口不引入"批量上传合并"动作

### 10.3 rerun 是否展示与上次产物的 diff

**决议：不做字面或语义 diff；rerun 等同新建任务，新任务详情顶部显示"由任务 #N 重跑而来"提示并提供反向链接。**

理由：
- LLM 输出非确定性，字面 diff 信噪比低
- 语义 diff 实现复杂、收益不稳定，超出 MVP 范围

**落点**：
- `requirement_gen_tasks` 新增字段 `parent_task_id BIGINT UNSIGNED DEFAULT 0`（已在 §3.2 表结构中体现），0 表示原生任务，非 0 表示由该任务 rerun 而来
- §四 4.2 `POST /tasks/:taskID/rerun` 响应增加 `parent_task_id` 字段
- 前端在任务详情顶部根据 `parent_task_id` 渲染面包屑式回链
- 任务列表新增"重跑链"图标列，方便识别 rerun 任务

### 10.4 Skill prompt 是否暴露 few-shot 用例样例

**决议：第一版不做 few-shot，但 prompt 模板预留 `{{few_shot_examples}}` 占位符，当前版本始终渲染为空串。**

理由：
- few-shot 效果取决于样例选取策略（随机 / 最新 / approved / 同模块），策略本身就是一个独立设计议题
- 第一版用结构化 prompt + JSON Schema 约束输出，先建质量基线
- 占位符预留让 v1.1 接入无需改 Skill 表结构和 prompt 渲染入口

**落点**：
- §3.4 `ai_skills.prompt_template` 注释明确列出三个占位符
- executor 端 prompt 渲染函数（template_renderer）当前直接把 `{{few_shot_examples}}` 替换为空串
- §11 演进项 1.1 列出 few-shot 启用路径

### 10.5 超长文档（> 50K 字）分块策略

**决议：本期硬截断 + UI 双重警示，复杂分块策略放到 v1.1。**

具体规则：
- 解析后纯文本长度（按字符数）若 > 50 000，**保留前 50 000 字**送入 LLM
- `requirement_docs.truncated=1` 且 `original_word_count` 记录原始长度
- UI 提示位置：
  1. **需求文档列表**：截断行显示"已截断"标签
  2. **文档查看抽屉**：顶部黄色警告条"文档原始 X 字，已截断前 50 000 字，剩余 Y 字未参与 AI 分析"
  3. **生成任务创建抽屉**：选择已截断文档时再次提示，避免用户遗漏
- 用户行动指引：若关键内容在文档后半部分，建议拆分文档或使用「粘贴文本」入口分段录入

**落点**：
- §3.1 `requirement_docs` 表新增 `truncated` / `original_word_count` 字段（已体现）
- §七 非功能要求新增"超长文档处理"行
- §11 演进项 1.5 列出真正分块策略的演进路径

## 十一、后续可演进项（v1.1+）

以下为 v1.0 不实现、但方案已为其预留扩展点的能力，待 MVP 上线后视使用反馈优先级评估。

### 11.1 Skill few-shot 范例支持

- `ai_skills` 增加 `few_shot_strategy` 枚举：`none / manual / latest_approved / similar_module`
- 新增 `few_shot_examples` 字段（manual 时存 JSON 数组），自动策略时在生成任务时查询用例库动态拼装
- 已预留的 `{{few_shot_examples}}` 占位符无需改动模板表结构即可启用

### 11.2 多文档合并 / 文档集合概念

- 新增 `requirement_doc_collections` 表（一对多挂载多个 doc）
- 任务源从 `requirement_doc_id` 扩展为支持 `collection_id`
- 溯源关联从 `1:1` 升级为 `1:N`，需要扩展用例 source 字段或独立关联表

### 11.3 真正的超长文档分块

- 新增 `requirement_doc_chunks` 表：按 token 数 / 章节边界切片
- executor 新增 `/requirement/chunk-and-summarize`：分块 → 每块独立生成 → 跨块去重 → 合并
- 任务详情可查看每个 chunk 的产物贡献

### 11.4 生成结果的语义 diff

- 同一需求文档的多次任务产物之间，按"覆盖场景"维度做 diff
- 实现路径：用 LLM 对两批用例的"覆盖场景集合"做对比，输出新增/缺失/重叠
- 落点候选：任务详情新增"对比任务"动作

### 11.5 Prompt token / 成本看板

- 任务表已采集 `prompt_tokens` / `completion_tokens`
- 后续可在系统管理新增"AI 成本看板"子菜单，按项目 / 时间段 / 模型聚合
- 与 AI 模型配置模块联动，可设置项目级月度配额报警

### 11.6 自动重试策略可配置化

- 当前网络层重试次数硬编码为 1 次、间隔 2s
- 可后续在系统管理 → AI 模型配置增加全局参数：`network_retry_count` / `network_retry_interval_ms`
- 业务层错误依然保持不重试的原则
