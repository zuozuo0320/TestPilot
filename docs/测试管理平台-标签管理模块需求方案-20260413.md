# 标签管理模块需求方案

> 版本：v1.3 · 日期：2026-04-13 · 状态：可开发版（v1.3 根据项目真实现状修正旧标签能力、迁移策略、删除口径、接口风格与权限表达）

---

## 一、概述

### 1.1 背景

当前测试用例模型中已经存在 `tags` 文本字段（`varchar(500)`），且前端用例模块已经提供了**逗号分隔的基础标签录入 / 编辑 / 列表展示能力**。但现状仍存在以下问题：

1. 标签只是自由文本，**没有统一标签库**，同义标签、重名标签、大小写差异无法治理。
2. 标签没有颜色、描述、创建人、关联统计等元数据，**无法管理和复用**。
3. 标签无法按项目统一维护，也缺少系统级治理入口，**无法支撑长期演进**。
4. 当前标签仅存于 `TestCase.Tags` 文本字段，难以支持结构化筛选、统计和权限治理。

本次需求的目标是：

1. 在**系统管理**下新增「标签管理」二级模块，支持标签的增删改查。
2. 标签按**项目级别**隔离，不同项目各自维护独立的标签库。
3. 在用例新增/编辑表单中增加标签选择器，支持**选择已有标签 + 快速新建**。
4. 标签支持**自定义颜色**，便于视觉区分。

### 1.2 核心设计决策

| 决策项 | 结论 |
|--------|------|
| 标签作用域 | **项目级**：每个项目拥有独立标签库，互不共享 |
| 标签颜色 | **支持**：每个标签可设置一个颜色值，格式为 `#RRGGBB`（6 位十六进制） |
| 用例绑定方式 | **选择 + 快速新建**：下拉选择已有标签，同时可输入新名称即时创建 |
| 旧数据迁移 | **必须迁移**：将现有 `TestCase.Tags` 历史文本标签按项目维度回填到 `tags` / `testcase_tags`；迁移完成后进入兼容期 |
| 兼容策略 | **v1.x 兼容双轨**：结构化标签以 `tags` / `testcase_tags` 为准，同时保留 `TestCase.Tags` 作为兼容缓存字段，供旧页面 / 导出 / 灰度期逻辑平滑过渡 |
| 写操作事务 | **强制事务**：所有涉及多表写入的操作（创建用例+绑定标签、删除标签+解绑关联、克隆用例+复制标签等）须在同一数据库事务中完成，确保原子性 |
| 标签名称归一化 | **保存前 trim**：标签名保存前自动去除首尾空格；项目内唯一性按归一化后的名称判定 |
| 默认排序 | 标签管理页默认按创建时间倒序；标签选择器 / 标签筛选下拉默认按标签名称升序 |

## 二、功能需求

### 2.1 标签管理页面（系统管理 → 标签管理）

#### FR-TAG-01 标签列表

- 入口路径：侧边栏 → 系统管理 → 标签管理（`/system/tags`）
- 页面顶部显示**项目选择器**（复用当前侧边栏项目切换逻辑），切换项目后加载该项目的标签数据
- 列表展示字段：

| 列 | 说明 |
|----|------|
| 颜色 | 圆形色块预览 |
| 标签名称 | 标签的文本名称 |
| 描述 | 可选的标签备注说明 |
| 关联用例数 | 当前仍存在的关联用例数量（当前项目 `TestCase` 为物理删除模型，无需依赖 `deleted_at` 过滤） |
| 创建人 | 标签创建者头像 + 名称 |
| 创建时间 | 创建时间 |
| 操作 | 编辑、删除 |

- 支持按名称模糊搜索
- 默认按创建时间倒序展示
- 支持分页（默认 20 条/页）

#### FR-TAG-01a 页面状态

- 未选择项目时，页面显示空态提示"请先选择项目"
- 切换项目后：
  - 清空标签管理页的关键词搜索与分页状态
  - 重新加载当前项目下的标签列表
  - 不保留上一个项目的标签缓存结果

#### FR-TAG-02 新建标签

- 弹窗表单，字段：
  - **标签名称**（必填，2-50 字符，当前项目内唯一）
  - **颜色**（必填，从预置色板选择或自定义，格式必须为 `#RRGGBB`，提供 12 个推荐色）
  - **描述**（选填，最多 200 字符）
- 点击「确认」后创建标签并刷新列表

预置推荐色板（12 色）：

| 色号 | HEX | 语义参考 |
|------|-----|----------|
| 1 | `#3B82F6` | 蓝色 |
| 2 | `#10B981` | 绿色 |
| 3 | `#F59E0B` | 琥珀 |
| 4 | `#EF4444` | 红色 |
| 5 | `#8B5CF6` | 紫色 |
| 6 | `#EC4899` | 粉红 |
| 7 | `#06B6D4` | 青色 |
| 8 | `#F97316` | 橙色 |
| 9 | `#14B8A6` | 蓝绿 |
| 10 | `#6366F1` | 靛蓝 |
| 11 | `#84CC16` | 黄绿 |
| 12 | `#78716C` | 灰色 |

#### FR-TAG-03 编辑标签

- 弹窗表单，预填当前标签数据
- 可修改：名称、颜色、描述
- 名称修改后自动更新所有已关联用例的显示（因为关联表独立，无需额外操作）

#### FR-TAG-04 删除标签

- 删除前检查关联用例数：
  - 若 `关联用例数 > 0`：弹窗提示"该标签已被 N 个用例使用，删除后将自动解除关联，是否继续？"
  - 若 `关联用例数 = 0`：常规确认弹窗"确认删除标签「XXX」？"
- 确认后（**同一事务内**）：
  1. 删除 `testcase_tags` 中所有关联记录
  2. 删除 `tags` 表中的标签记录
  3. 刷新列表

#### FR-TAG-04b 审计日志

- 标签管理涉及以下操作时，需写入审计日志：
  - 创建标签
  - 编辑标签（名称 / 颜色 / 描述）
  - 删除标签
  - 用例表单中快速新建标签
- 审计日志至少记录：操作人、项目、标签 ID、标签名称、操作前后数据摘要、操作时间

#### FR-TAG-04a 标签数量限制

- 每个项目最多创建 **100** 个标签（超出提示"当前项目标签数量已达上限（100）"）
- 每个用例最多绑定 **10** 个标签（超出提示"每个用例最多绑定 10 个标签"）

### 2.2 用例表单标签绑定

#### FR-TAG-05 用例新增/编辑时绑定标签

- 在用例新增/编辑抽屉表单中，增加「标签」字段
- 组件形态：**多选标签选择器**（`el-select` 多选 + 可搜索 + 可创建）
  - 下拉列表展示当前项目的所有标签，每个选项左侧显示颜色圆点
  - 支持输入关键字搜索过滤
  - 支持输入不存在的标签名并回车，即时创建新标签（自动分配默认颜色 `#3B82F6`）
  - 已选标签以彩色 Tag 形式横排展示
- 保存用例时，后端接收 `tag_ids []uint`，**在同一事务内**同步更新 `testcase_tags` 关联表

**快速新建标签重名处理**：若快速新建时后端返回 `409 TAG_NAME_DUPLICATE`，前端提示"标签已存在，已为您自动选中"，并从已有标签列表中找到同名标签自动选中，无需用户手动操作。

#### FR-TAG-05a 用例克隆时标签复制

- 克隆用例时，原用例的标签关联应**在同一事务内**一并复制到新用例
- 当前 `CloneCase` 仅复制 `TestCase.Tags` 旧字段，需改为同步复制 `testcase_tags` 关联记录

#### FR-TAG-05b 用例删除时标签关联清理

- 删除单个用例或批量删除用例时，需**在同一事务内**同步清理 `testcase_tags` 中的关联记录
- 避免产生孤立的关联数据

#### FR-TAG-06 用例列表标签展示

- 用例列表表格中增加「标签」列
  - 展示该用例绑定的所有标签（彩色 Tag 徽章）
  - 如标签数量 > 3，仅展示前 3 个，末尾加 `+N` 提示，hover 弹出全部

#### FR-TAG-07 用例列表按标签筛选

- 用例列表筛选栏增加「标签」筛选项
- 组件形态：多选下拉，可选当前项目的标签列表
- 筛选逻辑：**OR**（只要用例包含任一选中标签即命中）

#### FR-TAG-07a 用例关键词搜索兼容

- 当前用例列表关键词搜索已支持按标签文本命中
- 升级为结构化标签后，关键词搜索仍需支持**标签名称**匹配，避免现有使用习惯退化
- 实现方式可为：
  - 关键词搜索时 JOIN `tags` / `testcase_tags`
  - 或使用兼容缓存字段 `TestCase.Tags`
- 最终要求：用户输入标签名称时，仍可在用例列表中搜到对应标签的用例

## 三、数据库设计

### 3.1 新增表：`tags`

```sql
CREATE TABLE tags (
    id          BIGINT UNSIGNED  NOT NULL AUTO_INCREMENT PRIMARY KEY,
    project_id  BIGINT UNSIGNED  NOT NULL COMMENT '所属项目 ID',
    name        VARCHAR(50)      NOT NULL COMMENT '标签名称',
    color       VARCHAR(7)       NOT NULL DEFAULT '#3B82F6' COMMENT '标签颜色 HEX（#RRGGBB 格式）',
    description VARCHAR(200)     NOT NULL DEFAULT '' COMMENT '标签描述',
    created_by  BIGINT UNSIGNED  NOT NULL DEFAULT 0 COMMENT '创建人 ID',
    created_at  DATETIME(3)      NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at  DATETIME(3)      NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),

    UNIQUE INDEX uk_project_tag_name (project_id, name),
    INDEX idx_project_id (project_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 3.2 新增表：`testcase_tags`（关联表）

```sql
CREATE TABLE testcase_tags (
    testcase_id BIGINT UNSIGNED NOT NULL COMMENT '用例 ID',
    tag_id      BIGINT UNSIGNED NOT NULL COMMENT '标签 ID',
    created_at  DATETIME(3)     NOT NULL DEFAULT CURRENT_TIMESTAMP(3),

    PRIMARY KEY (testcase_id, tag_id),
    INDEX idx_tag_id (tag_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

> **注意**：关联表为物理删除（无软删除），且当前项目中的 `TestCase` 也是**物理删除模型**，并不存在 `deleted_at` 字段。因此 `case_count` 的实现口径应为"当前仍存在的关联用例数"，可直接基于现存 `test_cases` + `testcase_tags` 统计；不应把 `deleted_at IS NULL` 写入本期实现要求。

### 3.3 Go Model 定义

```go
// Tag 标签实体
type Tag struct {
    ID          uint      `json:"id" gorm:"primaryKey"`
    ProjectID   uint      `json:"project_id" gorm:"not null;uniqueIndex:uk_project_tag_name;index"`
    Name        string    `json:"name" gorm:"size:50;not null;uniqueIndex:uk_project_tag_name"`
    Color       string    `json:"color" gorm:"size:7;not null;default:#3B82F6"`
    Description string    `json:"description" gorm:"size:200;default:''"`
    CreatedBy   uint      `json:"created_by" gorm:"not null;default:0"`
    CreatedAt   time.Time `json:"created_at"`
    UpdatedAt   time.Time `json:"updated_at"`

    // 虚拟字段
    CaseCount        int64  `json:"case_count" gorm:"-"`          // SQL 子查询填充（按现存关联用例统计）
    CreatedByName    string `json:"created_by_name" gorm:"-"`     // JOIN / service 填充
    CreatedByAvatar  string `json:"created_by_avatar" gorm:"-"`   // JOIN / service 填充
}

// TestCaseTag 用例-标签关联
type TestCaseTag struct {
    TestCaseID uint      `json:"testcase_id" gorm:"primaryKey"`
    TagID      uint      `json:"tag_id" gorm:"primaryKey;index"`
    CreatedAt  time.Time `json:"created_at"`
}
```

### 3.4 旧字段迁移与兼容策略

当前项目真实情况是：`TestCase.Tags` 已经被前后端实际使用，不能直接宣告"不迁移、不兼容"。本期应采用**迁移 + 兼容**策略：

#### 3.4.1 历史数据迁移

- 对现有 `test_cases.tags` 做一次离线迁移：
  - 按英文逗号 `,` 和中文逗号 `，` 拆分
  - 对每个标签名做 `trim`
  - 过滤空字符串
  - 同一用例内去重
  - 同一项目下若标签不存在，则自动创建标签（默认颜色 `#3B82F6`，描述为空）
  - 为用例写入 `testcase_tags` 关联关系
- 迁移脚本需幂等，可重复执行而不产生重复标签或重复关联

#### 3.4.2 v1.x 兼容期

- **结构化标签为主数据源**：`tags` / `testcase_tags`
- **`TestCase.Tags` 保留为兼容缓存字段**，本期不立即删除
- 用例创建 / 更新 / 克隆后，后端应同步回写 `TestCase.Tags`，保证：
  - 旧页面在灰度期间不立即失效
  - xlsx 导出、旧筛选、旧搜索逻辑可平滑过渡
- 用例查询接口在兼容期内可同时返回：
  - `tag_list`：结构化标签列表
  - `tags`：逗号分隔文本缓存

#### 3.4.3 v2.0 清理计划

- 在确认所有调用方已切换到 `tag_ids` / `tag_list` 后，再通过独立 migration 物理删除 `test_cases.tags`

需同步清理的旧 `Tags` 代码引用点：

| 文件 | 位置 | 改动 |
|------|------|------|
| `repository/testcase_repo.go` | `TestCaseFilter.Tags` | 替换为 `TagIDs []uint` |
| `repository/testcase_repo.go` | `TestCaseListItem.Tags` | 替换为 `TagList []TagBrief`（id/name/color） |
| `repository/testcase_repo.go` | `ListPaged` 中 `tags LIKE` 筛选 | 改为 `EXISTS (SELECT 1 FROM testcase_tags ...)`，同时保留标签名称关键词命中能力 |
| `repository/testcase_repo.go` | `ListPaged` SELECT 中 `test_cases.tags` | 移除，查询后批量填充 `TagList` |
| `repository/testcase_repo.go` | Keyword 搜索中 `OR tags LIKE ?` | 移除 |
| `repository/testcase_repo.go` | `CloneCase` 中 `Tags: source.Tags` | 移除，改为在事务内复制 `testcase_tags` 关联 |
| `service/testcase_service.go` | `CreateTestCaseInput.Tags` | 替换为 `TagIDs []uint` |
| `service/testcase_service.go` | `UpdateTestCaseInput.Tags` | 替换为 `TagIDs []uint` |
| `service/testcase_service.go` | `Create` 中 `Tags: input.Tags` | 改为：结构化写 `testcase_tags`，并同步回写 `TestCase.Tags` 兼容缓存 |
| `service/testcase_service.go` | `Update` 中 `input.Tags` 比对逻辑 | 改为：全量覆盖 `testcase_tags`（先删后建，同一事务），并同步回写 `TestCase.Tags` |
| `api/handler_testcase.go` | `createTestCase` 中 `Tags: req.Tags` | 改为 `TagIDs: req.TagIDs` |
| `api/handler_testcase.go` | `listTestCases` 中 `c.Query("tags")` | 改为解析 `c.Query("tag_ids")` 为 `[]uint` |

## 四、API 接口设计

所有接口基于 `RESTful` 风格，前缀 `/api/v1`，需 JWT 认证。

> **统一响应风格说明**：本项目成功业务码统一为 `200000`，分页响应统一使用 `response.Page(...)` 包装，因此响应体字段仍为 `page_size`；但前端请求参数建议与现有模块保持一致，使用 `page` + `pageSize`。

### 4.1 标签 CRUD

#### `GET /projects/:projectID/tags` — 标签列表

**Query 参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `keyword` | string | 否 | 按名称模糊搜索 |
| `page` | int | 否 | 页码，默认 1 |
| `pageSize` | int | 否 | 每页数量，默认 20 |

> **建议**：不要复用 `pageSize=0` 这种特殊语义。标签选择器和筛选下拉建议单独提供轻量接口 `GET /projects/:projectID/tags/options`，用于一次性拉取全部标签候选，避免与当前项目通用分页参数风格冲突。

**响应示例（分页模式）**：

```json
{
  "code": 200000,
  "data": [
    {
      "id": 1,
      "project_id": 1,
      "name": "冒烟测试",
      "color": "#EF4444",
      "description": "核心路径冒烟验证",
      "case_count": 15,
      "created_by": 1,
      "created_by_name": "admin",
      "created_by_avatar": "/uploads/avatars/1.png",
      "created_at": "2026-04-13T10:00:00Z",
      "updated_at": "2026-04-13T10:00:00Z"
    }
  ],
  "total": 1,
  "page": 1,
  "page_size": 20
}
```

#### `GET /projects/:projectID/tags/options` — 标签候选列表（轻量接口）

- 用途：用例表单标签选择器、用例列表标签筛选下拉
- 特点：
  - 不分页
  - 返回轻量字段：`id/name/color`
  - 支持 `keyword` 模糊搜索
  - 默认按标签名称升序

#### `POST /projects/:projectID/tags` — 创建标签

**请求体**：

```json
{
  "name": "冒烟测试",
  "color": "#EF4444",
  "description": "核心路径冒烟验证"
}
```

**成功响应**（返回完整标签实体，前端快速新建场景依赖 `id`）：

```json
{
  "code": 200000,
  "data": {
    "id": 7,
    "project_id": 1,
    "name": "冒烟测试",
    "color": "#EF4444",
    "description": "核心路径冒烟验证",
    "created_by": 1,
    "created_by_avatar": "/uploads/avatars/1.png",
    "created_at": "2026-04-13T10:00:00Z",
    "updated_at": "2026-04-13T10:00:00Z"
  }
}
```

**校验规则**：

| 字段 | binding 规则 | service 层补充校验 |
|------|-------------|-------------------|
| `name` | `required,min=2,max=50` | 同项目内唯一（DB 唯一索引兜底） |
| `color` | `required,len=7` | 正则 `^#[0-9A-Fa-f]{6}$`（仅允许标准 `#RRGGBB` 格式） |
| `description` | `omitempty,max=200` | — |

> **颜色格式说明**：binding 的 `len=7` 仅校验长度，service 层必须额外用正则 `^#[0-9A-Fa-f]{6}$` 验证格式，杜绝非法值（如 `AAAA-BB`）通过。

**错误码**：

| HTTP | code | 说明 |
|------|------|------|
| 409 | `TAG_NAME_DUPLICATE` | 同项目下标签名已存在 |
| 400 | `TAG_LIMIT_EXCEEDED` | 当前项目标签数量已达上限（100） |
| 400 | `TAG_COLOR_INVALID` | 颜色格式不符合 `#RRGGBB` 规范 |

#### `PUT /projects/:projectID/tags/:tagID` — 更新标签

**请求体**（字段可选，仅传需要修改的字段）：

```json
{
  "name": "回归测试",
  "color": "#10B981",
  "description": "版本回归验证"
}
```

**Go 请求结构体**（使用指针字段，与现有 `updateUserRequest` 等风格一致）：

```go
type updateTagRequest struct {
    Name        *string `json:"name" binding:"omitempty,min=2,max=50"`
    Color       *string `json:"color" binding:"omitempty,len=7"`
    Description *string `json:"description" binding:"omitempty,max=200"`
}
```

> **service 层**：若 `Color` 不为 nil，需用正则 `^#[0-9A-Fa-f]{6}$` 二次校验格式。

> **安全校验**：后端需校验 `tagID` 确实属于路径中的 `projectID`，防止跨项目越权修改。删除接口同理。

#### `DELETE /projects/:projectID/tags/:tagID` — 删除标签

- **同一事务内**同时删除 `testcase_tags` 中关联记录，再删除 `tags` 表记录
- 返回被解除关联的用例数量

**响应**：

```json
{
  "code": 200000,
  "data": {
    "unlinked_case_count": 5
  }
}
```

### 4.2 用例接口变更

#### 创建/更新用例 — 新增 `tag_ids` 字段

现有 `createTestCaseRequest` / `updateTestCaseRequest` 新增：

```go
TagIDs []uint `json:"tag_ids" binding:"omitempty,max=10"`
```

binding 的 `max=10` 约束数组长度上限；service 层亦需做二次 `len` 检查并返回语义明确的错误：

```go
if len(input.TagIDs) > 10 {
    return nil, ErrBadRequest(CodeParamsError, "每个用例最多绑定 10 个标签")
}
```

- **创建用例**：传入 `tag_ids` 后**在同一事务内**批量写入 `testcase_tags`
- **更新用例**：`tag_ids` 采用**全量覆盖**策略（先删后建，同一事务），传空数组 `[]` 表示清除所有标签

#### 查询用例 — 响应新增标签数据

用例列表和详情接口的响应中，每个用例新增：

```json
{
  "tag_list": [
    { "id": 1, "name": "冒烟测试", "color": "#EF4444" },
    { "id": 3, "name": "P0核心", "color": "#3B82F6" }
  ]
}
```

#### 用例列表筛选 — 新增 `tag_ids` 查询参数

`GET /projects/:projectID/testcases` 新增：

| 参数 | 类型 | 说明 |
|------|------|------|
| `tag_ids` | string | 逗号分隔的标签 ID，如 `1,3,5`，OR 匹配 |

**边界行为**：
- `tag_ids=`（空字符串）或参数不传：不做标签过滤，等价于"全部"
- `tag_ids=1,abc,3`（含非法项）：**忽略非法项**，仅对解析成功的合法 ID（`1` 和 `3`）做 OR 筛选，不报错

### 4.3 快速新建标签（复用创建接口）

用例表单中"输入新标签名回车创建"的行为，前端调用 `POST /projects/:projectID/tags` 即可，创建成功后将返回的 `id` 追加到 `tag_ids` 中。

若接口返回 `409 TAG_NAME_DUPLICATE`，前端**不弹错误提示**，直接从已加载的标签列表中找到同名标签，将其 `id` 添加到已选列表并选中，提示"标签已存在，已为您自动选中"。

## 九、级联与边界行为

| 触发操作 | 级联行为 | 事务要求 |
|----------|----------|----------|
| 删除标签 | 同时删除 `testcase_tags` 中该标签的所有关联 | 同一事务 |
| 删除用例 / 批量删除用例 | 同时删除 `testcase_tags` 中该用例的所有关联 | 同一事务 |
| 克隆用例 | 复制原用例在 `testcase_tags` 中的所有标签关联到新用例 | 同一事务 |
| 删除项目 | 先删除该项目下所有 `testcase_tags` 关联，再删除 `tags`，再删除项目数据 | 同一事务 |
| 尝试创建第 101 个标签 | 返回 400，提示"当前项目标签数量已达上限（100）" | — |
| 尝试给用例绑定第 11 个标签 | 返回 400，提示"每个用例最多绑定 10 个标签" | — |
| `tag_ids` 筛选参数含非法值 | 忽略非法项，对合法 ID 做 OR 筛选，不报错 | — |
| 快速新建标签时同名已存在（409） | 前端不弹错，自动选中已有同名标签 | — |
| `case_count` 子查询 | 统计当前仍存在的关联用例数；本期不依赖 `deleted_at` | — |
| Excel 导入含"标签"列 | 本期忽略该列，不做标签回填；导入结果中需明确提示“标签未导入” | — |
