## 1. 改造结论

本轮“用例评审能力改造”当前已形成以下统一口径：

1. **评审动作统一收口到 `case-review` 模块**。
2. **测试用例模块不再承担 legacy 直评能力**，只保留评审结果投影与进入评审模块的入口。
3. **评审结果统一由评审计划流程回写 testcase 主表**，包括状态与评审结果文案。
4. **前端已形成列表页 + 创建页 + 详情页的完整评审入口**，不再是单一占位页。

一句话概括：

- 测试用例模块：看结果、做入口、看当前评审
- 用例评审模块：建计划、做评审、留记录、回写结果

## 2. 前端交付现状

### 2.1 已交付页面

当前前端已落地以下页面：

- `CaseReviewPage.vue`：评审计划列表页
- `CaseReviewCreate.vue`：评审计划创建页
- `CaseReviewDetail.vue`：评审计划详情页

### 2.2 列表页已支持

- 评审计划分页展示
- `all / created / assigned` 视图切换
- 状态、模式、关键词筛选
- 删除、关闭、复制
- 详情抽屉
- 在已创建计划中继续关联用例
- 抽屉内快捷单条评审（当前前端暴露通过/拒绝）

### 2.3 创建页已支持

- 分步骤创建评审计划
- 选择用例
- 选择评审人
- 创建计划
- 创建后继续关联用例

### 2.4 详情页已支持

- 查看计划概览
- 关联用例
- 移除用例
- 单条评审
- 批量评审
- 批量重新提审
- 批量改评审人
- 查看评审记录

## 3. 后端交付现状

### 3.1 当前真实接口

后端当前已交付 15 个用例评审接口：

- `GET /projects/{projectID}/case-reviews`
- `POST /projects/{projectID}/case-reviews`
- `GET /projects/{projectID}/case-reviews/{reviewID}`
- `PUT /projects/{projectID}/case-reviews/{reviewID}`
- `DELETE /projects/{projectID}/case-reviews/{reviewID}`
- `POST /projects/{projectID}/case-reviews/{reviewID}/close`
- `POST /projects/{projectID}/case-reviews/{reviewID}/copy`
- `GET /projects/{projectID}/case-reviews/{reviewID}/items`
- `POST /projects/{projectID}/case-reviews/{reviewID}/items/link`
- `POST /projects/{projectID}/case-reviews/{reviewID}/items/unlink`
- `POST /projects/{projectID}/case-reviews/{reviewID}/items/batch-reassign`
- `POST /projects/{projectID}/case-reviews/{reviewID}/items/batch-resubmit`
- `POST /projects/{projectID}/case-reviews/{reviewID}/items/batch-review`
- `POST /projects/{projectID}/case-reviews/{reviewID}/items/{itemID}/review`
- `GET /projects/{projectID}/case-reviews/{reviewID}/items/{itemID}/records`

### 3.2 已交付的核心能力

- 评审计划 CRUD
- 计划复制与关闭
- 评审项关联/移除
- 默认评审人持久化
- 评审项评审人分配
- 单条评审与批量评审
- 批量改评审人、批量重新提审
- 评审记录留痕
- 计划统计与状态自动重算
- testcase 主表评审结果回写

### 3.3 已补齐的关键治理点

当前实现已按代码落地以下治理：

- 单人评审按照最新 `reviewed_at` 提交确定最终结果
- 复制计划会带上计划级 `default_reviewer_ids`
- 更新计划支持持久化 `default_reviewer_ids`
- 评审项批处理会先做 `reviewID + projectID` 归属校验
- 已完成项在改评审人后会重置评审状态并回写 testcase
- 评审记录接口会校验 `itemID` 是否属于当前计划
- 错误码已统一切换到 `1201xx` 数值码

## 4. 用例列表能力调整结果

测试用例列表页当前已从“评审操作台”收敛为“评审摘要与入口页”：

- 保留 `review_result` 展示与筛选
- 保留 `in_review / current_review_id / current_review_name / related_review_count` 投影
- 保留单条/批量加入评审
- 保留查看当前评审
- 保留从用例详情查看当前评审入口
- 不再直接提交评审结论

这意味着：

- 用例列表关注“当前结果”和“当前上下文”
- 评审过程本身进入 `case-review` 页面执行

---

## 5. 已移除或不再推荐的旧能力

以下链路不再作为现行交付口径：

- testcase 维度 `submit-review`
- testcase 维度 `approve-review`
- testcase 维度 `reject-review`
- 用例列表页直接评审
- 用例详情页直接提交评审结论

当前若继续调用上述 legacy 路径，应视为无效调用或历史残留，不应继续接入。

---

## 6. 当前仍需注意的前后端差异

截至当前代码基线，仍有以下“后端已支持 / 前端未完全暴露”的差异：

1. 后端更新计划已支持 `default_reviewer_ids`，当前前端尚无完整编辑计划接入。
2. 后端批量评审支持 `needs_update`，当前前端页面主要暴露 `approved / rejected`。
3. 列表页抽屉中的快捷单条评审当前仅暴露通过/拒绝按钮。

这些差异不影响当前主流程，但需要在后续迭代中继续补齐前端体验。
