> 版本：v1.0 · 日期：2026-05-22 · 状态：可开发版
>

> 设计前提：
> - **部署形态**：多节点（Go 横向扩展），共享 MySQL + Redis
> - **并发目标**：单系统最低支撑 **10 个并发生成任务**
> - **任务交付**：Go 同步提交 → executor 后台执行 → 完成回调 Go
> - **进度推送**：前端轮询（2s）
> - **依赖**：可强依赖 Redis（已部署）

### 1.2 关键组件职责

| 组件 | 职责 | 不做什么 |
|---|---|---|
| **Go API (Gin)** | 路由 / 鉴权 / 业务编排 / 任务状态机 / 配额校验 / 回调接收 / 用例落库 | 不直接调用 LLM、不解析文件 |
| **Executor (FastAPI)** | 文件解析（docx/pdf/md/txt）/ 提示词渲染 / LLM 调用 / 输出 JSON Schema 校验 / 回调 Go | 不直接读写业务 DB（只通过回调） |
| **MySQL** | 持久化业务数据（文档 / 任务 / 产物 / Skill / 用例 / 审计） | 不做高频计数（走 Redis） |
| **Redis** | 项目级并发配额令牌 / 任务分布式锁 / 进度缓存 / 心跳 | 不做唯一数据源，DB 永远是 source of truth |
| **共享文件存储** | 需求文档原始文件 | 不做版本化（每文件唯一 UUID 命名） |

### 1.3 为什么是这套组合

| 选择 | 理由 |
|---|---|
| **多节点 Go + 共享 MySQL + Redis** | 与现有平台架构一致，无需引入新组件 |
| **executor 后台执行 + 回调 Go** | LLM 调用 60–180s，Go 不持有长连接；与现有 AI 脚本任务模式一致 |
| **轮询 2s** | 10 任务 × 5 观察者 × 0.5 qps ≈ 25 qps，Go 完全扛得住；SSE 在多节点部署需要事件总线，收益不抵成本 |
| **Redis 强依赖** | 配额限流必须强一致 + 高并发，DB 行锁性能不够；任务锁需要可重入 + 自动续期 |

## 二、并发场景与防护机制

本节穷举本模块下所有并发冲突点，每个场景都标注防护手段、实现位置、失败行为。这是整篇技术架构的核心，**架构图怎么变都不能漏掉这些防护**。

### 2.1 场景全景表

| # | 场景 | 防护手段 | 实现位置 | 失败行为 |
|---|---|---|---|---|
| A | 两人同时上传同名文档 | 允许同名，DB 不做 title unique | — | 正常入库，前端列表按 id 区分 |
| B | 两人同时基于同一需求文档发起任务 | 允许，每次新建独立 task；产物互不干扰 | — | 正常 |
| C | 项目并发 RUNNING 任务超上限 | **Redis 信号量 + Lua 原子扣减** | Service 层，创建任务时 | 超限直接拒绝创建（错误码 130103），不排队 |
| D | 两人同时采纳同一条产物 | **DB 行锁 + 唯一约束** | `requirement_gen_results.adopted` 字段 + 事务 | 后一个采纳收到 130301，前端提示"已被他人采纳" |
| E | 采纳进行中时另一人编辑产物 | 产物表加 `lock_version` 乐观锁字段 | Service 层 CAS 更新 | 编辑失败，提示"产物已变更，请刷新" |
| F | 同一任务被两人同时 rerun / 丢弃 | 状态机 CAS：`UPDATE ... WHERE status=? AND id=?` | Repository 层 | affected_rows=0 时返回 130102 |
| G | 两个 admin 同时编辑同一个 Skill | Skill 表加 `version` 乐观锁字段 | Service 层 CAS | 后写入收到 130204，提示重试 |
| H | 删除需求文档时另一人正在基于它发起任务 | 软删 + 创建任务时校验 `deleted_at IS NULL` | Service 层 | 已运行任务不中断；新建任务返回 130001 |
| I | Skill 被删后已采纳用例还能查溯源 | 软删 Skill，ID 不复用；用例侧 LEFT JOIN | Repository 层 | 显示"Skill 已删除（ID #N）" |
| J | LLM 调用期间 DB 事务长时间持有 | **LLM 调用一定在事务外**；状态变更与 LLM 调用拆分为多个短事务 | Service 层强约束 | — |
| K | executor 崩溃 / 重启，任务停在 RUNNING | **任务心跳 + 超时回收 worker** | Go 后台 goroutine，每 30s 扫描 | 超时（默认 5 分钟无心跳）自动转 FAILED |
| L | 多 Go 节点谁来跑超时回收 worker | **Redis 分布式锁选主**（30s 过期 + 续期） | Go 启动时尝试获取锁 | 同一时刻仅一个节点跑 worker，节点宕机 30s 后另一节点接手 |
| M | LLM 厂商限流（429） | Executor 重试策略：网络层 1 次，业务层不重试（已在需求方案 §10.1 决议） | Executor 内部 | 业务层错误立即 FAILED |
| N | 回调 Go 时 Go 节点已宕机 | Executor 回调失败时**写回调队列**（Redis List），定时重试 3 次 | Executor 侧 | 3 次失败则任务永久 FAILED，需人工 rerun |
| O | 同一文档同时被两人触发重新解析 | 文档表 `parse_status` 转 `parsing` 为 CAS | Repository 层 | 重复请求被吞，返回当前 parsing 状态 |
| P | 文件上传时磁盘满 | 上传前预检 + 上传失败回滚记录 | API 层 | 返回 130003 |

### 2.2 关键并发防护机制详解

#### 2.2.1 项目级配额限流（场景 C）

**目的**：每个项目同时最多 3 个 RUNNING 任务，全局最多 10 个 RUNNING 任务。

**实现**：Redis 信号量 + Lua 脚本原子操作。

```
KEY 设计：
- reqgen:quota:project:{project_id}    项目级信号量（初值 3）
- reqgen:quota:global                    全局信号量（初值 10）

Lua 脚本（原子扣减两层配额）:
  local proj_key = KEYS[1]
  local global_key = KEYS[2]
  local proj_limit = tonumber(ARGV[1])
  local global_limit = tonumber(ARGV[2])
  local task_id = ARGV[3]
  local ttl = tonumber(ARGV[4])  -- 任务最大执行时长 + 缓冲，如 600s

  local proj_count = tonumber(redis.call('GET', proj_key) or '0')
  local global_count = tonumber(redis.call('GET', 'reqgen:quota:global') or '0')

  if proj_count >= proj_limit then return {0, 'project_quota_exceeded'} end
  if global_count >= global_limit then return {0, 'global_quota_exceeded'} end

  redis.call('INCR', proj_key)
  redis.call('EXPIRE', proj_key, ttl)
  redis.call('INCR', 'reqgen:quota:global')
  redis.call('EXPIRE', 'reqgen:quota:global', ttl)

  -- 同时记录任务持有的配额，便于回收
  redis.call('SADD', 'reqgen:running:tasks', task_id)
  redis.call('HSET', 'reqgen:task:' .. task_id, 'project_id', ARGV[5])

  return {1, 'acquired'}
```

**释放配额**：任务完成（SUCCESS / FAILED）或被超时 worker 回收时调用 release Lua 脚本，DECR 两个计数器并 SREM。

**Redis 不可用时降级**：连接失败时退回 DB 行级计数（`SELECT COUNT(*) FROM requirement_gen_tasks WHERE project_id=? AND status='RUNNING' FOR UPDATE`），但性能下降，仅作为降级保障。

#### 2.2.2 产物采纳的双写防护（场景 D）

**目的**：多人同时采纳同一产物，必须保证只生成 1 条用例。

**实现**：DB 行锁 + 唯一约束。

```sql
-- 采纳事务伪代码：
BEGIN;
  SELECT id, adopted, adopted_case_id
    FROM requirement_gen_results
   WHERE id = ?
     FOR UPDATE;

  -- 已被采纳：返回 130301
  IF adopted = 1 THEN ROLLBACK; RETURN error 130301; END IF

  -- 创建用例
  INSERT INTO testcases (...) VALUES (...);
  SET @new_case_id = LAST_INSERT_ID();

  -- 标记产物已采纳
  UPDATE requirement_gen_results
     SET adopted = 1,
         adopted_case_id = @new_case_id,
         adopted_at = NOW(),
         adopted_by = ?
   WHERE id = ?;
COMMIT;
```

**批量采纳**：循环单条采纳，每条独立事务；前端展示"成功 X / 失败 Y"汇总。**不做整批回滚**，避免一条失败拖累整批。

#### 2.2.3 状态机的 CAS 推进（场景 F、O）

**目的**：所有任务/文档状态变更通过 `UPDATE ... WHERE id=? AND status=?` 实现，affected_rows=0 视为状态已变。

**示例**：

```sql
-- 启动任务（PENDING → RUNNING）
UPDATE requirement_gen_tasks
   SET status = 'RUNNING',
       started_at = NOW(),
       updated_at = NOW()
 WHERE id = ?
   AND status = 'PENDING';

-- 触发解析（not_parsed/parse_failed → parsing）
UPDATE requirement_docs
   SET parse_status = 'parsing',
       updated_at = NOW()
 WHERE id = ?
   AND parse_status IN ('not_parsed', 'parse_failed');
```

**好处**：天然避免重复触发，无需额外加锁。

#### 2.2.4 任务心跳与超时回收（场景 K、L）

**机制**：

1. **心跳写入**：Executor 在执行过程中每 30s 调用 Go 的 `PUT /api/v1/internal/requirement-gen/tasks/{task_id}/heartbeat`，Go 更新 `last_heartbeat_at` 字段。
2. **超时扫描 worker**：选主成功的 Go 节点每 30s 扫描一次：
   ```sql
   SELECT id FROM requirement_gen_tasks
    WHERE status = 'RUNNING'
      AND last_heartbeat_at < NOW() - INTERVAL 5 MINUTE
    LIMIT 50;
   ```
3. **回收动作**：超时任务转 FAILED，`fail_reason = '[TIMEOUT] executor heartbeat lost'`，释放 Redis 配额。

**选主**：Redis SET NX EX：
```
SET reqgen:worker:leader {node_id} NX EX 60
```
持有锁的节点每 30s 续期；进程异常退出后锁 60s 内过期，其他节点接手。

#### 2.2.5 回调可靠交付（场景 N）

**问题**：Executor 完成 LLM 调用后回调 Go，但 Go 节点可能宕机或网络抖动。

**实现**：

1. Executor 调用 `POST /api/v1/internal/requirement-gen/tasks/{task_id}/callback`（无 JWT，用 X-Internal-Token 鉴权）。
2. 失败时（HTTP 非 2xx 或超时）写入 Redis List `reqgen:callback:retry:{task_id}`，保存完整请求体。
3. Executor 内部 retry worker 每 60s 扫描 List，重试 3 次（间隔 1min/5min/15min）。
4. 全部失败后，任务依然在 Go 侧 `RUNNING` → 由 Go 侧超时回收 worker 兜底转 FAILED。

**幂等**：回调接口必须幂等。Go 侧检查任务当前状态：

- `RUNNING` → 接受回调，更新为 SUCCESS/FAILED
- 已是 SUCCESS/FAILED → 返回 200 + `{accepted: false, reason: 'already_terminal'}`，Executor 不再重试
- 其他 → 返回 409

```
   │                        │   ELSE:                                 │
   │                        │     INSERT testcase (draft, source_*)  │
   │                        │     UPDATE result.adopted=1            │
   │                        │   COMMIT                                │
   │                        │ 汇总 task.adopted_count                  │
   │                        │ 若全采纳/丢弃 → task 状态转 FULLY_CLOSED  │
   │                        │   或 PARTIAL_ADOPTED                     │
   │ ←── {adopted:[ids], skipped:[reasons]}                          │
```

## 五、Redis Key 设计与生命周期

| Key | 类型 | TTL | 用途 | 写入方 | 释放方 |
|---|---|---|---|---|---|
| `reqgen:quota:project:{pid}` | STRING (int) | 600s | 项目并发配额计数 | Service.CreateTask | Service.FinalizeTask + worker 回收 |
| `reqgen:quota:global` | STRING (int) | 600s | 全局并发配额计数 | 同上 | 同上 |
| `reqgen:running:tasks` | SET | 600s | 当前 RUNNING 任务 ID 集合 | 同上 | 同上 |
| `reqgen:task:{task_id}` | HASH | 600s | 任务持有的配额元信息（project_id 等） | 同上 | 同上 |
| `reqgen:worker:leader` | STRING (node_id) | 60s | worker 选主 | Go 启动 + 30s 续期 | 进程退出 / TTL 过期 |
| `reqgen:callback:retry:{task_id}` | LIST | 24h | 回调失败重试队列 | Executor | Executor retry worker |
| `reqgen:progress:{task_id}` | STRING (json) | 30s | 进度缓存，前端轮询命中 | Executor 心跳 | TTL 过期 |

**全部 Key 都有 TTL**：极端情况下（Redis 重启 / 节点崩溃）数据可重建，DB 始终是 source of truth。

## 六、Executor 端要点

### 6.1 文件解析

| 格式 | 库 | 备注 |
|---|---|---|
| .docx | python-docx | 提取段落 + 表格内容，忽略图片 |
| .pdf | pdfplumber | 仅文本层，扫描件不支持 |
| .md | 直接读 | 保留原始 markdown |
| .txt | 直接读 | UTF-8，自动探测编码（chardet） |

**统一输出**：纯文本字符串 + 字数。所有格式都先归一化为纯文本再供 LLM。

**超时**：单文件解析 60s 超时，超时返回 `parse_failed` + 原因。

### 6.2 LLM 调用

- 复用现有 `ai_model_configs.is_active=true` 的配置（executor 启动时同步缓存）
- 提示词模板渲染：用 Python `string.Template` 渲染 `{{requirement_text}}` `{{project_context}}` `{{few_shot_examples}}`（当前为空串）
- 输出强制要求 JSON 格式（`response_format={"type":"json_object"}` 或类似）
- JSON Schema 校验：使用 `jsonschema` 库，校验失败立即 FAILED（不重试，业务层错误）
- 重试策略：
  - 网络层错误（5xx / timeout / connection reset）：自动重试 1 次，间隔 2s
  - 业务层错误（4xx / JSON 不合法 / Schema 不通过）：不重试

### 6.3 回调 Go

```python
# pseudo-code
async def callback(task_id, result_payload):
    url = f"{GO_BASE}/api/v1/internal/requirement-gen/tasks/{task_id}/callback"
    headers = {"X-Internal-Token": INTERNAL_TOKEN}
    for attempt in range(3):
        try:
            r = await http.post(url, json=result_payload, headers=headers, timeout=10)
            if r.status_code in (200, 409):  # 200 ok / 409 已终态，无需重试
                return
        except Exception:
            pass
        await asyncio.sleep([60, 300, 900][attempt])  # 1min/5min/15min
    # 三次失败：写 Redis 重试队列，由独立 retry worker 继续兜底
    await redis.lpush(f"reqgen:callback:retry:{task_id}", json.dumps(result_payload))
```

### 6.4 心跳

LLM 调用前后各发一次心跳，调用进行中每 30s 一次（asyncio.create_task 后台心跳协程）。

## 七、Go Worker 设计

### 7.1 启动时序

```go
// cmd/server/main.go 内启动部分（伪代码）
leaderLock := leaderlock.New(redis, "reqgen:worker:leader", nodeID, 60*time.Second)
go func() {
    for {
        acquired, _ := leaderLock.TryAcquire(ctx)
        if acquired {
            go leaderLock.AutoRenew(ctx)   // 每 30s 续期
            runReqGenTimeoutWorker(ctx)    // 长循环
        }
        time.Sleep(30 * time.Second)
    }
}()
```

### 7.2 超时回收 worker

```go
func runReqGenTimeoutWorker(ctx context.Context) {
    ticker := time.NewTicker(30 * time.Second)
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            tasks := repo.ListTimeoutTasks(5 * time.Minute, 50)
            for _, t := range tasks {
                if err := svc.MarkFailed(t.ID, "[TIMEOUT] executor heartbeat lost"); err == nil {
                    quotaLimiter.Release(t.ID)
                }
            }
        }
    }
}
```

### 7.3 节点宕机处理

- 选主锁 60s 过期，其他节点 30s 内尝试一次，最坏 90s 后 worker 恢复
- 期间运行中的任务**继续被 executor 处理**，回调不受影响
- 仅"超时回收"功能短暂不可用，可接受

## 八、安全与可观测

### 8.1 安全

| 风险点 | 措施 |
|---|---|
| 内部回调接口暴露 | `X-Internal-Token` 鉴权，token 从环境变量注入；不走 JWT |
| 上传文件 RCE | 仅按扩展名白名单接受 + executor 内部禁止执行 |
| 文件路径穿越 | UUID 文件名，绝不使用用户传入的文件名拼接路径 |
| LLM 输出 XSS | 用例落库时保持纯文本，前端渲染默认转义；如步骤含富文本走 TipTap 已有的清洗 |
| 配额绕过 | 配额检查必须在创建任务的事务**之外**先做，避免事务期间 Redis 失败而漏扣 |
| LLM API Key 泄漏 | 复用现有 `ai_model_configs.APIKey`，Go 侧不向前端返回，executor 通过激活同步通道获取 |

### 8.2 可观测

- 任务表已采集 `prompt_tokens` / `completion_tokens` / `duration_ms` / `model_snapshot`
- slog 记录关键事件：`task_created / task_started / callback_received / task_completed / task_failed / quota_acquired / quota_released / timeout_recycled / leader_acquired / leader_lost`
- 每条日志带 `request_id` + `task_id` + `project_id`，便于全链路追踪
- 失败任务的 `fail_reason` 必须带类别前缀（`[NETWORK]` / `[BUSINESS]` / `[TIMEOUT]` / `[SCHEMA]`），便于统计

## 九、降级与故障应对

| 故障 | 影响 | 应对 |
|---|---|---|
| Redis 宕机 | 配额限流 / 选主 / 进度缓存失效 | 配额降级到 DB `SELECT COUNT(*) FOR UPDATE`；选主跳过；进度走 DB |
| Executor 全部宕机 | 新任务无法启动；已 RUNNING 任务最终被超时回收 | Go 侧创建任务前预检 executor 健康（`/health`），失败直接 503 |
| LLM 上游限流 | 部分任务 FAILED | 用户重跑；可在系统管理观察失败率，决定是否切换激活模型 |
| Go 单节点宕机 | 该节点上的 in-flight 请求中断；任务表已落 PENDING/RUNNING 的不受影响 | 配额由 TTL 自动回收；回调到旧节点失败时 executor 重试，最终命中其他节点（前提：使用负载均衡） |
| MySQL 主从切换 | 短暂不可写 | Go GORM 已有重试 |
| 共享文件存储不可用 | 上传失败 / 已上传文件不可读 | 上传返回 130003；解析时返回 130004 |
