# 测试管理平台-测试智编模块脚本编排工程化需求方案

> 版本：v0.1（创建于 2026-06-09）
> 项目：Aisight（后端 TestPilot / 前端 TestFront）
> 适用对象：产品、前端、后端、测试、架构
> 文档定位：用于明确测试智编从“录制生成脚本”升级为“固定业务场景生产 + 脚本编排 + 断言资产 + 工程化执行”的完整需求与前后端落地方案。

## 1. 背景与问题

当前测试智编模块已经支持：

- 录制增强模式，通过 Playwright Codegen 采集真实操作。
- 将录制稿转换为结构化步骤模型。
- 通过 AI 按工程化架构重写 Playwright 脚本。
- 支持生成后执行验证、编辑脚本、确认脚本、导出脚本。
- 支持多项目工程化文件产出，包括 `spec / page / shared / fixture / registry` 等文件。

但当前能力仍偏“单次录制生成脚本”，存在以下工程化问题：

1. 固定业务场景无法稳定复用，复杂脚本容易重复生成大量相似代码。
2. 复杂录制场景不能自动识别已有固定业务场景并进行引用组合。
3. 生成后的脚本虽然能执行，但缺少编排层，人工或 AI 修改容易变成直接改最终代码，后续难以重新生成和维护。
4. 自定义断言能力不足，断言没有作为独立资产沉淀。
5. 缺少固定场景、复杂编排、断言、生成代码、执行验证之间的完整资产闭环。

因此需要新增“脚本编排 / 测试场景编排”能力，并重新定义当前测试智编模块的产品边界。

## 2. 总体定位

建议将测试智编拆成三个核心资产能力：

| 模块 | 定位 | 主要职责 |
|---|---|---|
| 测试智编录制任务 | 固定业务场景生产入口 | 录制、AI 重写、执行验证、发布为固定场景 |
| 固定业务场景库 | 可复用流程资产库 | 管理登录、进入页面、创建任务等可复用业务流程 |
| 测试场景编排 | 复杂脚本组装工作台 | 引用固定场景、配置参数、插入断言、生成工程化代码、执行验证 |

目标不是继续让 AI 每次从零生成完整脚本，而是形成：

```text
录制任务 -> 固定业务场景 -> 场景编排 -> 工程化 Playwright 代码 -> 执行验证 -> 资产沉淀
```

## 3. 核心需求

### 3.1 固定业务场景

固定业务场景是由录制任务治理后发布出来的可复用流程资产。它不是普通代码片段，必须包含结构化契约。

固定场景应支持：

- 从已验证通过的测试智编任务发布。
- 手动新建或从已有脚本版本提取。
- 参数化入参，例如账号、项目、任务名称、目标 URL。
- 定义出参，例如任务名称、任务 ID、页面状态。
- 定义前置条件，例如“已登录”“已进入项目”。
- 定义后置条件，例如“任务创建成功”“页面跳转到列表页”。
- 维护版本号、发布状态、最近验证状态、成功率。
- 被其他固定场景或复杂编排引用。
- 支持引用关系和影响分析。

### 3.2 测试场景编排

测试场景编排用于创建复杂自动化测试场景。它不直接以一整段 Playwright 代码为核心，而是以编排步骤为核心。

编排步骤类型包括：

| 类型 | 说明 |
|---|---|
| FLOW_CALL | 引用固定业务场景 |
| ASSERTION | 引用断言资产 |
| ATOMIC_ACTION | 原子操作，例如点击、输入、等待、跳转 |
| CODE_BLOCK | 高级自定义代码块 |
| AI_GENERATED | AI 生成但尚未沉淀的临时步骤 |

编排能力应支持：

- 新建复杂场景。
- 添加固定业务场景引用。
- 添加断言。
- 添加原子操作或自定义代码块。
- 调整步骤顺序。
- 配置每一步参数。
- 将前一步输出映射给后一步输入。
- 生成 DSL。
- 由 DSL 编译为 Playwright 工程代码。
- 执行验证。
- 保存版本。
- AI 推荐组合方案。

### 3.3 自定义断言库

断言应作为独立资产管理，不应全部散落在脚本代码中。

首期建议支持以下断言：

- 元素可见。
- 元素不可见。
- 文本存在。
- URL 包含。
- 表格存在指定行。
- 表格某列值等于。
- 任务状态等于。
- API 响应字段等于。
- 自定义 TypeScript 断言代码。

断言资产应包含：

- 断言名称。
- 断言类型。
- 适用项目 / 模块 / 页面。
- 参数 Schema。
- 失败提示模板。
- 证据采集配置。
- 断言实现代码或模板。
- 版本号。
- 最近验证状态。
- 是否允许 AI 自动推荐。

### 3.4 AI 自动组合

AI 不应直接改最终 Playwright 代码，而应输出“编排计划”。

推荐流程：

```text
复杂录制结果
  -> 标准化步骤 DSL
  -> 检索固定业务场景库
  -> 检索断言库
  -> AI 生成组合计划
  -> 用户确认
  -> 生成编排场景
  -> 编译 Playwright 代码
  -> 执行验证
```

AI 输出示例：

```json
{
  "steps": [
    { "type": "FLOW_CALL", "flow_key": "login", "confidence": 0.94 },
    { "type": "FLOW_CALL", "flow_key": "open_ai_script_page", "confidence": 0.88 },
    { "type": "FLOW_CALL", "flow_key": "create_ai_script_task", "confidence": 0.82 },
    { "type": "ASSERTION", "assertion_key": "task_visible", "confidence": 0.86 }
  ]
}
```

## 4. 后端设计

后端必须按 Handler / Service / Repository 三层实现，所有 Service / Repository 方法首参为 `context.Context`，多表写入必须使用事务并显式传递 `tx`。

### 4.4 状态机

固定场景状态：

```text
DRAFT -> PUBLISHED -> ARCHIVED
```

编排场景状态：

```text
DRAFT -> GENERATED -> VALIDATING -> PASSED -> PUBLISHED
                         ↓
                       FAILED
```

脚本修改来源建议扩展：

```text
AUTO_GENERATED
AI_PATCHED
MANUAL_PATCHED
LOCKED
NEEDS_REVIEW
```

## 5. 前端设计

### 5.1 导航结构

建议测试智编下增加二级入口：

```text
测试智编
  - 录制任务
  - 固定场景库
  - 场景编排
  - 断言库
```

当前 `/ai-script` 保留为录制任务列表。新增建议路由：

```text
/ai-script/flows
/ai-script/compositions
/ai-script/compositions/:compositionId
/ai-script/assertions
```

### 5.2 场景编排列表页

列表页用于管理复杂编排场景。

布局：

```text
顶部：标题 + 新建编排 + AI 从录制任务生成
筛选区：项目 / 状态 / 关键词 / 最近验证状态
主体：编排场景表格
底部：分页
右侧或弹窗：新建编排表单
```

表格列建议：

- 场景名称。
- 所属项目。
- 引用固定场景数量。
- 断言数量。
- 状态。
- 最近验证状态。
- 创建人。
- 更新时间。
- 操作：查看、编辑、生成代码、验证、发布、废弃。

### 5.3 场景编排详情页

建议使用三栏工作台布局。

```text
┌────────────────────────────────────────────┐
│ 顶部：返回 / 场景名称 / 状态 / 保存 / AI 优化 / 生成代码 / 验证 / 发布 │
├───────────────┬────────────────┬───────────┤
│ 左侧资产库     │ 中间编排步骤区   │ 右侧配置区 │
│ 固定场景       │ 步骤列表         │ 参数配置   │
│ 断言库         │ 拖拽排序         │ 输出映射   │
│ 原子操作       │ 错误提示         │ 版本选择   │
├───────────────┴────────────────┴───────────┤
│ 底部：生成代码 / 执行日志 / AI 建议 / 引用关系 │
└────────────────────────────────────────────┘
```

#### 左侧资产库

- 固定场景 Tab。
- 断言 Tab。
- 原子操作 Tab。
- 支持搜索、标签筛选、最近使用。
- 每个资产卡片展示名称、描述、版本、最近验证状态。

#### 中间编排区

- 步骤列表式编排，首期不强制做画布。
- 支持拖拽排序。
- 支持新增步骤、删除步骤、禁用步骤。
- 每步显示类型、名称、版本、参数完整性、验证结果。
- AI 推荐步骤需要显示置信度与“采纳/忽略”。

#### 右侧配置区

点击步骤后展示：

- 步骤基本信息。
- 引用资产版本。
- 入参表单。
- 出参定义。
- 参数映射。
- 失败处理策略。
- 是否采集截图、日志、DOM 证据。

#### 底部面板

- DSL 预览。
- 生成代码预览。
- 执行日志。
- 验证历史。
- AI 修改建议。
- 引用关系。

### 5.4 固定场景库页面

布局：

```text
顶部：固定场景库标题 + 新建 + 从录制任务发布
筛选区：项目 / 标签 / 状态 / 最近验证状态
主体：卡片或表格
详情抽屉：契约、版本、代码、引用关系、验证记录
```

### 5.5 断言库页面

布局：

```text
顶部：断言库标题 + 新建断言
左侧：断言类型筛选
主体：断言列表
右侧：断言参数 Schema、实现模板、失败提示、证据配置
```

## 6. 落地阶段

### 阶段一：固定场景发布与引用

- 当前测试智编任务详情增加“发布为固定场景”。
- 新增固定场景库。
- 后端完成 `ai_flow_asset / ai_flow_asset_version`。
- 编译代码时支持固定场景 import/call。

### 阶段二：场景编排 MVP

- 新增场景编排列表页和详情页。
- 支持步骤列表式编排。
- 支持引用固定场景。
- 支持参数配置。
- 支持生成 Playwright 代码。

### 阶段三：断言库

- 新增断言资产表和断言库页面。
- 编排步骤支持插入断言。
- 执行验证时记录断言结果。

### 阶段四：AI 编排建议

- 从复杂录制任务生成编排建议。
- AI 推荐固定场景和断言。
- 用户确认后生成编排场景。

### 阶段五：版本、影响分析与 AI 修复

- 支持引用关系图。
- 修改固定场景时展示影响范围。
- 支持 AI 基于失败日志生成修复建议。
- 支持 Diff、回滚、锁定人工修改。

## 7. 验收标准

### 7.1 后端验收

- 新增表结构满足固定场景、断言、编排、引用关系管理。
- 所有接口具备 Swagger 注解。
- Handler 只做参数校验和鉴权，Service 负责业务规则，Repository 只做数据访问。
- 多表写入使用事务。
- 支持固定场景发布、编排保存、代码生成、执行验证、引用分析。
- 支持项目级隔离和 RBAC。

### 7.2 前端验收

- 新增固定场景库、场景编排、断言库入口。
- 场景编排详情页三栏布局清晰，支持高信息密度工作台体验。
- 步骤可添加、排序、删除、配置参数。
- 固定场景和断言可以被引用。
- 生成代码、执行日志、验证结果可查看。
- AI 推荐步骤必须可解释、可采纳、可忽略。

### 7.3 业务验收

- 用户可以从一个录制任务发布固定场景。
- 用户可以用多个固定场景组合一个复杂测试场景。
- 用户可以插入自定义断言。
- 编排场景可以生成 Playwright 工程代码。
- 生成代码可以触发执行验证。
- 修改固定场景可以看到影响哪些编排场景。

## 8. 关键原则

1. 先工程化，再智能化。
2. 先支持人工确认，再逐步提升 AI 自动化。
3. AI 输出编排计划，不直接无确认覆盖最终代码。
4. 固定场景必须是结构化资产，不是普通代码片段。
5. 断言必须独立资产化，不能全部散落在脚本里。
6. 所有生成、修改、发布都必须可验证、可回滚、可追溯。

## 9. DSL 标准

DSL 是场景编排的唯一源数据，Playwright 工程代码是 DSL 编译后的产物。后端、前端、AI 编排和代码生成器必须共同遵守同一份 DSL Schema。

### 9.1 DSL 顶层结构

```json
{
  "schema_version": "1.0",
  "scenario": {
    "scenario_key": "create_ai_script_task_and_verify",
    "scenario_name": "创建测试智编任务并验证",
    "project_id": 1
  },
  "variables": {
    "taskName": {
      "type": "string",
      "default": "自动化任务",
      "description": "测试智编任务名称"
    }
  },
  "env": ["ADMIN_USER", "ADMIN_PASSWORD", "BASE_URL"],
  "steps": [],
  "policies": {
    "failure_strategy": "STOP_ON_FAILURE",
    "retry": { "enabled": false, "times": 0, "interval_ms": 0 },
    "evidence": { "screenshot": "ON_FAILURE", "trace": true, "dom_snapshot": false }
  },
  "ai_meta": {
    "generated_by_ai": false,
    "plan_id": "",
    "confidence": 0,
    "explanation": ""
  }
}
```

### 9.2 步骤结构

所有步骤必须具备稳定 `id`，禁止仅依赖顺序号识别步骤。

```json
{
  "id": "step_001",
  "name": "登录系统",
  "type": "FLOW_CALL",
  "enabled": true,
  "ref": {
    "flow_id": 12,
    "flow_version_id": 30,
    "flow_key": "login"
  },
  "inputs": {
    "username": "${env.ADMIN_USER}",
    "password": "${env.ADMIN_PASSWORD}"
  },
  "outputs": {
    "loginUser": "$.user",
    "token": "$.auth.token"
  },
  "depends_on": [],
  "failure_strategy": "INHERIT",
  "retry": { "enabled": false, "times": 0, "interval_ms": 0 },
  "evidence": { "screenshot": "INHERIT", "trace": true },
  "ai_meta": {
    "recommended": true,
    "confidence": 0.94,
    "reason": "录制步骤与已发布登录固定场景高度匹配"
  }
}
```

步骤类型规则：

| 类型 | 必填字段 | 说明 |
|---|---|---|
| FLOW_CALL | ref.flow_id、ref.flow_version_id、inputs | 引用已发布固定业务场景 |
| ASSERTION | ref.assertion_id、inputs | 引用已发布断言资产 |
| ATOMIC_ACTION | action、selector / url / value | 原子点击、输入、跳转、等待等动作 |
| CODE_BLOCK | code_block、language | 高级自定义代码块，默认需要人工确认 |
| AI_GENERATED | code_block 或 action、ai_meta | AI 临时步骤，发布前必须转为其他确定类型或人工锁定 |

### 9.3 参数引用语法

| 语法 | 含义 | 示例 |
|---|---|---|
| `${variables.xxx}` | 当前编排变量 | `${variables.taskName}` |
| `${env.xxx}` | 执行环境变量 | `${env.ADMIN_USER}` |
| `${steps.step_id.outputs.xxx}` | 前置步骤输出 | `${steps.step_001.outputs.token}` |
| `${literal.xxx}` | 字面量引用 | `${literal.enabled}` |
| `$.field.path` | 当前步骤结果 JSONPath | `$.response.data.id` |

规则：

- 环境变量只允许引用白名单中的 key，禁止在 DSL 中保存明文密码、Token、API Key。
- 引用前置步骤输出时，后端必须校验 `depends_on` 或步骤顺序，禁止引用未来步骤。
- 参数缺失时，前端必须展示字段级错误，后端必须返回结构化校验错误。
- DSL 保存前必须通过 JSON Schema 校验；执行前必须再次做权限、引用、版本和参数校验。

### 9.4 失败、重试和证据策略

`failure_strategy` 支持：

| 值 | 说明 |
|---|---|
| STOP_ON_FAILURE | 当前步骤失败后立即终止整个编排 |
| CONTINUE_ON_FAILURE | 当前步骤失败后继续执行后续步骤 |
| SOFT_ASSERT | 仅断言步骤可用，记录失败但不终止 |
| INHERIT | 继承顶层策略 |

`evidence.screenshot` 支持：

| 值 | 说明 |
|---|---|
| ALWAYS | 每步都截图 |
| ON_FAILURE | 失败时截图 |
| NEVER | 不截图 |
| INHERIT | 继承顶层策略 |

发布状态为 `PUBLISHED` 的编排禁止包含未确认的 `AI_GENERATED` 步骤；如存在 `CODE_BLOCK`，必须标记 `manual_reviewed=true` 并记录审核人。

## 10. 固定场景发布详细流程

固定场景必须从“已验证、可参数化、可复用”的脚本资产发布，禁止把未经治理的代码片段直接沉淀为固定场景。

### 10.1 从录制任务发布

流程：

```text
录制任务详情点击“发布为固定场景”
  -> 校验任务存在且属于当前项目
  -> 校验最新脚本版本已验证通过
  -> 弹窗填写名称、flow_key、描述、标签
  -> 系统分析可参数化字段
  -> 用户确认入参、出参、前置条件、后置条件
  -> 生成 flow DSL
  -> 创建 ai_flow_asset 和 ai_flow_asset_version
  -> 写入 ai_asset_reference
  -> 记录审计日志
```

### 10.2 发布前校验

发布固定场景必须满足：

- 来源脚本版本最近一次验证状态为 `PASSED`。
- `flow_key` 在同一项目内唯一，且只能包含小写字母、数字和下划线。
- 入参、出参 Schema 必须完整，必填参数必须有说明。
- 前置条件和后置条件必须至少填写业务说明。
- DSL 必须通过 Schema 校验。
- 生成代码必须可编译。
- 发布操作必须具备项目写权限；若配置为仅管理员/项目经理发布，则开发者只能保存草稿。

### 10.3 参数抽取原则

优先抽取以下字段为入参：

- 用户账号、项目、环境、基础 URL。
- 业务名称、编号、状态、时间、标签等测试数据。
- 选择器中稳定业务文本。
- API 请求中的业务参数。

禁止抽取：

- 密码、Token、API Key 的明文值。
- 运行时临时 ID，除非明确作为上游步骤输出。
- 不稳定 DOM 节点序号。

### 10.4 版本规则

- 首次发布生成 `version_no=1`。
- 修改已发布固定场景时，必须生成新版本，不允许原地覆盖已发布版本。
- 已被编排引用的版本禁止物理删除，只能 `ARCHIVED`。
- 新版本发布后不自动影响已有编排，编排默认锁定 `flow_version_id`。
- 用户升级引用时必须展示影响分析和差异说明。

## 11. 代码生成规则

### 11.1 基本边界

代码生成必须遵守：

```text
DSL 是源头。
Playwright 工程代码是产物。
默认不从人工修改代码反向解析 DSL。
人工修改生成代码后，编排标记为 MANUAL_PATCHED 或 LOCKED。
LOCKED 状态禁止自动覆盖代码，除非用户显式解除锁定。
```

### 11.2 生成服务职责

`CompositionCodegenService` 负责：

- 校验 DSL Schema、引用版本、参数完整性和权限。
- 将 `FLOW_CALL` 编译为固定场景函数调用。
- 将 `ASSERTION` 编译为断言资产函数调用。
- 将 `ATOMIC_ACTION` 编译为 Playwright 操作。
- 将已审核 `CODE_BLOCK` 插入指定位置。
- 生成工程化文件并写入代码快照。
- 返回生成摘要、文件列表和潜在风险。

### 11.3 文件结构

生成文件应复用当前测试智编工程化结构：

```text
spec/
  <scenario_key>.spec.ts
page/
  <page_object>.ts
shared/
  flows/
    <flow_key>.ts
  assertions/
    <assertion_key>.ts
fixture/
  <project_fixture>.ts
registry/
  flowRegistry.ts
  assertionRegistry.ts
```

首期可以只生成 `spec/<scenario_key>.spec.ts`，并通过 registry import 已发布固定场景和断言；后续再拆分 page/shared 文件。

### 11.4 编译映射

| DSL 步骤 | 生成方式 |
|---|---|
| FLOW_CALL | `await flows.<flow_key>(ctx, inputs)` |
| ASSERTION | `await assertions.<assertion_key>(ctx, inputs)` |
| ATOMIC_ACTION.click | `await page.locator(selector).click()` |
| ATOMIC_ACTION.fill | `await page.locator(selector).fill(value)` |
| ATOMIC_ACTION.goto | `await page.goto(url)` |
| CODE_BLOCK | 插入已审核代码块，并加中文说明注释 |
| AI_GENERATED | 发布前不允许直接编译为正式代码 |

### 11.5 生成后状态

- 生成成功：编排状态从 `DRAFT` 变为 `GENERATED`。
- 验证通过：状态变为 `PASSED`。
- 验证失败：状态变为 `FAILED`，保留失败日志和证据。
- 发布成功：状态变为 `PUBLISHED`，同时生成不可变版本快照。

## 12. 引用与版本治理

### 12.1 引用锁定

- 编排引用固定场景时，必须保存 `flow_id` 和 `flow_version_id`。
- 编排引用断言时，必须保存 `assertion_id`；后续如断言支持版本，也必须锁定 `assertion_version_id`。
- 发布新版本不自动升级引用方。
- 引用方升级版本必须走影响分析、Diff 查看、重新生成代码和验证。

### 12.2 固定场景相互引用

允许固定场景引用固定场景，但必须满足：

- 最大引用深度为 3 层。
- 后端保存引用关系时必须做 DAG 检查，禁止循环依赖。
- 引用链中任一资产为 `ARCHIVED` 或验证失败时，AI 不得自动推荐。
- 运行前必须展开引用链并校验所有版本仍可用。

### 12.3 影响分析

修改、归档或发布新版本时，必须展示：

- 直接引用该资产的编排场景。
- 间接受影响的固定场景和编排场景。
- 当前引用版本、最新版本、状态差异。
- 是否存在已发布编排。
- 建议动作：保持旧版本、升级引用、重新验证、归档替代。

### 12.4 删除与归档

- 已发布或被引用资产禁止物理删除。
- 草稿且无引用资产允许删除。
- `ARCHIVED` 资产不可新增引用，但历史编排可继续锁定旧版本运行。
- 归档操作必须记录审计日志和原因。

## 13. 断言执行模型

### 13.1 断言类型

| 类型 | 说明 | 关键参数 |
|---|---|---|
| UI_VISIBLE | 元素可见 | selector、timeout_ms |
| UI_HIDDEN | 元素不可见 | selector、timeout_ms |
| TEXT_CONTAINS | 页面文本存在 | text、scope_selector |
| URL_CONTAINS | URL 包含 | expected |
| TABLE_ROW_EXISTS | 表格存在指定行 | table_selector、match |
| TABLE_CELL_EQUALS | 表格单元格等于 | table_selector、row_match、column、expected |
| API_FIELD_EQUALS | API 响应字段等于 | response_ref、json_path、expected |
| CUSTOM_CODE | 自定义 TypeScript 断言 | code、param_schema |

### 13.2 断言结果结构

执行验证时，每个断言步骤必须产出结构化结果：

```json
{
  "assertion_id": 5,
  "assertion_name": "任务可见",
  "step_id": "step_004",
  "status": "PASSED",
  "expected": "任务名称存在于列表",
  "actual": "找到匹配行",
  "failure_message": "",
  "screenshot_url": "",
  "trace_url": "",
  "duration_ms": 328
}
```

### 13.3 执行规则

- 一个步骤允许配置多个断言，但建议首期保持一个 `ASSERTION` 步骤对应一个断言资产。
- 默认使用硬断言，失败后按 `failure_strategy` 决定是否中断。
- 软断言必须记录失败并继续执行，最终验证结果为 `FAILED_WITH_SOFT_ASSERTIONS`。
- API 断言只能读取当前步骤或前置步骤显式输出，禁止隐式读取任意全局变量。
- 表格断言必须使用稳定选择器和字段匹配，禁止依赖纯行号。

### 13.4 验证结果补充表

建议新增独立表承载编排验证记录，执行底座复用现有 Playwright Executor 和项目级 workspace lock。

`ai_composition_validation`：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| composition_id | bigint | 编排场景 ID |
| composition_version_id | bigint | 编排版本 ID |
| project_id | bigint | 项目 ID |
| status | varchar(32) | PENDING / RUNNING / PASSED / FAILED / CANCELED |
| executor_job_id | varchar(128) | Executor 任务 ID |
| workspace_id | varchar(128) | 工作区标识 |
| log_url | varchar(500) | 执行日志 |
| report_url | varchar(500) | 测试报告 |
| trace_url | varchar(500) | Playwright Trace |
| started_at / finished_at | datetime | 起止时间 |
| created_by / created_at | - | 审计字段 |

`ai_composition_assertion_result`：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| validation_id | bigint | 验证记录 ID |
| step_id | varchar(64) | DSL 步骤 ID |
| assertion_id | bigint | 断言资产 ID |
| status | varchar(32) | PASSED / FAILED / SKIPPED |
| expected_json | json | 期望值 |
| actual_json | json | 实际值 |
| failure_message | varchar(1000) | 失败说明 |
| evidence_json | json | 截图、Trace、DOM 等证据 |
| duration_ms | int | 耗时 |

## 14. AI 编排约束

AI 只能生成建议，不得无确认修改已发布资产或覆盖最终代码。

### 14.1 AI 输入上下文

AI Planner 输入必须包含：

- 录制步骤摘要和标准化步骤。
- 当前项目可用固定场景列表，仅包含 `PUBLISHED` 且允许 AI 复用的资产。
- 当前项目可用断言列表，仅包含 `PUBLISHED` 且允许 AI 推荐的资产。
- 资产版本、入参 Schema、出参 Schema、前置条件、后置条件。
- 最近验证状态、成功率和标签。
- 当前用户权限和项目上下文。

### 14.2 AI 输出 Schema

```json
{
  "plan_id": "plan_20260609_001",
  "summary": "推荐复用登录、进入测试智编页、创建任务和任务可见断言",
  "confidence": 0.87,
  "steps": [
    {
      "type": "FLOW_CALL",
      "flow_id": 12,
      "flow_version_id": 30,
      "flow_key": "login",
      "confidence": 0.94,
      "reason": "录制中的输入账号和点击登录与登录固定场景匹配",
      "inputs": {
        "username": "${env.ADMIN_USER}",
        "password": "${env.ADMIN_PASSWORD}"
      }
    }
  ],
  "warnings": [
    "创建任务步骤缺少任务类型参数，需要用户确认"
  ]
}
```

### 14.3 采纳规则

- AI 总置信度低于 0.75 时，只展示建议，不允许一键生成编排。
- 单步骤置信度低于 0.8 时，必须要求用户逐条确认。
- AI 不允许引用未发布、已归档、验证失败或用户无权限访问的资产。
- AI 输出必须通过 DSL Schema 校验和资产引用校验。
- AI 计划先进入草稿，不直接发布、不直接覆盖已生成代码。
- AI 修复建议只能生成 Diff 或补丁建议，用户确认后才能应用。

## 15. API DTO 明细

所有 API 必须使用 `{code, message, data}` 响应结构，业务判断使用数字错误码。Handler 必须完成 RBAC 和项目隔离校验，Service 负责业务规则，Repository 只做数据访问。

## 16. 前端交互状态

前端必须遵守分层规则：页面负责编排，复杂逻辑下沉到 composable，接口调用统一走 API 层，远程数据必须暴露加载态、错误态、空态和数据态。

### 16.1 固定场景库

页面状态：

- 加载态：首次进入和筛选变更时显示表格/卡片加载。
- 空态：无固定场景时提供“从录制任务发布”入口。
- 错误态：展示可重试错误，不直接暴露后端技术栈。
- 发布中：发布按钮 loading，禁止重复提交。
- 版本过期：展示“已有新版本，可升级引用”提示。
- 不可用：归档、验证失败、无权限资产必须禁用引用操作。

关键交互：

- 删除或归档必须二次确认。
- 发布弹窗必须展示入参、出参、前置条件、后置条件确认区。
- 引用关系抽屉必须展示直接引用和间接影响。

### 16.2 场景编排详情页

页面状态：

- `loading`：加载编排详情、资产库、验证历史。
- `saving`：保存 DSL 和步骤配置。
- `generating`：生成 Playwright 代码。
- `validating`：触发执行验证和轮询结果。
- `aiPlanning`：生成 AI 编排建议。
- `dirty`：当前 DSL 有未保存修改。
- `locked`：人工修改代码后锁定自动生成。
- `permissionDenied`：无编辑或发布权限。

交互闭环：

- 参数缺失时，步骤卡片和右侧配置区同时提示。
- 拖拽排序后必须标记未保存。
- 删除步骤必须二次确认。
- 生成代码前必须校验 DSL 和参数完整性。
- 验证前必须确保已生成代码，未生成时提示先生成。
- 发布前必须要求最近一次验证通过。
- AI 推荐步骤必须支持采纳、忽略、查看原因。
- 低置信度步骤必须显示警示并要求人工确认。

### 16.3 断言库

页面状态：

- 新建/编辑断言必须有参数 Schema 校验。
- 自定义代码断言必须显示安全提示和审核状态。
- 断言发布前必须通过模板校验或试运行。
- 被引用断言归档时，必须展示影响分析。

### 16.4 代码、日志和验证面板

底部面板至少包含：

- DSL 预览：只读 JSON，支持复制和格式化。
- 生成代码：只读预览，人工编辑需进入明确编辑模式。
- 执行日志：支持刷新、自动滚动、失败定位。
- 验证历史：展示状态、耗时、执行人、报告、Trace、断言结果。
- AI 建议：展示置信度、原因、警告和采纳状态。

### 16.5 可访问性和体验要求

- 所有按钮必须有明确中文动作文案。
- 图标按钮必须具备 tooltip 或 aria-label。
- 表单项必须有 label，必填项必须可感知。
- 状态不能只靠颜色表达，必须有文字或图标说明。
- 工作台必须支持键盘焦点态和基础键盘操作。
- 管理端页面保持高信息密度和清晰扫描效率，禁止营销式大图和无意义动画。

## 17. 研发落地质量门禁

本方案进入研发后，必须按前后端规范执行质量门禁。

### 17.1 后端门禁

- 新增或修改 Go 代码必须遵守 Handler / Service / Repository 分层。
- 所有 Service / Repository 方法首参必须为 `context.Context`。
- 多表写入必须通过事务包裹，并显式传递 `tx`。
- 新增 Handler 必须补完整 Swagger 注解。
- 新增错误场景必须定义 6 位数字错误码。
- Service 层必须补表驱动单元测试，覆盖成功、权限失败、参数失败、状态非法、事务回滚、幂等重复提交。
- 完成后运行 `gofmt`、`go vet`、`golangci-lint`、`gosec`、`go test ./...`、`go build ./cmd/server`。

### 17.2 前端门禁

- 新增页面、组件、样式、交互前必须使用 UI/UX 质量约束。
- 页面只做编排，复杂状态、异步请求、保存、生成、验证、AI 建议逻辑下沉到 composable。
- API 类型必须定义入参和返回值，禁止直接使用 `axios`。
- 远程数据必须覆盖加载态、空态、错误态、数据态。
- 修改 composable、store、utils、复杂组件交互、权限判断、API 参数转换时必须补 Vitest 测试。
- 完成后运行 `npm run lint`、`npm run type-check`、`npm run test`、`npm run build`。

## 18. E2E 全自动化增强方案

本章针对代码评审中识别的两个能力缺口给出详细设计，目标是把"录制 → AI 编排 → 生成 → 验证 → 发布"补齐为可全自动执行的 E2E 闭环：

1. AI Planner 当前为启发式关键词匹配，未真正调用大模型做语义匹配（缺口一）。
2. 固定场景 DSL 编译遇到不支持步骤仅警告跳过，可能产出不完整脚本（缺口二）。

### 18.1 缺口一：AI Planner 接入大模型语义匹配

#### 18.1.1 现状与问题

- `AIPlanFromTask` 基于录制文本与资产名称/描述/Schema 的 Token 重叠打分，置信度是规则估算值，不具备语义理解能力。
- 录制操作与固定场景描述措辞不同（如"输入工号点确定" vs "登录系统"）时无法匹配，复杂场景命中率低。
- 置信度不可信会污染 14.3 的采纳规则：阈值 0.75/0.8 失去统计含义。

#### 18.1.2 目标架构

复用已有 AI 模型配置模块（`ai_model_config` + Python Executor 的 LLM 通道，与脚本生成共用 `syncActiveModelForLLM` 同步的激活模型），新增 LLM Planner 通道，启发式打分降级为兜底与交叉验证：

```text
录制标准化步骤 ──┐
                 ├─→ PlannerContextBuilder ─→ LLM Planner（Executor /execute/plan）─→ Plan 候选
项目资产快照 ────┘                                   │（超时/失败降级）
（PUBLISHED + allow_ai_reuse）                       ↓
                                          HeuristicPlanner（现有逻辑保留）
                                                     ↓
                                        PlanValidator（Schema/引用/置信度校验）
                                                     ↓
                                        草稿编排（DRAFT，不自动发布）
```

#### 18.1.3 Prompt 上下文构造（PlannerContextBuilder）

- 输入一：录制标准化步骤序列（动作类型、目标元素语义名、输入值脱敏后的占位符、页面 URL 路径）。敏感值必须替换为 `${env.*}` 占位，禁止把明文凭据送入模型。
- 输入二：候选资产清单，仅含 `PUBLISHED` 且 `allow_ai_reuse=true` 的固定场景与断言，每项携带 `flow_key`、名称、描述、入参/出参 Schema、前置/后置条件、最近验证状态、标签。
- 输入三：项目环境变量白名单（仅 key，不含值）与参数表达式语法说明（`${env.*}`、`${steps.*.outputs.*}`、`${variables.*}`、`${literal.*}`）。
- 候选资产超过模型上下文预算时，先用现有启发式打分做 Top-N（默认 N=30）预筛，再交给 LLM 精排。

#### 18.1.4 LLM 输出契约与校验（PlanValidator）

- LLM 必须按 14.2 的 JSON Schema 输出（plan_id、summary、confidence、steps[]、warnings[]），后端用严格 Schema 校验，解析失败重试 1 次后降级启发式。
- 逐步校验：引用的 `flow_id`/`flow_version_id`/断言必须存在于候选清单（防幻觉）；inputs 中表达式必须通过现有 DSL 引用校验与敏感明文检测；不满足者剔除该步并写入 warnings。
- 置信度策略：以 LLM 自报置信度为主，与启发式得分加权交叉（建议 0.7×LLM + 0.3×启发式），落库到 `ai_confidence`；14.3 的 0.75/0.8 阈值与人工确认规则不变。
- 无法映射为 FLOW_CALL/ASSERTION 的录制片段，由 LLM 输出 ATOMIC_ACTION 或 AI_GENERATED 占位步骤，仍走现有"AI 临时步骤必须采纳确认后才能生成代码"的约束。

#### 18.1.5 降级与可观测性

- LLM 调用超时（默认 60s）、模型未配置、输出连续 2 次不合法时，自动降级为现有启发式 Planner，plan.summary 中必须注明"启发式降级"。
- 记录操作日志：模型 ID、耗时、token 用量、降级原因、候选数/采纳数，支撑后续准确率统计。
- 新增错误码：模型未配置、LLM 输出非法、planner 超时各占一个 6 位错误码。

#### 18.1.6 API 与数据变更

- 复用 `POST /api/v1/ai-script/compositions/ai-plan`，请求增加可选 `planner_mode`（`auto` 默认 / `llm` / `heuristic`），响应增加 `planner_used` 与 `degraded_reason`。
- `ai_scenario_step` 无表结构变更；操作日志表记录 planner 维度字段（JSON 扩展，不加列）。

### 18.2 缺口二：固定场景发布前 DSL 可编译性强校验

#### 18.2.1 现状与问题

- `compileFlowDSLStatements` 对不支持/字段不全的步骤仅产生 warning 并跳过，引用方在生成编排代码时才发现固定场景是"半截脚本"，且跳过行为静默改变业务语义。
- 仅当固定场景编译结果为空时才报错，部分可编译时照常生成，违反"DSL 是唯一源数据、代码是确定性编译产物"的原则。

#### 18.2.2 设计原则

- 把"可编译性"作为固定场景发布门禁：发布即保证该版本 DSL 100% 步骤可编译，引用方永远拿到完整脚本。
- 校验逻辑与编译器同源：直接复用编译器做 dry-run，避免校验与编译规则漂移。

#### 18.2.3 发布门禁规则（在 10 章发布流程中追加）

1. `Publish` / `PublishFromTask` 时对 DSL 执行 dry-run 编译：任一步骤命中"不支持的步骤类型"或"必填字段缺失"即拒绝发布，错误信息逐条列出步骤序号、类型与原因。
2. 支持的步骤类型清单显式化（NAVIGATE/GOTO、CLICK、INPUT/FILL、SELECT、KEY_PRESS、WAIT、ASSERT/ASSERT_CANDIDATE、FLOW_CALL），文档与编译器以同一清单为准；新增类型必须同步更新编译器与本清单。
3. 存量已发布且含不可编译步骤的资产：不回溯下架，但在资产详情与引用校验中标记 `compile_health=PARTIAL`，编排生成代码时若引用 PARTIAL 资产必须显式确认。
4. 草稿保存阶段不强校验（允许半成品），但保存响应返回 dry-run 警告列表，前端在编辑器内逐条提示。

#### 18.2.4 编排侧一致性

- 编排 `GenerateCode` 时固定场景若 dry-run 失败，从"警告+跳过"改为硬错误（6 位错误码），错误信息定位到 flow_key 与步骤序号。
- 原子操作能力对齐：`compileAtomicAction` 补齐 `select`、`press`、`wait_for`（定位器等待），与固定场景 DSL 编译器能力保持同一清单。

#### 18.2.5 API 变更

- `POST /api/v1/ai-script/flows/:flowID/publish` 失败响应 data 中返回 `compile_failures: [{step_no, step_type, reason}]`。
- 新增 `POST /api/v1/ai-script/flows/:flowID/compile-check`：草稿阶段手动触发 dry-run，返回与发布门禁相同结构的结果，前端"发布前自检"按钮使用。

### 18.3 E2E 全自动化闭环

两项增强落地后，目标闭环为：

1. 录制：Playwright Codegen 产出原始脚本 → 标准化步骤（已有）。
2. 规划：LLM Planner 语义匹配固定场景/断言，生成草稿编排（18.1）。
3. 人工确认：仅低置信度步骤（<0.8）与 AI 临时步骤需要确认，其余自动采纳（已有规则，置信度可信后实际确认量显著下降）。
4. 生成：DSL 编译为完整 Playwright 脚本，引用资产保证可编译（18.2）。
5. 验证：Executor 回放 + 断言结果回写（已有）。
6. 发布：验证 PASSED 后发布版本，资产引用版本锁定（已有）。
7. 回归：已发布编排纳入定时回归执行，失败时 AI 生成修复 Diff 建议、人工确认后应用（复用 14.3 修复建议约束）。

全自动化的度量指标：

| 指标 | 定义 | 目标 |
| --- | --- | --- |
| 复用命中率 | AI 计划中 FLOW_CALL 步骤数 / 可复用步骤总数 | ≥ 70% |
| 计划采纳率 | 用户未修改直接采纳的步骤占比 | ≥ 80% |
| 一次验证通过率 | 生成后首次验证 PASSED 占比 | ≥ 85% |
| 人工干预步骤数 | 单编排平均需人工确认的步骤数 | 持续下降 |

### 18.4 实施阶段

- 阶段一（先行）：18.2 发布门禁 + compile-check API + 原子操作对齐（纯后端确定性逻辑，风险低）。
- 阶段二：18.1 LLM Planner（Executor 新增 /execute/plan、上下文构造、输出校验、降级链路）。
- 阶段三：回归调度与 AI 修复建议闭环，按 18.3 指标建立看板。

各阶段均执行 17 章质量门禁。
