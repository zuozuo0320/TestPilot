# 测试管理平台-测试智编模块团队标准脚本模板规范

> 版本：v0.1（创建于 2026-03-29）
> 关联文档：`测试管理平台-测试智编模块需求方案-20260327.md`、`测试管理平台-测试智编模块详细设计-20260328.md`、`测试管理平台-测试智编模块研发拆解与步骤模型设计-20260329.md`、`测试管理平台-测试智编模块AI重构提示词规范-20260329.md`
> 文档定位：用于统一测试智编模块输出的 `Playwright TypeScript` 标准脚本模板，作为 AI 重构目标模板与人工维护基线
> 2026-04-02 对齐说明：当前主链路已经切换为“多项目、多文件、POM 工程化输出”，本文所述“单文件模板”仅保留为 `pw_workspace` / LLM 不可用 / legacy 兼容场景下的 V0 fallback 参考，主链路请以 `playwright-architecture-design.md` 为准。

## 1. 目标

本文档用于明确测试智编模块首期输出脚本的标准形态，确保：
- AI 重构输出有固定模板可遵循
- 人工查看和维护脚本时风格一致
- 回放验证、版本对比、导出文件的结果可预测

当前适用范围：
- 仅适用于 V0 单文件兼容 fallback
- 文件后缀固定 `.spec.ts`
- 默认用于 `pw_workspace` 临时验证或 LLM 不可用时的兜底输出
- 不拆 Page Object
- 不生成多文件

## 2. 模板原则

V0 fallback 模板遵循以下原则：
- 单文件优先，降低生成复杂度
- 结构清晰，便于 AI 和人工共同维护
- 步骤顺序与 `step_model_json` 保持一致
- 尽量使用稳定 locator
- 保留必要注释，不堆砌说明性废话
- 至少包含 1 个最终结果断言

## 3. 文件级约定

### 3.1 文件命名

建议格式：
- `{biz-action}.spec.ts`
- `{case-key}-{biz-action}.spec.ts`

示例：
- `login-user-list.spec.ts`
- `case2001-login-user-list.spec.ts`

命名要求：
- 使用小写字母和中划线
- 不使用中文文件名
- 不在文件名中写版本号

### 3.2 输出形式

首期固定输出：
- 单个 `.spec.ts` 文件
- 基于 `@playwright/test`
- 不依赖额外业务框架文件

首期不输出：
- Page Object 文件
- 通用 helper 文件
- fixture 扩展文件
- 多用例集合文件

## 4. 顶层代码结构

推荐固定结构如下：

```ts
import { test, expect } from '@playwright/test';

test.describe('模块或场景名', () => {
  test('用例标题', async ({ page }) => {
    await test.step('步骤 1', async () => {
    });

    await test.step('步骤 2', async () => {
    });

    await test.step('结果校验', async () => {
    });
  });
});
```

首期要求：
- 必须 `import { test, expect } from '@playwright/test'`
- 必须使用 `test.describe`
- 必须使用单个 `test`
- 建议使用 `test.step`

## 5. 标题与命名规范

### 5.1 `describe` 标题

建议体现业务域，不要写得太泛：
- `用户管理-登录流程`
- `订单中心-创建订单`

不建议：
- `自动化测试`
- `AI生成脚本`

### 5.2 `test` 标题

建议格式：
- `应该 + 业务结果`
- `当 ... 时，应该 ...`

示例：
- `应该成功登录并进入用户列表页`
- `当提交正确账号密码时，应该进入首页`

### 5.3 `test.step` 标题

建议使用业务动作短语：
- `打开登录页`
- `输入用户名和密码`
- `点击登录按钮`
- `校验进入用户列表页`

要求：
- 贴近业务意图
- 不直接照抄 locator 文本
- 不写过长句子

## 6. fixture 约定

首期统一使用：
- `async ({ page })`

首期不引入：
- 自定义 fixture
- 自定义上下文对象
- 额外浏览器会话管理

原因：
- 降低 AI 生成复杂度
- 降低执行和回放差异
- 便于先打通主链路

## 7. locator 规范

### 7.1 优先级

建议优先级：
1. `getByTestId`
2. `getByRole`
3. `getByLabel`
4. `getByText`
5. `locator(css)`
6. `locator(xpath)`

### 7.2 选择规则

AI 或规则引擎应遵守：
- 默认使用 `step_model_json.locator.preferred`
- 若使用 fallback，必须在风险提示中标记
- 不允许无依据把高稳定 locator 替换为低稳定 locator

### 7.3 不推荐写法

首期尽量避免：
- 过长 css 链式选择器
- 绝对 xpath
- 基于动态 class 的 locator
- 明显易变的 nth-child 选择器

## 8. 步骤编排规范

### 8.1 保真要求

必须遵守：
- 步骤顺序严格与 `step_model_json.steps` 一致
- `is_critical = true` 的步骤不得删除
- 不允许跨步骤合并导致语义变化

### 8.2 每步内容

每个 `test.step` 内建议包含：
- 1 到 3 个核心操作
- 必要时少量过程断言
- 避免把整个用例全部塞进一个 step

### 8.3 页面跳转

页面跳转类操作建议：
- 使用明确的 `page.goto`
- 对关键跳转补充 URL 或页面标题校验

## 9. 断言规范

### 9.1 必备断言

每条脚本至少包含：
- 1 个最终结果断言

可选包含：
- 1 到 3 个高置信度过程断言

### 9.2 断言类型优先级

建议优先使用：
1. URL / 标题 / 主标题断言
2. 元素可见性断言
3. 状态类断言
4. 文本结果断言
5. 表格/列表结果断言

### 9.3 低置信度断言

处理原则：
- 不直接写进脚本主逻辑
- 输出到 `assertion_suggestions`
- 在页面上展示为“待确认项”

## 10. 注释规范

### 10.1 允许的注释

建议只保留两类：
- 关键业务步骤说明
- 风险 locator 或待确认点说明

示例：

```ts
// 关键登录动作，依赖测试账号数据
await page.getByLabel('用户名').fill(username);

// 风险提示：当前按钮 locator 稳定性一般，后续建议补充 testid
await page.getByText('登录').click();
```

### 10.2 不建议的注释

不要生成：
- `// 点击按钮`
- `// 输入用户名`
- `// 断言成功`

这类注释信息密度太低，没有维护价值。

## 11. 风险提示规范

如果脚本存在潜在不稳定点，建议以代码外元数据返回，不优先写进正文。

风险类型示例：
- 使用了 text locator
- 使用了 fallback locator
- 某步缺少高置信度断言
- 页面存在动态加载但等待条件较弱

推荐返回位置：
- `risk_hints[]`

## 12. 标准脚本示例

```ts
import { test, expect } from '@playwright/test';

test.describe('用户管理-登录流程', () => {
  test('应该成功登录并进入用户列表页', async ({ page }) => {
    await test.step('打开登录页', async () => {
      await page.goto('https://test.example.com/login');
      await expect(page).toHaveURL(/login/);
    });

    await test.step('输入用户名和密码', async () => {
      await page.getByLabel('用户名').fill('test_user');
      await page.getByLabel('密码').fill('***');
    });

    await test.step('点击登录按钮', async () => {
      await page.getByRole('button', { name: '登录' }).click();
    });

    await test.step('校验进入用户列表页', async () => {
      await expect(page).toHaveURL(/user\\/list/);
      await expect(page.getByRole('heading', { name: '用户列表' })).toBeVisible();
    });
  });
});
```

这个模板体现了：
- 单文件
- `test.describe`
- `test.step`
- 关键动作和最终结果断言分离
- locator 以 role/label 为主

## 13. 禁止模式

首期 AI 不应输出以下形式：

- 多个 `test` 混在一个文件里
- 没有 `expect` 的纯动作脚本
- 所有逻辑堆在一个超长 `test.step`
- 完全基于 css/xpath 的脆弱 locator
- 将真实密码、Token 写入脚本
- 与 `step_model_json` 顺序不一致的脚本

## 14. 与 AI 提示词的配合方式

建议将本文档提炼为结构化输入，传给 AI：

```json
{
  "file_suffix": ".spec.ts",
  "single_file_only": true,
  "use_test_describe": true,
  "use_test_step": true,
  "fixtures": ["page"],
  "locator_priority": ["testid", "role", "label", "text", "css", "xpath"],
  "require_final_assertion": true,
  "allow_inline_comments": "only_when_high_value"
}
```

这样 AI 不需要“理解整份文档”，而是消费规则化摘要。

## 15. 首期最小模板

如果第一版要尽快上线，最小模板建议固定为：
- 单文件
- `import { test, expect } from '@playwright/test'`
- `test.describe`
- 单个 `test`
- 每个关键业务动作一个 `test.step`
- 至少一个最终结果断言
- 不拆多文件

这是最容易稳定生成、最容易回放、最容易维护的方案。
