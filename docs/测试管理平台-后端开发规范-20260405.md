## 一、 日志标准 (Structured Logging)

本项目统一使用 Go 官方 `log/slog`，**严禁** `log.Println` / `fmt.Println`。

### 1.1 注入方式
Service 层结构体必须持有 `logger *slog.Logger`，通过构造函数注入。

```go
type CaseReviewService struct {
    logger     *slog.Logger
    reviewRepo repository.CaseReviewRepository
    txMgr      *repository.TxManager
}

func NewCaseReviewService(
    logger *slog.Logger,
    repo   repository.CaseReviewRepository,
    txMgr  *repository.TxManager,
) *CaseReviewService {
    return &CaseReviewService{
        logger:     logger.With("module", "case_review"),
        reviewRepo: repo,
        txMgr:      txMgr,
    }
}
```

### 1.2 日志分级标准

| Level | 场景 | 示例 |
|-------|------|------|
| `DEBUG` | SQL 参数、中间状态 | `s.logger.Debug("query params", "filter", f)` |
| `INFO` | 业务操作开始/结束 | `s.logger.Info("create review success", "id", review.ID)` |
| `WARN` | 可容忍异常（重试/降级） | `s.logger.Warn("cache miss, fallback to db")` |
| `ERROR` | 业务终止、DB 错误 | `s.logger.Error("create review failed", "error", err)` |

### 1.3 Key 命名规范
统一使用 **snake_case**：`project_id`, `review_id`, `user_id`, `error`, `elapsed_ms`。

### 1.4 敏感数据脱敏（强制）
```go
// ✅ 正确：仅记录用户 ID
s.logger.Info("user login", "user_id", user.ID)

// ❌ 错误：严禁记录密码、Token
s.logger.Info("user login", "password", req.Password) // 违规！
```

### 1.5 中间件层日志（HTTP 请求入口）
每个 HTTP 请求必须在中间件中记录 **方法、路径、状态码、耗时、请求者 ID**，这是生产排障的第一入口：

```go
func requestLogger(logger *slog.Logger) gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        c.Next()
        elapsed := time.Since(start)

        logger.Info("http request",
            "method",     c.Request.Method,
            "path",       c.Request.URL.Path,
            "status",     c.Writer.Status(),
            "elapsed_ms", elapsed.Milliseconds(),
            "user_id",    c.GetHeader("X-User-ID"),
            "client_ip",  c.ClientIP(),
        )

        // 慢请求告警（> 3 秒）
        if elapsed > 3*time.Second {
            logger.Warn("slow request detected",
                "path", c.Request.URL.Path, "elapsed_ms", elapsed.Milliseconds(),
            )
        }
    }
}
```

### 1.6 请求链路追踪 (Request ID / Trace ID)
为每个请求分配唯一 ID，贯穿 Handler → Service → Repository 全链路，便于在海量日志中定位单次请求：

```go
// 中间件：注入 Request-ID
func requestIDMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        reqID := c.GetHeader("X-Request-ID")
        if reqID == "" {
            reqID = uuid.NewString() // 自动生成
        }
        c.Set("request_id", reqID)
        c.Header("X-Request-ID", reqID) // 回传给调用方
        c.Next()
    }
}

// Handler 层：将 request_id 注入 context
func (a *API) createReview(c *gin.Context) {
    reqID, _ := c.Get("request_id")
    ctx := context.WithValue(c.Request.Context(), "request_id", reqID)
    review, err := a.svc.CreateReview(ctx, ...)
}

// Service 层：日志自动携带 request_id
func (s *CaseReviewService) CreateReview(ctx context.Context, ...) {
    reqID, _ := ctx.Value("request_id").(string)
    s.logger.Info("create review start", "request_id", reqID, "name", input.Name)
}
```

### 1.7 异步/定时任务日志
后台任务（如定时统计、数据清理）没有 HTTP 上下文，必须手动标记 `task_name` 和 `run_id`：

```go
func (s *StatsService) RunDailyReport(ctx context.Context) {
    runID := uuid.NewString()[:8]
    s.logger.Info("cron task start", "task_name", "daily_report", "run_id", runID)

    start := time.Now()
    err := s.generateReport(ctx)
    elapsed := time.Since(start)

    if err != nil {
        s.logger.Error("cron task failed",
            "task_name", "daily_report", "run_id", runID,
            "elapsed_ms", elapsed.Milliseconds(), "error", err,
        )
        return
    }
    s.logger.Info("cron task success",
        "task_name", "daily_report", "run_id", runID,
        "elapsed_ms", elapsed.Milliseconds(),
    )
}
```

### 1.8 各层日志职责总结

| 层级 | 记录什么 | 不记录什么 |
|------|---------|-----------|
| **中间件** | method, path, status, elapsed_ms, client_ip | 请求体内容 |
| **Handler** | 参数解析失败、权限拒绝 | 业务逻辑细节 |
| **Service** | 业务开始/结束、关键 ID、状态变更 | SQL 语句 |
| **Repository** | ❌ 通常不主动打日志 | — |
| **定时任务** | task_name, run_id, elapsed_ms, error | — |

## 二、 架构标准 (Three-Layer Architecture)

### 2.1 三层结构与职责边界

```
┌──────────────────────────────────────────────────────┐
│  Handler (API 层)                                     │
│  • 参数校验 (binding tag)                              │
│  • DTO 转换 (Request → ServiceInput)                  │
│  • HTTP 状态码映射                                     │
│  • RBAC 鉴权拦截 (requireRole)                        │
│  • Swagger 注解维护                                    │
├──────────────────────────────────────────────────────┤
│  Service (业务逻辑层)                                  │
│  • 全部业务规则与状态机                                 │
│  • 事务编排 (txMgr.WithTx)                            │
│  • 日志记录                                            │
│  • 调用 Repository 的原子操作                          │
├──────────────────────────────────────────────────────┤
│  Repository (数据访问层)                                │
│  • 单表/单资源的 CRUD                                  │
│  • 禁止包含业务逻辑                                    │
│  • 方法签名必须接受 context.Context                    │
└──────────────────────────────────────────────────────┘
```

### 2.2 事务管理范式
涉及多表更新的操作**必须**通过 `txMgr.WithTx` 包裹：

```go
func (s *CaseReviewService) LinkItems(ctx context.Context, ...) error {
    return s.txMgr.WithTx(ctx, func(tx *gorm.DB) error {
        // 1. 创建评审项
        if err := s.reviewRepo.CreateItem(ctx, tx, item); err != nil {
            return err
        }
        // 2. 分配评审人
        if err := s.reviewRepo.CreateReviewers(ctx, tx, reviewers); err != nil {
            return err
        }
        // 3. 重算统计
        return s.reviewRepo.RecalcReviewStats(ctx, tx, reviewID)
    })
}
```

> **⚠️ 强制规则：`WithTx` 回调内调用的所有 Repository 方法必须接收并使用 `tx` 参数。**
>
> Repository 方法应通过 `getDB(tx)` helper 选择使用事务连接或主连接：
> ```go
> func (r *repo) getDB(tx *gorm.DB) *gorm.DB {
>     if tx != nil { return tx }
>     return r.db
> }
>
> func (r *repo) Delete(ctx context.Context, tx *gorm.DB, id uint) error {
>     return r.getDB(tx).WithContext(ctx).Where("id = ?", id).Delete(&model.Tag{}).Error
> }
> ```
>
> **真实案例**：`TagService.Delete` 在事务内调用 `tagRepo.Delete(ctx, tagID)` 但 Delete 使用
> `r.db`（主连接）而非事务 `tx`，导致 SQLite 单写者死锁。MySQL 生产环境不触发但单元测试 100% 复现。

### 3.4 代码注释规范 (强制中文)

> 代码注释是团队协作的第一道文档，**必须使用中文**，确保每位成员无障碍阅读。

#### 3.4.1 注释分级标准

| 级别 | 适用对象 | 格式 | 必填 |
|------|---------|------|------|
| **包级** | 每个 package | `// Package xxx ...` 文件顶部 | ✅ |
| **结构体** | 所有导出 struct | `// XxxService 管理 xxx 的业务逻辑` | ✅ |
| **函数级** | 所有导出函数/方法 | `// FuncName 做什么（含参数、返回值说明）` | ✅ |
| **关键逻辑** | 复杂算法、业务规则、变通方案 | 行内 `//` | ✅ |
| **字段级** | 结构体字段（非自解释的） | 行尾 `//` | 推荐 |
| **TODO/FIXME** | 待办/已知问题 | `// TODO(作者): 描述` | 视情况 |

#### 3.4.2 范式 — Service 层

```go
// Package service 实现测试管理平台的核心业务逻辑。
// 每个 Service 对应一个业务领域，通过构造函数注入依赖。
package service

// TagService 管理项目级标签的全生命周期，
// 包括创建、编辑、删除、分页查询、用例关联解绑等操作。
type TagService struct {
    logger  *slog.Logger              // 结构化日志（带 module 前缀）
    tagRepo repository.TagRepository  // 标签数据访问层
    txMgr   *repository.TxManager     // 事务管理器
}

// Delete 删除指定标签，同时解除所有用例关联。
// 在同一事务内先解绑关联关系，再删除标签本体，保证数据一致性。
//
// 参数:
//   - ctx:       请求上下文（携带 trace_id）
//   - projectID: 项目 ID（用于归属校验）
//   - tagID:     待删除的标签 ID
//
// 返回:
//   - unlinked: 被解绑的用例数量
//   - err:      BizError（标签不存在 / 归属不匹配）或系统错误
func (s *TagService) Delete(ctx context.Context, projectID, tagID uint) (unlinked int64, err error) {
    // 1. 查询标签是否存在
    tag, err := s.tagRepo.GetByID(ctx, tagID)
    if err != nil {
        return 0, ErrNotFound(CodeTagNotFound, "标签不存在")
    }

    // 2. 校验标签归属项目（防止跨项目误删）
    if tag.ProjectID != projectID {
        return 0, ErrForbidden(CodeTagForbidden, "标签不属于当前项目")
    }

    // 3. 事务内执行：先解绑关联，再删除标签
    // ⚠️ 所有 repo 方法必须使用 tx 参数（参见 §2.2 事务规则）
    err = s.txMgr.WithTx(ctx, func(tx *gorm.DB) error {
        // 解除该标签与所有用例的关联关系
        unlinked, err = s.tagRepo.DeleteRelsByTagID(ctx, tx, tagID)
        if err != nil {
            return err
        }
        // 删除标签本体
        return s.tagRepo.Delete(ctx, tx, tagID)
    })

    if err == nil {
        s.logger.Info("标签删除成功",
            "tag_id", tagID, "project_id", projectID, "unlinked", unlinked,
        )
    }
    return
}
```

#### 3.4.3 范式 — Repository 层

```go
// TagRepository 定义标签数据访问层的接口。
// 所有方法签名必须接受 context.Context；
// 在事务内调用的方法必须接受 tx *gorm.DB 参数。
type TagRepository interface {
    Create(ctx context.Context, tag *model.Tag) error           // 创建标签
    Update(ctx context.Context, tag *model.Tag) error           // 更新标签信息
    Delete(ctx context.Context, tx *gorm.DB, id uint) error     // 删除标签（支持事务）
    GetByID(ctx context.Context, id uint) (*model.Tag, error)   // 按 ID 查询
    ListPaged(ctx context.Context, projectID uint, f TagFilter) ([]model.Tag, int64, error) // 分页列表
    DeleteRelsByTagID(ctx context.Context, tx *gorm.DB, tagID uint) (int64, error) // 解除标签关联
}
```

#### 3.4.4 范式 — Handler 层

```go
// @Summary 获取标签列表
// @Description 分页查询指定项目下的标签，支持关键词搜索和排序
// @Tags Tag
// @Param projectID path int true "项目 ID"
// @Param keyword query string false "搜索关键词"
// @Param sortBy query string false "排序字段: case_count | name | created_at（默认 case_count）"
// @Param page query int false "页码（默认 1）"
// @Param pageSize query int false "每页条数（默认 20）"
// @Success 200 {object} response.Response{data=[]model.Tag}
// @Router /projects/{projectID}/tags [get]
func (a *API) listTags(c *gin.Context) {
    // 获取当前登录用户信息（由 JWT 中间件注入）
    user := currentUser(c)

    // 解析路径参数中的项目 ID
    projectID, ok := parseUintParam(c, "projectID")
    if !ok {
        return
    }

    // 校验用户是否有该项目的访问权限
    if !a.requireProjectAccess(c, user, projectID) {
        return
    }

    // 构建查询过滤器
    f := repository.TagFilter{
        Keyword:  c.Query("keyword"),
        Page:     parsePositiveIntWithDefault(c.Query("page"), 1),
        PageSize: parsePositiveIntWithDefault(c.Query("pageSize"), 20),
        SortBy:   c.Query("sortBy"),
    }

    // 调用 Service 层获取分页数据
    tags, total, err := a.tagSvc.ListPaged(c.Request.Context(), projectID, f)
    if err != nil {
        response.HandleError(c, err)
        return
    }

    // 返回标准分页响应
    response.Page(c, tags, total, f.Page, f.PageSize)
}
```

#### 3.4.5 禁止事项

```go
// ❌ 无意义的废话注释
var count int // 定义 count 变量

// ❌ 英文注释（团队约定中文优先）
// get tag by id
tag, err := s.tagRepo.GetByID(ctx, id)

// ✅ 解释"为什么"而非"是什么"
// 标签名长度 2~50：与前端校验规则保持一致，避免创建无效标签
if len(input.Name) < 2 || len(input.Name) > 50 {
    return ErrBadRequest(CodeTagInvalidName, "标签名称长度需在 2-50 字符之间")
}

// ✅ 解释变通方案的原因
// 使用 getDB(tx) 而非直接使用 r.db，确保在事务上下文中执行
// 否则 SQLite 单写者模式下会死锁（详见 §2.2 事务管理范式）
return r.getDB(tx).WithContext(ctx).Delete(&model.Tag{}, id).Error
```

## 四、 错误码标准 (6-Digit Error Codes)

### 4.1 编码规则
格式为 `[服务2位][模块2位][序号2位]`。

### 4.2 完整速查表

| 错误码 | 常量名 | 含义 | HTTP 状态码 |
|--------|-------|------|------------|
| `200000` | `CodeSuccess` | 成功 | 200 |
| `400000` | `CodeBadRequest` | 通用参数错误 | 400 |
| `401000` | `CodeUnauthorized` | 未认证 | 401 |
| `403000` | `CodeForbidden` | 无权限 | 403 |
| `500000` | `CodeInternal` | 系统内部错误 | 500 |
| `100101` | `CodeReviewNotFound` | 评审计划不存在 | 404 |
| `100102` | `CodeReviewStatusInvalid` | 评审状态非法 | 400 |
| `100103` | `CodeReviewOwnerInvalid` | 归属校验失败 | 403 |
| `100104` | `CodeReviewItemNotFound` | 评审项不存在 | 404 |
| `100105` | `CodeReviewForbidden` | 非指定评审人 | 403 |
| `100106` | `CodeReviewMissingName` | 名称为空 | 400 |
| `100107` | `CodeReviewEmptyReviewer` | 未配置评审人 | 400 |
| `100108` | `CodeReviewItemMismatch` | 评审项归属不匹配 | 400 |

### 4.3 前后端对接协议
后端返回体固定为：
```json
{
  "code": 100101,
  "message": "评审计划不存在",
  "data": null
}
```
前端通过 `code` 数字进行**精准匹配**（而非字符串匹配），实现多语言与精细化提示。

### 4.4 定义位置
`internal/service/errors.go`

## 七、 工业级稳定性保障 (Stability)

### 7.1 Context 全链路透传 (强制)
```go
// ✅ 所有函数签名必须接受 context.Context
func (s *CaseReviewService) GetReview(ctx context.Context, projectID, reviewID uint) (*model.CaseReview, error)

// Handler 层传递 Gin 的 Context
```

### 7.3 优雅退出 (Graceful Shutdown)
```go
func main() {
    srv := &http.Server{Addr: ":8080", Handler: router}
    go func() { srv.ListenAndServe() }()

    // 监听系统信号
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit

    slog.Info("shutting down server...")
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()
    if err := srv.Shutdown(ctx); err != nil {
        slog.Error("server forced to shutdown", "error", err)
    }
    slog.Info("server exited")
}
```

## 十三、 限流标准 (Rate Limiting)

生产环境**必须**配置限流，防止恶意请求和突发流量击垮服务。

### 13.1 Gin 中间件范式
```go
import "golang.org/x/time/rate"

func rateLimitMiddleware(rps int) gin.HandlerFunc {
    limiter := rate.NewLimiter(rate.Limit(rps), rps*2) // 令牌桶
    return func(c *gin.Context) {
        if !limiter.Allow() {
            c.AbortWithStatusJSON(429, gin.H{
                "code": 429000, "message": "请求过于频繁，请稍后重试",
            })
            return
        }
        c.Next()
    }
}
```

|------|------|--------|

| 单用户 | 令牌桶 | 60 RPM |
| 登录接口 | 固定窗口 | 5 次/分钟 |

## 十四、 缓存策略 (Caching)

Redis 已在基础设施中部署，应按 **Cache-Aside** 模式使用。

### 14.1 Cache-Aside 范式
```go
func (s *ProjectService) GetByID(ctx context.Context, id uint) (*model.Project, error) {
    key := fmt.Sprintf("project:%d", id)

    // 1. 查缓存
    cached, err := s.redis.Get(ctx, key).Result()
    if err == nil {
        var p model.Project
        json.Unmarshal([]byte(cached), &p)
        return &p, nil
    }

    // 2. 查 DB
    p, err := s.projectRepo.GetByID(ctx, id)
    if err != nil {
        return nil, err
    }

    // 3. 回填缓存 (TTL 5 分钟)
    data, _ := json.Marshal(p)
    s.redis.Set(ctx, key, data, 5*time.Minute)
    return p, nil
}
```

### 14.2 缓存失效规则

| 操作 | 失效方式 |
|------|---------|
| Update/Delete | 立即删除对应 key |
| 批量操作 | 删除关联 key + 列表 key |
| 数据迁移 | 全量清除 namespace |

### 14.3 防护措施

- **缓存穿透**：对不存在的 ID 缓存空值 (TTL 1 分钟)
- **缓存雪崩**：TTL 加随机抖动 `5min + rand(0~60s)`
- **缓存击穿**：热点 key 使用 `singleflight` 防止并发回源

## 十七、 游标分页 (Cursor Pagination)

Offset 分页在大数据量下性能退化严重（`OFFSET 10000` 仍需扫描前 10000 行）。当表行数可能超过 **10 万**时，应改用 Keyset/Cursor 分页。

### 17.1 Repository 范式
```go
type CursorFilter struct {
    Cursor   uint   // 上一页最后一条 ID (0 = 首页)
    PageSize int
    SortBy   string // 排序字段
}

func (r *repo) ListByCursor(ctx context.Context, f CursorFilter) ([]model.TestCase, error) {
    q := r.db.WithContext(ctx)
    if f.Cursor > 0 {
        q = q.Where("id < ?", f.Cursor) // 倒序游标
    }
    var items []model.TestCase
    return items, q.Order("id DESC").Limit(f.PageSize).Find(&items).Error
}
```

### 17.2 API 响应格式
```json
{
  "code": 200000,
  "data": [...],
  "next_cursor": "12345",
  "has_more": true
}
```

### 18.2 Handler 范式
```go
import "github.com/go-playground/validator/v10"

func handleValidationError(c *gin.Context, err error) {
    var ve validator.ValidationErrors
    if errors.As(err, &ve) {
        fieldErrors := make([]gin.H, 0, len(ve))
        for _, fe := range ve {
            fieldErrors = append(fieldErrors, gin.H{
                "field":   fe.Field(),
                "message": translateFieldError(fe),
            })
        }
        c.JSON(400, gin.H{
            "code": 400000, "message": "参数校验失败", "errors": fieldErrors,
        })
        return
    }
    c.JSON(400, gin.H{"code": 400000, "message": err.Error()})
}
```
