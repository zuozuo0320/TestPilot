## 0. 实施目标

本次实施目标：

1. 去除 testcase 页面中的直接评审动作；
2. 保留 testcase 页面中的评审结果展示与筛选；
3. 保留 testcase 页面跳转到评审计划/评审记录的轻入口；
4. 将 testcase 级直接评审接口标记为待废弃；
5. 补齐 testcase 列表的评审投影字段，满足前端展示需求。

---

## 1. 当前现状盘点

## 1.1 前端现状

当前 `TestFront` 中已经同时存在两套评审能力：

### A. testcase 页面中的直接评审能力

相关文件：

- `TestFront/src/views/TestCasePage.vue`
- `TestFront/src/api/testcase.ts`

当前 testcase 页面仍保留以下动作：

- 提审
- 评审通过
- 评审驳回

当前对应 API 仍在前端调用：

- `submitReview(projectId, testcaseId)`
- `approveReview(projectId, testcaseId)`
- `rejectReview(projectId, testcaseId)`

### B. 独立的用例评审模块

相关文件：

- `TestFront/src/views/CaseReviewPage.vue`
- `TestFront/src/views/CaseReviewDetail.vue`
- `TestFront/src/api/caseReview.ts`
- `TestFront/src/router/index.ts`

评审模块已支持：

- 评审计划列表；
- 评审计划详情；
- 单条评审；
- 批量评审；
- 批量改评审人；
- 批量重新提审；
- 评审记录查看。

## 1.2 后端现状

当前 `TestPilot` 中也同时存在两套评审链路：

### A. testcase 维度的旧评审接口

相关文件：

- `TestPilot/internal/api/router.go`

当前仍暴露：

- `POST /projects/:projectID/testcase/:testcaseID/submit-review`（已移除）
- `POST /projects/:projectID/testcase/:testcaseID/approve-review`（已移除）
- `POST /projects/:projectID/testcase/:testcaseID/reject-review`（已移除）

### B. case-review 模块的正式评审接口

相关文件：

- `TestPilot/internal/api/handler_case_review.go`

## 2. 推荐实施顺序

建议按以下顺序推进：

### 第一阶段：前端收敛入口

- 先隐藏 testcase 页面上的直接评审按钮；
- 保证所有评审动作只在 case-review 页面执行；
- 避免继续扩大旧接口使用面。

### 第二阶段：后端补齐投影字段

- 为 testcase 列表补充评审投影能力；
- 让前端在不做评审动作的前提下，也能看清评审状态和跳转入口。

### 第三阶段：接口废弃治理

- 将 testcase 级旧评审接口标记为 deprecated；
- 清理 swagger / 前端调用 / 联调文档；
- 最终删除旧接口和旧服务实现。

## 3.2 P1：补齐 testcase 页面跳转入口

- `TestFront/src/views/TestCasePage.vue`

  - `查看评审记录`
  - `查看关联评审计划`

验收标准：

- 用户可从 testcase 页面进入对应评审计划；
- 无需在 testcase 页面直接完成评审。

## 4.2 P1：补齐 testcase 列表评审投影字段

### BE-03 为 testcase 列表增加评审投影字段

目标文件：

- `TestPilot/internal/repository/testcase_repo.go`
- `TestPilot/internal/model/models.go`
- `TestPilot/internal/service/testcase_service.go`

建议新增字段：

- `in_review`
- `current_review_id`
- `current_review_name`
- `related_review_count`

推荐实现方式：

1. 在 `TestCaseListItem` 中增加投影字段；
2. 在列表 SQL 中通过关联 `case_review_items` / `case_reviews` 计算：
   - 是否存在 `not_started/in_progress` 的关联计划；
   - 当前进行中评审计划 ID / 名称；
   - 历史关联评审计划数量；
3. 前端列表页使用这些字段做展示与跳转。

注意：

- `review_result` 仍然保留；
- 但它只能表达最终结果，不能完整表达“是否评审中”。

验收标准：

- testcase 列表不依赖直接评审按钮，也能体现当前评审上下文；
- 用户可以从列表页顺利跳转到评审模块。

### BE-04 testcase 详情增加评审摘要

目标文件：

- `TestPilot/internal/model/models.go`
- `TestPilot/internal/api/handler_testcase.go`
- `TestPilot/internal/repository/testcase_repo.go`

处理内容：

- 若当前详情接口直接返回 `model.TestCase`，建议增加虚拟字段或单独 DTO；
- 至少补充：
  - `in_review`
  - `current_review_id`
  - `current_review_name`
  - `related_review_count`

验收标准：

- testcase 详情页可以展示评审关联信息；
- 前端无需额外拼装多个接口才能显示基本评审摘要。

本次实施完成后，应满足以下标准：

1. testcase 页面不存在直接评审动作入口；
