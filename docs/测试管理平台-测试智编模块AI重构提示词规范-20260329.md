# 测试管理平台-测试智编模块AI重构提示词规范

> 版本：v0.1（创建于 2026-03-29）
> 关联文档：`测试管理平台-测试智编模块需求方案-20260327.md`、`测试管理平台-测试智编模块详细设计-20260328.md`、`测试管理平台-测试智编模块数据与接口设计-20260328.md`、`测试管理平台-测试智编模块研发拆解与步骤模型设计-20260329.md`、`测试管理平台-测试智编模块团队标准脚本模板规范-20260329.md`
> 文档定位：用于录制增强模式下，规范 AI 基于 `step_model_json` 重构 `Playwright TypeScript` 标准脚本的提示词设计与输出约束

## 1. 目标

本文档用于明确 AI 在“测试智编”模块中的重构职责，确保：
- AI 只负责将录制结果整理为符合团队规范的 `Playwright TypeScript` 脚本
- AI 不改变核心业务执行逻辑
- AI 输出结果可验证、可追溯、可维护

本文档默认适用场景：
- `generation_mode = RECORDING_ENHANCED`
- 输入主来源为 `step_model_json`
- 输出目标为单文件 `Playwright TypeScript` 测试脚本

## 2. AI 角色定义

AI 的定位不是“自由生成脚本”，而是“受控重构器”。

AI 允许做的事：
- 按团队标准模板组织代码结构
- 将步骤模型映射为 `test`、`test.step`、fixture、helper 调用
- 补充高置信度断言
- 补充必要注释、风险提示、断言建议
- 统一 locator 写法和命名风格

AI 不允许做的事：
- 改变 `step_model_json` 中的步骤顺序
- 删除关键步骤
- 无依据地替换核心 locator
- 伪造不存在的业务前置条件
- 凭空引入录制结果中未出现的业务动作
- 输出与团队约定模板不一致的多文件结构

## 3. 输入契约

AI 重构时，建议输入至少由 4 部分组成：
- 任务上下文
- 团队脚本标准
- `step_model_json`
- 可选参考信息

### 3.1 任务上下文

建议输入：

```json
{
  "task_id": 9001,
  "task_name": "登录并进入用户列表",
  "project_name": "Aisight Web",
  "case_ids": [2001, 2002],
  "scenario_desc": "使用测试账号登录系统并进入用户列表页",
  "framework_type": "Playwright",
  "language": "TypeScript",
  "generation_mode": "RECORDING_ENHANCED"
}
```

### 3.2 团队脚本标准

建议作为独立结构传入，而不是散落在自然语言中。

最少应包括：
- 文件模板
- 是否必须使用 `test.describe`
- 是否必须使用 `test.step`
- fixture 约定
- helper API 约定
- 命名规则
- 注释规则
- 断言规则

示例：

```json
{
  "file_suffix": ".spec.ts",
  "use_test_describe": true,
  "use_test_step": true,
  "single_file_only": true,
  "fixtures": ["page"],
  "helper_apis": [],
  "locator_priority": ["testid", "role", "label", "text", "css", "xpath"],
  "assertion_policy": {
    "allow_high_confidence_inline": true,
    "low_confidence_as_suggestion": true
  }
}
```

### 3.3 `step_model_json`

这是 AI 重构的唯一业务事实来源。

建议至少包含：
- 顶层任务信息
- `steps`
- `assertion_candidates`
- `risk_hints`

关键要求：
- `steps` 顺序即业务执行顺序
- `locator.preferred` 为首选 locator
- `is_critical = true` 的步骤不得删除

### 3.4 可选参考信息

可选附加输入：
- 原始录制稿 `raw_script_content`
- 页面标题或页面快照摘要
- 关联用例标题、预期结果
- 团队已有标准脚本样例

建议用途：
- 用于辅助命名、注释、断言优选
- 不作为覆盖 `step_model_json` 的依据

## 4. 输出契约

建议 AI 输出统一 JSON，而不是直接返回裸代码。

推荐格式：

```json
{
  "script_content": "import { test, expect } from '@playwright/test';\n...",
  "risk_hints": [
    "步骤 4 使用 text locator，稳定性中等"
  ],
  "assertion_suggestions": [
    "可补充用户列表页表格首行存在性断言"
  ],
  "generation_summary": {
    "critical_step_count": 5,
    "preserved_step_count": 5,
    "fallback_locator_used": false
  }
}
```

输出要求：
- `script_content` 必填
- `risk_hints` 可选，但建议始终返回数组
- `assertion_suggestions` 可选
- `generation_summary` 用于后处理校验

输出禁止：
- 不要输出 Markdown 代码块
- 不要输出解释性长文
- 不要输出与脚本无关的建议段落

## 5. 硬约束规则

### 5.1 步骤顺序

必须遵守：
- 严格按 `steps[].step_no` 顺序生成
- 不允许跨步合并导致语义改变
- 不允许将关键步骤前移或后移

### 5.2 关键步骤保留

必须遵守：
- `is_critical = true` 的步骤必须保留
- 若某步无法稳定表达，需在 `risk_hints` 说明，不能静默删除

### 5.3 locator 使用

必须遵守：
- 优先使用 `locator.preferred`
- 仅在 `preferred` 明显不可用时才允许用 `fallbacks`
- 若使用 fallback，必须在 `risk_hints` 说明

不允许：
- 自行发明新的 locator
- 在没有证据时把 role/label 改写成 css/xpath

### 5.4 断言策略

允许：
- 直接写入高置信度断言
- 输出低置信度断言建议

不允许：
- 为了“看起来完整”而凭空添加业务断言
- 删除 `step_model_json.assertion_candidates` 中的高置信度断言而不说明原因

### 5.5 代码结构

必须遵守：
- 输出单文件 `Playwright TypeScript`
- 文件后缀约定为 `.spec.ts`
- 遵循团队既定模板
- 可使用 `test.describe`
- 可使用 `test.step`

### 5.6 数据与敏感信息

必须遵守：
- 不在脚本中写死真实密码、Token、验证码
- 输入值尽量使用占位或数据引用
- 脱敏值不可直接还原成真实敏感信息

## 6. 推荐系统提示词

以下内容建议作为系统提示词基线：

```text
你是测试智编模块中的受控代码重构器。

你的任务是基于结构化步骤模型，将录制结果重构为符合团队规范的 Playwright TypeScript 测试脚本。

你必须遵守以下规则：
1. 只能依据输入的 step_model_json、团队规范和任务上下文生成代码。
2. 不得改变步骤顺序。
3. 不得删除关键步骤。
4. 不得无依据替换核心 locator。
5. 可以优化代码结构、命名、注释和断言表达，但不得改变业务执行逻辑。
6. 输出必须是 JSON，对应字段为 script_content、risk_hints、assertion_suggestions、generation_summary。
7. 不要输出 markdown，不要输出解释性文字。
8. 如果某些步骤无法稳定表达，必须保留并在 risk_hints 中说明。
9. 如果低置信度断言不适合直接写入脚本，应放入 assertion_suggestions。
```

## 7. 推荐用户提示词模板

建议执行服务传给模型的用户提示词采用固定模板：

```text
请根据以下输入，将录制结果重构为符合团队标准的 Playwright TypeScript 测试脚本。

【任务上下文】
{{task_context_json}}

【团队脚本标准】
{{team_standard_json}}

【结构化步骤模型】
{{step_model_json}}

【可选参考信息】
{{optional_reference_json}}

请严格遵守：
1. 不改变步骤顺序。
2. 不删除关键步骤。
3. 优先使用 preferred locator。
4. 输出 JSON，不要使用 markdown。
5. script_content 必须是单文件 Playwright TypeScript 脚本。
```

## 8. 推荐 few-shot 示例策略

如果后续模型效果不稳定，建议补 few-shot，但只放 1 到 2 组，不要太多。

建议示例类型：
- 登录类场景
- 列表查询类场景

few-shot 示例应体现：
- `step_model_json` 到标准脚本的映射方式
- locator 如何落到 `test.step`
- 高置信度断言如何直接写入脚本
- 低置信度断言如何进入建议区

## 9. 后处理校验清单

执行服务在接收 AI 输出后，建议做自动校验：

### 9.1 结构校验

- 输出是否为合法 JSON
- 是否存在 `script_content`
- `risk_hints`、`assertion_suggestions` 是否为数组

### 9.2 业务校验

- `critical_step_count == preserved_step_count`
- 关键 locator 是否仍然存在
- 步骤顺序是否覆盖完整
- 是否生成至少 1 个最终结果断言

### 9.3 代码校验

- 是否为 `TypeScript`
- 是否符合 `Playwright` 基本语法
- 是否符合单文件输出要求
- 是否使用 `.spec.ts` 约定命名

### 9.4 降级处理

若校验失败：
- 当前标准脚本不进入“可确认”流程
- 保留原始录制稿
- 返回错误原因
- 允许重新触发 AI 重构

## 10. 失败与降级策略

### 10.1 AI 重构失败

处理建议：
- 保留录制稿与步骤模型
- 任务标记为 `生成失败`
- 前端允许用户再次触发重构
- 必要时允许用户导出原始录制稿

### 10.2 AI 输出质量不达标

处理建议：
- 不覆盖已有有效版本
- 标记本次重构失败
- 返回校验不通过原因
- 允许用户修改团队规范后重试

### 10.3 模型服务不可用

处理建议：
- 默认回退为“仅保留录制稿 + 步骤模型”
- 不阻塞任务本身的录制结果保存
- 允许稍后重新发起 AI 重构

## 11. 最小提示词落地方案

如果第一版要快速落地，建议先只做：
- 固定系统提示词
- 固定用户提示词模板
- 固定 JSON 输出格式
- 基础结构校验
- 校验失败自动回退

第一版可以暂时不做：
- few-shot
- 多模板动态切换
- 场景分类专属提示词
- 模型评分路由
