# 测试管理平台-测试智编模块 AI 重构提示词规范

> 版本：v0.3（更新于 2026-04-02）
> 关联文档：`测试管理平台-测试智编模块需求方案-20260327.md`、`测试管理平台-测试智编模块详细设计-20260328.md`、`playwright-architecture-design.md`
> 文档定位：用于录制增强模式下，规范 AI 基于 `raw_script + step_model + page-registry` 进行 Playwright 多文件工程化重构的提示词设计与输出约束

## 1. 当前适用范围

本文档对应当前已落地实现，适用前提如下：

- `generation_mode = RECORDING_ENHANCED`
- 当前主路径是 `Playwright` 原始录制稿重构，不是自由生成脚本
- 输出目标是多项目、多文件工程化结果，不是单文件 `script_content`
- 运行载体是 `pw_projects/projects/<project_key>/` 下的标准 Playwright 工程

若与旧版“单文件脚本输出”口径冲突，以本文和 `playwright-architecture-design.md` 为准。

## 2. AI 角色定义

AI 在当前系统中的定位不是“自由脚本生成器”，而是“受控工程化重构器”。

AI 允许做的事：

- 按既定项目架构拆分 `spec / pages / registry_updates`
- 在已判定页面中补充 locator、action、等待和断言
- 组织 import、生成 `test.step`、补充中文注释
- 在不破坏旧语义的前提下，执行非破坏性增量扩展
- 输出风险提示与人工审核项

AI 不允许做的事：

- 自由改变页面边界
- 自由创建新的 shared 页面对象
- 改写、简化、替换或补发明原始录制稿中的 locator
- 凭空发明固定 URL ready 条件
- 直接修改 `base.fixture.ts`
- 输出跨项目路径或跨项目 import
- 引用目标 Page Object 中不存在的字段或方法

## 3. 事实来源优先级

当前实现中，AI 重构的事实来源优先级如下：

1. `raw_script`
2. `step_model_json`
3. `project_scope`
4. `page-registry.json`
5. 团队模板与固定 shared 白名单
6. 可选场景说明、用例标题、页面摘要

关键原则：

- `raw_script` 是录制增强模式下 locator 的唯一事实来源
- `step_model_json` 是步骤顺序与动作类型的结构化事实来源
- `page-registry.json` 是当前项目页面归属、已有 action、fixture 注入关系的唯一事实来源
- 可选参考信息只用于命名、注释和风险提示，不能覆盖前三类事实

## 4. 输入契约

建议传给模型的输入至少包括以下内容：

### 4.1 项目上下文

```json
{
  "project_id": 1,
  "project_key": "project_1",
  "project_name": "AiSight Demo",
  "workspace_root": "pw_projects/projects/project_1",
  "registry_file": "registry/page-registry.json",
  "auth_state_dir": "auth_states",
  "base_url_env": "BASE_URL",
  "auth_strategy": "storage_state"
}
```

### 4.2 当前项目注册表

至少需要包含：

- 页面文件路径
- `fixture_name`
- `module_key`
- `page_identity`
- 已有 `locators`
- 已有 `actions`
- `shared_dependencies`
- `page_update_mode`

### 4.3 原始录制稿

必须传入完整 `raw_script`，不能只传摘要。原因如下：

- 需要保留完整 locator chain
- 需要识别是否存在 `waitForURL` / URL 断言
- 需要识别复杂入口 locator 是否被显式保留
- 需要识别三次点击、键盘操作、特殊输入等动作细节

### 4.4 步骤模型

`step_model_json` 至少应包含：

- `steps`
- `total_steps`
- `generation_steps`
- `generation_total_steps`
- `normalization_items`
- `generation_script`

其中：

- `steps` 表示原始录制步骤，仅用于追溯和保真校验
- `generation_steps` 表示平台归一化后的生成步骤，是生成阶段唯一业务动作序列
- `normalization_items` 表示平台显式执行过的归一化动作清单
- `generation_script` 表示仅按 `normalization_items` 裁剪后的生成用录制稿

每个步骤项至少需要包含：

- `action_type`
- `locator`
- `input_value`
- `page_url`

### 4.5 辅助上下文

可选传入：

- `scenario_desc`
- 关联用例标题与预期结果
- 原始页面快照摘要
- 已存在的 shared 白名单模板

## 5. 输出契约

当前实现要求 AI 输出结构化 JSON，而不是裸代码。推荐格式如下：

```json
{
  "spec_file": {
    "path": "tests/<module>/<name>.spec.ts",
    "content": "完整 spec 内容"
  },
  "page_creates": [],
  "page_updates": [],
  "registry_updates": {},
  "manual_review_items": [],
  "risk_hints": [],
  "generation_summary": "中文摘要"
}
```

输出要求：

- `spec_file`、`page_creates`、`page_updates`、`registry_updates` 使用当前项目相对路径
- `page_updates` 只允许增量，不允许整文件覆盖已有 Page Object
- `manual_review_items` 必须显式输出，不允许埋在说明文字中
- `generation_summary` 需要说明本次创建了哪些文件、追加了哪些 action、是否存在风险

输出禁止：

- 不要输出 Markdown 代码块
- 不要输出解释性散文
- 不要输出 fixture 修改内容
- 不要输出跨项目路径

## 6. 当前硬约束

### 6.1 原始 locator 零改写

必须遵守：

- 禁止改写、简化、替换、补发明任何新的 `page.getBy...()` / `page.locator()` / `page.frameLocator()` 表达式
- 如果原始 locator 是链式定位器，抽取到 Page Object 时必须保留完整 locator chain
- 对于卡片、列表项、表格行里的复杂入口 locator，必须新增一一对应的专用 locator / action
- 禁止把复杂入口 locator 偷换成 `openTaskListByIndex(index)`、`viewTaskButtons.nth(index)` 之类的泛化 helper

### 6.2 步骤顺序与动作语义

必须遵守：

- 若 `step_model_json` 同时存在 `steps` 与 `generation_steps`，生成阶段必须以 `generation_steps` 为准
- `steps` 只用于追溯原始录制，不得直接拿来覆盖 `generation_steps`
- 除 `normalization_items` 中平台显式标记的归一化外，不得删除关键步骤、合并会改变语义的步骤、重排业务动作
- 当前唯一允许的步骤折叠是 `collapse_duplicate_opener_click`，仅用于折叠“第一次点击已打开弹窗/抽屉/表单、第二次同定位器点击为冗余”的 opener 重复点击
- 对未被归一化的父级菜单展开、二次点击、键盘操作等录制细节，仍必须保留

### 6.3 URL 语义约束

必须遵守：

- 只有录制稿中明确出现 `waitForURL` 或 URL 断言时，才允许保留 URL 级等待
- 若录制稿中未出现 URL 语义，页面 ready 必须优先使用真实业务控件
- 禁止凭空写死 `/assetsScan`、`/unitIndex`、`/domainTask` 这类固定路由断言

### 6.4 页面与 shared 边界

必须遵守：

- 页面边界由系统和 registry 判定，AI 不自由拆分页面
- 当前 shared 固定白名单仅允许：`NavigationPage / DialogPage / ToastPage / LoginPage`
- 业务弹窗、业务抽屉、业务页内 Tab 默认归当前 BusinessPage，不额外新建 shared 页面

### 6.5 spec / Page Object / fixture 分层

必须遵守：

- `spec` 层只表达业务意图，不直接写 locator
- Page Object 层负责 locator、action、必要断言和等待
- `base.fixture.ts` 由程序根据 registry 重建，AI 不输出 fixture 修改
- 生成 `spec` 前必须检查所调用的 locator 字段与 action 方法是否真的存在
- 生成 `spec` 前必须检查调用参数类型是否与 Page Object 方法签名一致

### 6.6 中文注释与可维护性

必须遵守：

- 每个类必须有中文职责注释
- 每个 action 方法必须有中文 JSDoc 注释
- 关键业务逻辑必须有中文行内注释
- 允许补充高置信度断言和稳定等待，但不能改变业务语义

### 6.7 平台与 Windows 兼容性

必须遵守：

- 输出内容必须可在项目工作区中直接运行
- 不依赖平台私有运行时才能执行
- 中文 locator、中文注释必须按 UTF-8 语义输出，避免在 Windows 上合并后乱码

## 7. 建议系统提示词基线

```text
你是测试智编模块中的受控 Playwright 工程化重构器。

你的任务不是自由生成脚本，而是把原始录制稿整理为符合项目架构的多文件输出。

必须遵守：
1. raw_script 是 locator 的唯一事实来源，不允许改写、简化、替换 locator。
2. step_model_json.steps 是原始录制步骤；step_model_json.generation_steps 是生成阶段唯一业务动作序列。
3. 只能在当前 project_scope 与 page-registry.json 作用域内输出文件。
4. spec 中不得出现 locator。
5. 不允许编辑 fixture。
6. 共享页面仅允许 NavigationPage / DialogPage / ToastPage / LoginPage。
7. 若录制稿未出现 waitForURL / URL 断言，不允许发明固定 URL ready 条件。
8. 复杂入口 locator 必须显式保留，不允许泛化成索引 helper。
9. 若 normalization_items 中存在 collapse_duplicate_opener_click，不允许把被平台折叠的第二次 opener 点击重新输出回来。
10. 所有类、action 方法和关键逻辑必须带中文注释。
11. 输出必须是 JSON，包含 spec_file、page_creates、page_updates、registry_updates、manual_review_items、risk_hints、generation_summary。
```

```text
【项目上下文】
{{PROJECT_SCOPE}}

【当前项目注册表】
{{PAGE_REGISTRY}}

【原始录制稿】
{{RAW_SCRIPT}}

【步骤模型】
{{STEP_MODEL}}

【场景描述】
{{SCENARIO_DESC}}

请严格遵守：
1. 只做结构组织，不改 locator。
2. 若步骤模型中存在 generation_steps / generation_script，必须以其为准；steps 仅用于追溯原始录制。
3. 不发明 URL ready 条件。
4. spec 不得出现 locator。
5. 不输出 fixture 修改。
6. 除平台通过 normalization_items 明确标记的 collapse_duplicate_opener_click 外，不得删改业务步骤。
7. 如果存在页面归属不确定、复杂入口无法安全复用、可能破坏旧语义的情况，写入 manual_review_items。
```

## 9. 程序侧校验要求

模型输出后，程序仍必须继续执行以下校验：

- JSON 结构校验
- 路径作用域校验
- 原始 locator 保留校验
- 复杂原始 locator 显式覆盖校验
- URL 语义保留校验
- shared 白名单校验
- 中文注释硬约束校验
- spec 成员存在性与方法签名匹配校验

## 10. 人工审核触发条件

以下场景必须进入 `manual_review`：

- 需要破坏性更新已有页面文件
- 需要删除或重命名已有 locator / action
- 无法确认页面归属
- 无法保留复杂入口 locator
- 生成结果引用了不存在的成员
- 需要修改 fixture
- 路径越界或存在跨项目 import
