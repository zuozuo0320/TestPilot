## 1. 通用约定

### 1.1 基础前缀

- Base Path：`/api/v1`
- 业务前缀：`/projects/{projectID}/case-reviews`

### 1.2 认证与权限

- 认证方式：Bearer Token
- 所有接口都先校验项目访问权限
- `admin` 通过全局角色直接放行
- 其余角色按接口能力做二次校验

### 1.3 响应结构

统一使用平台响应包裹：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 1.4 错误码

当前用例评审模块采用 6 位数值错误码：

| 错误码 | 常量 | 说明 |
|---|---|---|
| `120101` | `CodeReviewNotFound` | 评审计划不存在 |
| `120102` | `CodeReviewStatusInvalid` | 当前计划/评审项状态不允许该操作 |
| `120103` | `CodeReviewOwnerInvalid` | 创建者/归属信息不合法 |
| `120104` | `CodeReviewItemNotFound` | 评审项不存在 |
| `120105` | `CodeReviewForbidden` | 当前用户无权执行该评审动作 |
| `120106` | `CodeReviewMissingName` | 缺少计划名称 |
| `120107` | `CodeReviewEmptyReviewer` | 未配置默认评审人或条目评审人 |
| `120108` | `CodeReviewItemMismatch` | 评审项不属于当前 `reviewID + projectID` |

## 6. 评审计划接口

### 6.1 获取评审计划列表

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews`

#### Query 参数

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `view` | string | 否 | `all / created / assigned` |
| `keyword` | string | 否 | 按计划 ID 或名称搜索 |
| `status` | string | 否 | 计划状态 |
| `review_mode` | string | 否 | 评审模式 |
| `reviewer_id` | number | 否 | 指定评审人过滤 |
| `module_id` | number | 否 | 所属模块过滤 |
| `created_by` | number | 否 | 创建人过滤 |
| `page` | number | 否 | 默认 `1` |
| `pageSize` | number | 否 | 默认 `20`，最大 `100` |

#### 说明

- `view=created`：查“我创建的”
- `view=assigned`：查“分配给我的”
- 其他值默认按 `all` 处理

### 6.2 创建评审计划

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews`

#### 请求体

```json
{
  "name": "登录模块功能用例评审",
  "module_id": 12,
  "review_mode": "parallel",
  "default_reviewer_ids": [7, 9],
  "planned_start_at": "2026-04-06T09:00:00+08:00",
  "planned_end_at": "2026-04-08T18:00:00+08:00",
  "description": "版本 V3 提测前集中评审",
  "testcase_ids": [201, 202, 203],
  "auto_submit": true
}
```

#### 规则

- `name` 必填
- `review_mode` 必填
- `default_reviewer_ids` 必填且至少一人
- `auto_submit=true` 时，对应 testcase 会回写为待评审

### 6.3 获取评审计划详情

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}`

#### 说明

- 返回计划基础信息、统计信息、默认评审人信息

### 6.4 更新评审计划

- 方法：`PUT`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}`

#### 请求体

```json
{
  "name": "登录模块功能用例评审（第二轮）",
  "module_id": 12,
  "review_mode": "parallel",
  "default_reviewer_ids": [7, 9, 15],
  "planned_start_at": "2026-04-06T09:00:00+08:00",
  "planned_end_at": "2026-04-10T18:00:00+08:00",
  "description": "补充异常路径后继续评审"
}
```

#### 规则

- 当前后端已支持更新 `default_reviewer_ids`
- `closed` 计划不可编辑
- `completed` 计划不可修改 `review_mode`

### 6.5 删除评审计划

- 方法：`DELETE`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}`

#### 规则

- `not_started / closed`：允许删除
- `in_progress / completed`：
  - 若已有评审记录，不允许删除
  - 若无评审记录，允许删除

### 6.6 关闭评审计划

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/close`

#### 说明

- 将计划状态置为 `closed`
- 关闭后不再允许编辑、关联、批处理评审项

### 6.7 复制评审计划

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/copy`

#### 请求体

```json
{
  "name": "登录模块功能用例评审-复制",
  "include_cases": true,
  "reset_reviewers": false
}
```

#### 规则

- 默认复制计划级 `default_reviewer_ids`
- `include_cases=true` 时复制评审项，但重置新计划状态
- `reset_reviewers=false` 时复制评审人分配
- 历史评审记录不复制

## 7. 评审项接口

### 7.1 获取评审项列表

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items`

#### Query 参数

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `keyword` | string | 否 | 按 testcase ID 或标题快照搜索 |
| `review_status` | string | 否 | `pending / reviewing / completed` |
| `final_result` | string | 否 | `pending / approved / rejected / needs_update` |
| `reviewer_id` | number | 否 | 指定评审人过滤 |
| `module_id` | number | 否 | 模块过滤 |
| `page` | number | 否 | 默认 `1` |
| `pageSize` | number | 否 | 默认 `20` |

### 7.2 关联用例

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/link`

#### 请求体

```json
{
  "items": [
    { "testcase_id": 201, "reviewer_ids": [7, 9] },
    { "testcase_id": 202 }
  ],
  "auto_submit": true
}
```

#### 规则

- 每条 item 可单独指定 `reviewer_ids`
- 未指定时回退使用计划 `default_reviewer_ids`
- 若两者都为空，返回 `120107`
- 同一 testcase 已存在其他活跃评审计划时，不允许重复关联
- `auto_submit=true` 时 testcase 回写为待评审

### 7.3 移除评审项

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/unlink`

#### 请求体

```json
{
  "item_ids": [1001, 1002]
}
```

#### 规则

- 删除前先校验这些 `item_ids` 全部属于当前 `reviewID + projectID`
- 已产生评审记录的评审项不允许移除
- 删除后自动重算计划统计

### 7.4 批量改评审人

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/batch-reassign`

#### 请求体

```json
{
  "item_ids": [1001, 1002],
  "reviewer_ids": [7, 9]
}
```

#### 规则

- 会先删除旧评审人分配，再重建新分配
- 若评审项此前已有最终结果，则自动重置为新一轮待评审
- testcase 主表同步回写为：
  - `status = pending`
  - `review_result = 待评审`

### 7.5 批量重新提审

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/batch-resubmit`

#### 请求体

```json
{
  "item_ids": [1001, 1002]
}
```

#### 规则

- 评审项重置为待评审
- `current_round_no + 1`
- 评审人当前状态重置
- testcase 主表同步回写为：
  - `status = pending`
  - `review_result = 重新提审`

## 8. 评审提交接口

### 8.1 批量评审

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/batch-review`

#### 请求体

```json
{
  "item_ids": [1001, 1002],
  "result": "approved",
  "comment": "通过"
}
```

#### 规则

- `result` 允许：`approved / rejected / needs_update`
- 逐条调用单条评审逻辑并汇总成功/失败数量

### 8.2 单条评审

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/{itemID}/review`

#### 请求体

```json
{
  "result": "needs_update",
  "comment": "建议补充账号锁定步骤"
}
```

#### 返回示例

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "item_id": 1001,
    "review_status": "completed",
    "final_result": "needs_update",
    "current_round_no": 1,
    "testcase_status": "draft",
    "testcase_review_result": "建议修改",
    "next_pending_item_id": 1002
  }
}
```

#### 聚合规则

##### `single`

- 最后一次有效提交为准
- 当前实现依赖评审人查询排序：`reviewed_at DESC, id DESC`

##### `parallel`

- 仍有评审人未提交时：
  - `final_result = pending`
  - `review_status = reviewing`
- 所有人均提交后：
  - 任一 `rejected` => `rejected`
  - 否则若全部 `approved` => `approved`
  - 否则若存在 `needs_update` => `needs_update`

#### testcase 回写规则

| 聚合结果 | `testcase_status` | `testcase_review_result` |
|---|---|---|
| `approved` | `active` | `已通过` |
| `rejected` | `draft` | `已驳回` |
| `needs_update` | `draft` | `建议修改` |
| `pending` | `pending` | `待评审` |

## 9. 评审记录接口

### 9.1 获取评审记录

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/{itemID}/records`

#### Query 参数

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `page` | number | 否 | 默认 `1` |
| `pageSize` | number | 否 | 默认 `20` |
| `round_no` | number | 否 | 按轮次过滤 |

#### 规则

- 返回前先校验 `itemID` 是否属于当前 `reviewID + projectID`
- 该接口用于查看历史轮次评审记录，不负责聚合状态

## 10. 与前端页面的当前对齐说明

### 10.1 当前前端已对齐

- 列表页视图值使用 `all / created / assigned`
- 真实提交路径使用：
  - `items/batch-reassign`
  - `items/batch-resubmit`
  - `items/{itemID}/review`

### 10.2 当前前端未完全暴露的能力

- 后端 `PUT /case-reviews/{reviewID}` 已支持更新 `default_reviewer_ids`
- 当前前端类型和页面尚未完整暴露计划编辑能力
- 后端批量评审支持 `needs_update`，当前前端页面只暴露了 `approved / rejected`
