## 1. 说明

本文档按平台现有接口风格输出“用例评审模块”首期接口草案。

约定：

- 基础前缀：`/api/v1`
- 认证方式：沿用平台现有登录态 / Bearer Token
- 响应结构：沿用统一包装 `code + message + data`
- 所有接口均以 `projectID` 为项目隔离边界
- 新前端页面只调用评审计划接口，不再直接依赖单条用例 `approve-review/reject-review`

## 5. 评审计划接口

## 5.1 获取评审计划列表

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews`
- 用途：列表页展示 `全部 / 我评审的 / 我创建的`

查询参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `view` | string | 否 | `all / assigned / created` |
| `keyword` | string | 否 | 按 `ID / 评审名称 / 标签` 搜索，首期可先实现 `ID / 名称` |
| `status` | string | 否 | 计划状态 |
| `review_mode` | string | 否 | 评审模式 |
| `reviewer_id` | number | 否 | 指定评审人筛选 |
| `module_id` | number | 否 | 所属模块 |
| `created_by` | number | 否 | 创建人 |
| `page` | number | 否 | 默认 `1` |
| `pageSize` | number | 否 | 默认 `20` |

成功响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "id": 101,
        "name": "登录模块功能用例评审",
        "review_mode": "parallel",
        "status": "in_progress",
        "case_total_count": 18,
        "pending_count": 5,
        "approved_count": 10,
        "rejected_count": 1,
        "needs_update_count": 2,
        "pass_rate": 55.56
      }
    ],
    "total": 1,
    "page": 1,
    "pageSize": 20
  }
}
```

## 5.2 创建评审计划

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews`

请求体：

```json
{
  "name": "登录模块功能用例评审",
  "module_id": 12,
  "review_mode": "parallel",
  "default_reviewer_ids": [7, 9],
  "planned_start_at": "2026-04-05T09:00:00+08:00",
  "planned_end_at": "2026-04-08T18:00:00+08:00",
  "description": "版本 V3 提测前集中评审",
  "testcase_ids": [201, 202, 203],
  "auto_submit": true
}
```

字段说明：

- `default_reviewer_ids` 为计划默认评审人。
- `testcase_ids` 为初始关联用例。
- `auto_submit=true` 表示创建后立即把关联用例回写成 `pending + 待评审`。

## 5.3 获取评审计划详情

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}`

返回内容：

- 计划基础信息
- 计划统计信息
- 默认评审人
- 操作权限标记

## 5.4 更新评审计划

- 方法：`PUT`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}`

请求体：

```json
{
  "name": "登录模块功能用例评审-第一轮",
  "module_id": 12,
  "review_mode": "parallel",
  "default_reviewer_ids": [7, 9, 11],
  "planned_start_at": "2026-04-05T09:00:00+08:00",
  "planned_end_at": "2026-04-09T18:00:00+08:00",
  "description": "增加安全评审人"
}
```

更新规则：

- `completed/closed` 状态下不允许直接改评审模式。
- 已存在评审记录的 item，不允许被静默删除；只能通过显式移除未评审 item。

## 5.5 删除评审计划

- 方法：`DELETE`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}`

建议规则：

- 仅 `not_started` 或“无任何评审记录”的计划允许删除。
- 若已有记录，建议改走“关闭评审计划”而非物理删除。

## 5.6 关闭评审计划

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/close`

请求体：

```json
{
  "reason": "当前轮评审结束，剩余未评审项转入下个计划"
}
```

说明：

- 关闭后不可再新增评审动作。
- 关闭不修改已形成最终结果的主表状态。

## 5.7 复制评审计划

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/copy`

请求体：

```json
{
  "name": "登录模块功能用例评审-复制",
  "include_cases": true,
  "reset_reviewers": false
}
```

复制规则：

- 复制计划基础信息。
- 可复制关联用例。
- 评审状态全部重置。
- 评审记录不复制。

## 6. 评审项接口

## 6.1 获取评审项列表

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items`

查询参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `keyword` | string | 否 | `ID / 用例名称` |
| `review_status` | string | 否 | `pending / reviewing / completed` |
| `final_result` | string | 否 | `pending / approved / rejected / needs_update` |
| `reviewer_id` | number | 否 | 按评审人筛选 |
| `module_id` | number | 否 | 按模块筛选 |
| `page` | number | 否 | 默认 `1` |
| `pageSize` | number | 否 | 默认 `20` |

## 6.2 关联用例

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/link`

请求体：

```json
{
  "items": [
    {
      "testcase_id": 201,
      "reviewer_ids": [7, 9]
    },
    {
      "testcase_id": 202,
      "reviewer_ids": [9]
    }
  ],
  "auto_submit": true
}
```

处理规则：

- 新增 item 时生成当前 reviewer 分配。
- `auto_submit=true` 时回写 `test_cases.status = pending`。
- 若发现用例已存在于其他进行中计划，返回冲突。

## 6.3 移除关联用例

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/unlink`

请求体：

```json
{
  "item_ids": [1001, 1002]
}
```

建议规则：

- 已存在评审记录的 item 不允许直接移除。
- 未提交过评审记录的 item 可移除，并重算计划统计。

## 6.4 批量修改评审人

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/batch-reassign`

请求体：

```json
{
  "item_ids": [1001, 1002],
  "reviewer_ids": [7, 9, 11]
}
```

建议规则：

- 已经出最终结果的 item 不建议直接改评审人；如需重评，先重新提审。
- 变更后应同步重置该 item 的 reviewer 分配表。

## 6.5 批量重新提审

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/batch-resubmit`

请求体：

```json
{
  "item_ids": [1001, 1002]
}
```

结果：

- `current_round_no + 1`
- item 重置为 `pending`
- reviewer 当前状态重置
- 主表回写成 `pending + 重新提审`

## 6.6 批量评审

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/batch-review`

请求体：

```json
{
  "item_ids": [1001, 1002],
  "result": "approved",
  "comment": "批量通过，细节已核验"
}
```

说明：

- 仅对当前操作者有权限的 item 生效。
- 本质上是对每个 item 逐条调用单条评审事务。
- 响应中应返回成功数、失败数、失败原因列表。

## 7. 单条评审接口

## 7.1 提交单条评审

- 方法：`POST`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/{itemID}/review`

请求体：

```json
{
  "result": "needs_update",
  "comment": "步骤缺少异常分支和锁定场景"
}
```

成功响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "item_id": 1001,
    "review_status": "reviewing",
    "final_result": "pending",
    "current_round_no": 1,
    "testcase_projection": {
      "status": "pending",
      "review_result": "待评审"
    },
    "next_pending_item_id": 1002
  }
}
```

事务要求：

1. 校验当前用户属于该 item 评审人
2. 写入评审记录
3. 更新 reviewer 当前状态
4. 聚合 item 最终结果
5. 如已形成最终结果，回写 `test_cases`
6. 重算计划通过率与统计
7. 返回下一条待评审 item ID

## 7.2 获取单条评审记录

- 方法：`GET`
- 路径：`/projects/{projectID}/case-reviews/{reviewID}/items/{itemID}/records`

查询参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `round_no` | number | 否 | 不传则返回全部轮次 |
| `page` | number | 否 | 默认 `1` |
| `pageSize` | number | 否 | 默认 `20` |

## 8. 兼容现有单条用例接口的建议

当前后端已有：

- `POST /projects/{projectID}/testcase/{testcaseID}/submit-review`
- `POST /projects/{projectID}/testcase/{testcaseID}/approve-review`
- `POST /projects/{projectID}/testcase/{testcaseID}/reject-review`

为避免两套评审入口长期并存，建议：

### 8.1 新前端页面

- 不再调用 `approve-review/reject-review`
- 统一改为评审计划接口

### 8.2 兼容期策略

首期可保留旧接口，但需要收敛语义：

- `submit-review`：仅作为兼容接口，后续可限制为“加入默认快速评审计划”或直接下线
- `approve-review/reject-review`：建议只保留给临时运维脚本，不再在正式页面暴露

### 8.3 必须共用统一回写逻辑

无论旧接口是否保留，都必须共用同一个投影逻辑：

- 统一更新 `test_cases.status`
- 统一更新 `test_cases.review_result`
- 统一写审计日志

否则后续会出现状态口径不一致。

## 9. 建议错误码

建议新增以下错误码：

| 错误码 | 说明 |
|---|---|
| `CASE_REVIEW_NOT_FOUND` | 评审计划不存在 |
| `CASE_REVIEW_ITEM_NOT_FOUND` | 评审项不存在 |
| `CASE_REVIEW_FORBIDDEN` | 当前用户无评审权限 |
| `CASE_REVIEW_STATUS_INVALID` | 当前计划状态不允许该操作 |
| `CASE_ALREADY_IN_ACTIVE_REVIEW` | 用例已存在进行中评审 |
| `CASE_REVIEW_DUPLICATE_REVIEWER` | 评审人重复 |
| `CASE_REVIEW_EMPTY_REVIEWER` | 未配置评审人 |
| `CASE_REVIEW_RECORD_EXISTS` | 已有评审记录，不允许移除 |
| `CASE_REVIEW_RESULT_INVALID` | 评审结果非法 |
| `CASE_REVIEW_ROUND_CONFLICT` | 轮次冲突，请刷新后重试 |

## 11. 本轮结论

本接口草案明确了 4 件事：

1. 用例评审模块以后端独立 API 资源组形式存在；
2. 前端页面以“评审计划 -> 评审项 -> 评审记录”三层对象交互；
3. 评审提交后由后端统一聚合并回写用例主表状态；
4. 旧的单条审批接口即使保留，也必须退居兼容层。

如果继续推进，我下一步建议直接补：

- GORM Model 结构草案
- Repository / Service 方法签名
- 后端实现拆解清单
