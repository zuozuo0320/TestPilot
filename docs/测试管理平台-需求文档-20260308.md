# 测试管理平台需求文档（当前基线）

> 版本：v2.5（更新于 2026-03-21）
> 项目：Aisight（后端 TestPilot / 前端 TestFront）
> 适用对象：产品、研发、测试、运维

## 1. 项目目标

构建一套工业级测试管理平台，覆盖：
- 测试用例管理（目录树、筛选、分页、编辑）
- 系统管理（用户/角色/项目）
- IAM 权限与审计闭环
- 个人中心与头像管理
- Docker 真环境稳定部署

## 2. 业务范围

### 2.1 In Scope（范围内）
1. 用户与角色管理
2. 项目管理与项目切换联动
3. 用例列表、筛选、分页、目录树管理
4. 用例创建/编辑（基础信息、状态、步骤）
5. 个人资料维护与头像上传
6. 基础审计日志（关键变更）

### 2.2 Out of Scope（暂不纳入）
1. 自动化执行引擎（仅预留 E2E 导航）
2. 第三方缺陷系统双向集成
3. 完整工作流编排引擎

## 3. 关键业务规则（已落地）

1. 预置全局系统角色：`admin / manager / tester / reviewer / developer / readonly`（详见 2.2 角色定义）
2. 预置角色不可删除（返回 409）
3. admin 用户不可删除（前后端双重限制）
4. 用户创建时：
   - 必须至少选择 1 个角色，支持一次选择多个角色
   - 角色选择为多选，权限按所选角色并集生效
   - 不可分配 admin 角色
   - 必须校验 name / email / phone
   - 头像不允许创建时设置（默认头像）
5. 个人中心：
   - 支持头像上传（jpg/jpeg/png/webp，<=2MB）
   - 邮箱不可修改（后端硬拒绝）
6. 用户-角色为多对多；用户-项目为多对多；角色为全局角色，与项目成员关系解耦
7. 用户角色的分配、调整、回收仅允许 `admin` 执行
8. 用户删除为逻辑删除（`deleted_at`）
9. 目录树规则：
   - 单根"全部用例"
   - 最多 5 层
   - 仅允许从叶子目录删除
   - 删除目录会级联删除目录下用例

### 3.2 跨模块联动规则（用户 ↔ 角色 ↔ 项目）

| 序号 | 触发操作 | 联动行为 |
|------|---------|---------|
| 1 | 用户被逻辑删除 | 自动从所有项目成员列表中移除；该用户创建的用例/缺陷的 `created_by` 展示为「用户名（已注销）」 |
| 2 | 用户被禁用 | 仅禁止登录，项目成员关系、数据归属均保持不变 |
| 3 | 项目归档 | 成员不再能在项目选择器中看到该项目（admin 除外），数据只读 |
| 4 | 删除自定义角色 | 需先解除所有用户关联；用户至少保留一个角色，不允许出现无角色用户 |
| 5 | 创建用户 | 默认绑定「AiSight Demo」项目，该项目不设成员上限 |
| 6 | 用户从项目中移除 | 其在该项目下创建的用例/缺陷归属不变，仅收回访问权限 |
| 7 | 用户角色调整（增/减/升/降） | 仅 `admin` 可操作；变更立即生效，已创建的数据保留，但按新角色权限控制后续操作 |
| 8 | 删除空项目 | 同步清理所有成员关系记录 |

系统预置 6 个全局角色，不可删除、不可修改标识名（`name`），仅允许修改描述。

| 角色标识 | 显示名称 | 定位 | 典型用户 |

| `admin` | 系统管理员 | 全权限，系统配置、用户/角色/项目管理 | 运维负责人、平台管理员 |
| `manager` | 项目管理员 | 项目级管理权限，可管理项目成员、配置模块 | 测试经理、QA Lead |
| `tester` | 测试工程师 | 用例增删改、执行、提交缺陷、导入导出 | 测试执行人员 |
| `reviewer` | 评审员 | 评审用例、修改评审状态，不可增删用例 | 开发 TL、产品经理 |

| `readonly` | 只读访客 | 纯查看，不可修改任何数据 | 外部审计、上级领导 |

##### 2.2.3 角色业务规则

1. 平台采用全局角色模型，角色不随项目切换而变化
2. 用户-角色为多对多关系，一个用户可同时拥有多个角色，取并集权限
3. 用户角色的分配、调整、回收仅允许 `admin` 执行；`manager` 仅能管理项目成员，不可修改角色
4. 用户创建时必须至少选择 1 个非 `admin` 角色，支持一次选择多个角色
5. 用户创建时角色选择为多选，最终权限按所选角色的并集生效
6. 用户角色的分配仅允许 `admin` 执行；用户创建时不可分配 `admin` 角色（仅 `admin` 可在编辑中授予）
7. 预置角色不可删除（后端返回 409）；自定义角色删除前需检查关联用户，有关联则禁止删除并提示
8. 自定义角色不允许配置用户管理、角色管理、项目创建、项目删除等系统级权限
9. 角色权限被修改后，所有关联用户立即生效，无需重新登录，并记录审计日志
10. 用户仅由管理员创建，不开放自主注册
11. `developer` 角色的核心定位是「缺陷协作」，不可修改用例，但可查看所有测试数据
12. 数据可见性：所有角色均可看到当前项目内的全部用例，不按创建人隔离
13. 邮箱在任何场景下均不可修改（用户管理编辑、个人中心，后端统一硬拒绝）
14. 用户必须绑定至少一个项目（`admin` 角色除外，admin 自动拥有所有项目访问权）
15. 个人中心所有用户均可访问，可查看自己的角色、关联项目等信息
16. 无权限访问页面时，前端不跳转，直接在页面内展示占位符提示「无权限访问该页面，请联系管理员」

##### 2.3.1 项目业务规则

1. 项目名称全局唯一，创建/编辑时前后端双重校验
2. 归档项目内所有数据变为只读（用例、目录、缺陷、执行记录均不可增删改）
3. 仅 `admin` 可恢复已归档项目为活跃状态
4. 仅 `admin` 可删除项目，且必须满足：已归档 + 无用例 + 无缺陷
5. 新建项目后自动创建根目录「全部用例」
6. 项目选择器按权限过滤：普通用户只看到已加入的项目，admin 看到全部
7. 项目成员关系仅用于控制项目访问范围，不承载角色语义；项目成员管理不改变用户全局角色
8. 项目中带 `admin` / `manager` 全局角色的成员为受保护成员，不可被移除；`admin` 和 `manager` 仅可移除其他成员
9. 切换项目后前端自动刷新所有业务模块数据，不保留上一个项目的缓存状态
10. 种子项目「AiSight Demo」不允许删除和归档（前后端双重限制）
11. 仪表盘数据范围：`admin` / `manager` 看所有项目汇总，其他角色只看当前选中项目的统计
12. 项目列表默认按最后活跃时间降序，归档项目始终排在底部

## 6. 验收标准（摘要）

1. 删除预置角色返回 409
2. 删除 admin 用户返回 409，前端按钮禁用
3. 新建用户必须至少选择 1 个非 `admin` 角色，支持多选，且字段校验生效
4. 用户编辑允许修改角色和关联项目，但不允许保存为无角色用户；非 `admin` 用户不允许保存为无项目用户
5. 自定义角色不可配置用户管理、角色管理、项目创建、项目删除等系统级权限
6. 角色权限变更后，关联用户立即生效，无需重新登录，且记录审计日志
7. 项目中带 `admin` / `manager` 角色的成员不可被移除
8. 个人中心邮箱不可编辑，后端拒绝邮箱变更
9. 头像上传成功后可展示，失败有回退
10. 刷新后停留当前导航与项目，数据自动加载
11. 目录删除提示保留删除数量
