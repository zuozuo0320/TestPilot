# 测试管理平台-测试智编模块研发拆解与步骤模型设计

> 版本：v0.2（更新于 2026-04-02）
> 关联文档：`测试管理平台-测试智编模块需求方案-20260327.md`、`测试管理平台-测试智编模块详细设计-20260328.md`、`测试管理平台-测试智编模块数据与接口设计-20260328.md`、`测试管理平台-测试智编模块技术架构与集成方案-20260328.md`
> 文档定位：用于默认“录制增强模式”方案下的研发拆解、任务分工与 `step_model_json` 结构收口

## 1. 目标

本文档解决两件事：
- 将“录制增强模式为主、AI直生模式为辅”的方案拆解为可执行研发任务
- 定义录制增强模式下用于 AI 重构的 `step_model_json` 数据结构

本文档默认遵循以下前提：
- 首期默认生成模式为 `RECORDING_ENHANCED`
- 首期最终输出脚本固定为 `Playwright TypeScript`
- `Playwright` 录制负责保留基础 locator 与执行逻辑
- AI 负责按团队标准框架重构代码，不负责改变业务执行逻辑

## 2. 总体落地策略

建议按“先保真、后成型、再验证”落地：

1. `Playwright` 录制保留原始操作与原始 locator
2. 执行服务提取 `step_model_json`，同时构建 `generation_steps / normalization_items / generation_script`
3. AI 根据生成阶段投影结果重构标准脚本
4. 平台对标准脚本执行回放验证
5. 通过后进入人工确认与版本管理

关键原则：
- 原始录制稿必须保留
- 步骤模型必须可追溯
- AI 重构不得擅自更改关键步骤顺序
- 最终是否可用，以回放验证结果为准

## 3. 研发拆解

### 3.1 前端 TestFront

P0 必做：
- 在新建任务页增加 `generation_mode` 选择，默认 `录制增强模式`
- 在任务详情页根据 `generation_mode` 展示 `开始录制` 或 `执行生成`
- 在任务详情页新增“原始录制稿 / 步骤模型”展示区
- 在脚本区支持“原始录制稿 / 标准化脚本”对照查看
- 在列表页、详情页展示生成模式和生成来源

P1 建议：
- 录制结束后自动刷新脚本重构状态
- 步骤模型展示支持折叠查看每一步的 locator、动作和断言线索
- 在脚本对比区支持“录制稿 vs 标准脚本”差异查看

前端接口依赖：
- `POST /api/ai-script/tasks`
- `POST /api/ai-script/tasks/{taskId}/recording/start`
- `POST /api/ai-script/tasks/{taskId}/recording/finish`
- `POST /api/ai-script/tasks/{taskId}/recording/fail`
- `GET /api/ai-script/tasks/{taskId}/recordings/latest`
- `GET /api/ai-script/tasks/{taskId}/current-script`

### 3.2 平台后端 TestPilot

P0 必做：
- 新增 `generation_mode` 字段及相关校验
- 新增录制相关接口路由与权限控制
- 新增 `ai_script_recording_session` 持久化
- 扩展 `source_type` 枚举与写库逻辑
- 在任务详情、脚本详情接口返回生成模式、录制稿引用、步骤模型引用
- 增加 `AI_DIRECT` 与 `RECORDING_ENHANCED` 的执行分流

P1 建议：
- 录制会话状态与任务状态分离展示
- 增加录制稿、步骤模型、标准脚本之间的追溯接口
- 在任务列表接口增加 `generation_mode` 筛选能力

后端关键判断：
- `execute` 接口只服务 `AI_DIRECT`
- `recording/start`、`recording/finish` 只服务 `RECORDING_ENHANCED`
- 后端必须校验模式与接口匹配关系，不能只靠前端按钮态

### 3.3 执行服务

P0 必做：
- 接入 `Playwright codegen` 或等价录制能力
- 保存原始录制稿
- 提取 `step_model_json`
- 构建生成阶段投影：`generation_steps / normalization_items / generation_script`
- 根据 `step_model_json` 调用 AI 做标准框架重构
- 保存标准脚本版本
- 对标准脚本执行回放验证

P1 建议：
- 对录制稿提取“定位风险级别”
- 对步骤模型生成“断言候选”
- 在 AI 重构失败时提供降级结果：仅返回原始录制稿

执行服务内部推荐模块：
- `recording_runner`
- `step_model_builder`
- `framework_refactor_runner`
- `validation_runner`
- `artifact_uploader`

### 3.4 测试与验收

P0 必做：
- 覆盖默认录制增强模式完整链路
- 覆盖 AI直生模式保底链路
- 验证录制稿、步骤模型、标准脚本之间的关联正确性
- 验证回放失败后可重新编辑并重验
- 验证 `source_type` 与 `generation_mode` 展示正确

重点验收点：
- 录制稿可追溯
- AI 重构前后关键步骤不丢失
- 标准脚本可回放
- 回放通过后才能确认

## 5. `step_model_json` 设计目标

`step_model_json` 的定位不是最终脚本，也不是原始录制稿。

它的作用是：
- 抽取稳定执行逻辑
- 固化步骤顺序
- 固化 locator 与输入
- 给 AI 提供受控输入
- 支持后续重新生成标准脚本

因此它必须具备这些特征：
- 结构化
- 可回放理解
- 可用于 AI 重构
- 可用于人工审核
- 不与具体代码模板强绑定

## 6. `step_model_json` 当前落地结构

以下结构是当前代码实现中的权威口径，优先级高于旧版“富字段理想模型”。旧版 `assertion_candidates / metadata / locator.preferred` 等设计可以继续保留为扩展方向，但当前生成链路实际依赖的是下面这组字段：

```json
{
  "total_steps": 10,
  "steps": [],
  "generation_total_steps": 9,
  "generation_steps": [],
  "normalization_items": [],
  "generation_script": "..."
}
```

顶层字段说明：
- `steps`：原始录制步骤，完整保留录制动作顺序与原始 locator
- `total_steps`：原始步骤总数
- `generation_steps`：平台归一化后的生成步骤，是 LLM 与代码生成链路的唯一业务动作序列
- `generation_total_steps`：生成步骤总数
- `normalization_items`：平台显式执行过的归一化动作清单
- `generation_script`：只按 `normalization_items` 裁剪后的生成用录制稿

## 7. `steps` 与 `generation_steps` 结构

当前实现中，`steps` 与 `generation_steps` 都采用轻量结构。

### 7.1 `steps` 原始步骤

```json
{
  "step_no": 1,
  "action_type": "CLICK",
  "locator": "page.getByRole('button', { name: '新建任务' })",
  "input_value": "",
  "page_url": "https://test.example.com/taskWorkbench"
}
```

原始步骤至少包含：
- `step_no`
- `action_type`
- `locator`
- `input_value`
- `page_url`

### 7.2 `generation_steps` 生成步骤

```json
{
  "step_no": 1,
  "raw_step_no": 1,
  "action_type": "CLICK",
  "locator": "page.getByRole('button', { name: '新建任务' })",
  "input_value": "",
  "page_url": "https://test.example.com/login",
  "normalized": false
}
```

生成步骤至少包含：
- `step_no`
- `raw_step_no`
- `action_type`
- `locator`
- `input_value`
- `page_url`
- `normalized`

关键约束：
- `generation_steps` 允许重编序号，但必须保留 `raw_step_no` 追溯原始录制来源
- locator 仍必须直接来自原始录制稿，不允许改写
- 若 `steps` 与 `generation_steps` 不一致，生成链路必须以 `generation_steps` 为准

## 8. 当前固定归一化规则

当前平台只允许一种程序侧归一化：

```json
{
  "type": "collapse_duplicate_opener_click",
  "kept_raw_step_no": 3,
  "removed_raw_step_no": 4,
  "locator": "page.getByRole('button', { name: '新建任务' })",
  "next_action_type": "INPUT"
}
```

规则说明：
- 仅用于折叠“第一次点击已打开弹窗/抽屉/表单，第二次同定位器点击为冗余”的 opener 重复点击
- 只允许平台程序产生，LLM 不允许自行发明新的归一化类型
- `generation_script` 只能删除 `normalization_items` 明确标记的冗余步骤，其余录制内容必须原样保留

## 9. AI 重构约束建议

AI 使用 `step_model_json` 重构标准脚本时，必须遵循：

1. 不得改写任何原始 locator
2. 若存在 `generation_steps`，必须以 `generation_steps` 作为生成阶段唯一动作序列
3. 除 `normalization_items` 明确允许的归一化外，不得擅自删除、合并、重排步骤
4. 允许按标准框架抽方法、拆 `test.step`、补中文注释
5. 允许补充高置信度等待与断言，但不能改变业务语义
6. 若某一步无法稳定表达，必须输出风险提示或人工审核项，不得静默删除

建议给 AI 的系统约束可以概括为：
- “允许重构代码结构，不允许改变业务执行逻辑；允许使用平台显式归一化结果，不允许自行新增归一化。”

## 10. 标准脚本生成建议

从 `step_model_json` 到标准脚本，建议分两段：

### 10.1 规则层生成

适合规则生成的内容：
- 文件模板
- `test.describe`
- `test`
- `test.step`
- fixture 注入
- 注释骨架
- 公共 import

### 10.2 AI 增强层生成

适合 AI 生成的内容：
- 步骤命名优化
- 注释语义化
- 断言选择
- 风险提示文案
- 代码可读性整理

推荐策略：
- 结构靠规则
- 语义靠 AI

## 11. 当前最小可交付范围

当前代码已经落地并应作为默认口径的能力包括：

- 任务支持 `RECORDING_ENHANCED / AI_DIRECT`
- 默认 `RECORDING_ENHANCED`
- 录制稿可落库
- `step_model_json` 可落库
- `step_model_json` 已包含 `generation_steps / normalization_items / generation_script`
- AI 能基于生成阶段投影结果生成标准脚本
- 标准脚本已切换为多项目、多文件工程化输出
- 标准脚本能回放验证
- 验证通过后可确认

当前仍可继续增强、但不是主链路阻塞项的内容：
- 录制稿与标准脚本可视化逐行 diff
- 更丰富的断言候选与风险画像
- 更多平台级归一化规则
- 更细粒度的人工审核工作台

## 12. 建议下一步

如果继续推进，推荐马上做下面两件事：

1. 持续让 `需求方案 / 详细设计 / 接口文档 / 验收清单` 与当前已实现的 `generation_steps` 口径保持同步
2. 基于真实项目样本继续补充平台侧固定归一化规则，但必须坚持“原始 locator 零改写、LLM 只做结构组织”的边界
