# 测试管理平台-测试智编模块研发拆解与步骤模型设计

> 版本：v0.1（创建于 2026-03-29）
> 关联文档：`测试管理平台-测试智编模块需求方案-20260327.md`、`测试管理平台-测试智编模块详细设计-20260328.md`、`测试管理平台-测试智编模块数据与接口设计-20260328.md`、`测试管理平台-测试智编模块技术架构与集成方案-20260328.md`
> 文档定位：用于默认“录制增强模式”方案下的研发拆解、任务分工与 `step_model_json` 结构收口

## 1. 目标

本文档解决两件事：
- 将“录制增强模式为主、AI直生模式为辅”的方案拆解为可执行研发任务
- 定义录制增强模式下用于 AI 重构的 `step_model_json` 数据结构

本文档默认遵循以下前提：
- 首期默认生成模式为 `RECORDING_ENHANCED`
- 首期最终输出脚本固定为 `Playwright TypeScript`
- `Playwright` 录制负责保留基础 locator 与执行逻辑
- AI 负责按团队标准框架重构代码，不负责改变业务执行逻辑

## 2. 总体落地策略

建议按“先保真、后成型、再验证”落地：

1. `Playwright` 录制保留原始操作
2. 执行服务提取步骤模型
3. AI 根据步骤模型重构标准脚本
4. 平台对标准脚本执行回放验证
5. 通过后进入人工确认与版本管理

关键原则：
- 原始录制稿必须保留
- 步骤模型必须可追溯
- AI 重构不得擅自更改关键步骤顺序
- 最终是否可用，以回放验证结果为准

## 3. 研发拆解

### 3.1 前端 TestFront

P0 必做：
- 在新建任务页增加 `generation_mode` 选择，默认 `录制增强模式`
- 在任务详情页根据 `generation_mode` 展示 `开始录制` 或 `执行生成`
- 在任务详情页新增“原始录制稿 / 步骤模型”展示区
- 在脚本区支持“原始录制稿 / 标准化脚本”对照查看
- 在列表页、详情页展示生成模式和生成来源

P1 建议：
- 录制结束后自动刷新脚本重构状态
- 步骤模型展示支持折叠查看每一步的 locator、动作和断言线索
- 在脚本对比区支持“录制稿 vs 标准脚本”差异查看

前端接口依赖：
- `POST /api/ai-script/tasks`
- `POST /api/ai-script/tasks/{taskId}/recording/start`
- `POST /api/ai-script/tasks/{taskId}/recording/finish`
- `GET /api/ai-script/tasks/{taskId}/recordings/latest`
- `GET /api/ai-script/tasks/{taskId}/scripts/current`

### 3.2 平台后端 TestPilot

P0 必做：
- 新增 `generation_mode` 字段及相关校验
- 新增录制相关接口路由与权限控制
- 新增 `ai_script_recording_session` 持久化
- 扩展 `source_type` 枚举与写库逻辑
- 在任务详情、脚本详情接口返回生成模式、录制稿引用、步骤模型引用
- 增加 `AI_DIRECT` 与 `RECORDING_ENHANCED` 的执行分流

P1 建议：
- 录制会话状态与任务状态分离展示
- 增加录制稿、步骤模型、标准脚本之间的追溯接口
- 在任务列表接口增加 `generation_mode` 筛选能力

后端关键判断：
- `execute` 接口只服务 `AI_DIRECT`
- `recording/start`、`recording/finish` 只服务 `RECORDING_ENHANCED`
- 后端必须校验模式与接口匹配关系，不能只靠前端按钮态

### 3.3 执行服务

P0 必做：
- 接入 `Playwright codegen` 或等价录制能力
- 保存原始录制稿
- 提取 `step_model_json`
- 根据 `step_model_json` 调用 AI 做标准框架重构
- 保存标准脚本版本
- 对标准脚本执行回放验证

P1 建议：
- 对录制稿提取“定位风险级别”
- 对步骤模型生成“断言候选”
- 在 AI 重构失败时提供降级结果：仅返回原始录制稿

执行服务内部推荐模块：
- `recording_runner`
- `step_model_builder`
- `framework_refactor_runner`
- `validation_runner`
- `artifact_uploader`

### 3.4 测试与验收

P0 必做：
- 覆盖默认录制增强模式完整链路
- 覆盖 AI直生模式保底链路
- 验证录制稿、步骤模型、标准脚本之间的关联正确性
- 验证回放失败后可重新编辑并重验
- 验证 `source_type` 与 `generation_mode` 展示正确

重点验收点：
- 录制稿可追溯
- AI 重构前后关键步骤不丢失
- 标准脚本可回放
- 回放通过后才能确认

## 4. 推荐研发顺序

建议按以下顺序推进：

### 第一阶段：数据与接口骨架

- 增加 `generation_mode`
- 增加 `ai_script_recording_session`
- 打通录制相关接口占位
- 打通详情接口字段返回

目标：
- 前后端能先看到“录制增强模式”完整任务形态

### 第二阶段：录制稿落库

- 接入 `Playwright` 录制
- 保存原始录制稿
- 保存录制证据
- 页面可查看录制结果

目标：
- 不依赖 AI，也能先完成录制原稿沉淀

### 第三阶段：步骤模型提取

- 从录制稿提取结构化步骤
- 保存 `step_model_json`
- 页面可查看步骤模型

目标：
- 把“执行逻辑”和“代码生成”彻底解耦

### 第四阶段：AI 标准化重构

- 将步骤模型交给 AI
- 输出标准 `Playwright TypeScript` 脚本
- 保存为新脚本版本

目标：
- 完成“录制保真 + AI 成型”

### 第五阶段：回放验证与确认闭环

- 对标准脚本执行回放验证
- 打通确认、废弃、重验
- 补齐审计日志

目标：
- 完成首期闭环交付

## 5. `step_model_json` 设计目标

`step_model_json` 的定位不是最终脚本，也不是原始录制稿。

它的作用是：
- 抽取稳定执行逻辑
- 固化步骤顺序
- 固化 locator 与输入
- 给 AI 提供受控输入
- 支持后续重新生成标准脚本

因此它必须具备这些特征：
- 结构化
- 可回放理解
- 可用于 AI 重构
- 可用于人工审核
- 不与具体代码模板强绑定

## 6. `step_model_json` 顶层结构建议

```json
{
  "model_version": "1.0",
  "task_id": 9001,
  "recording_id": 5001,
  "generation_mode": "RECORDING_ENHANCED",
  "framework_type": "Playwright",
  "language": "TypeScript",
  "start_url": "https://test.example.com/login",
  "case_ids": [2001, 2002],
  "scenario_desc": "使用测试账号登录系统并进入用户列表页",
  "preconditions": [],
  "steps": [],
  "assertion_candidates": [],
  "risk_hints": [],
  "metadata": {}
}
```

顶层字段建议：
- `model_version`：步骤模型版本，方便后续演进
- `task_id`：关联任务
- `recording_id`：关联录制会话
- `generation_mode`：标识来源模式
- `framework_type`：首期固定 `Playwright`
- `language`：首期固定 `TypeScript`
- `steps`：核心步骤数组
- `assertion_candidates`：断言候选
- `risk_hints`：风险提示
- `metadata`：扩展信息

## 7. `steps` 数组结构建议

每一步建议结构如下：

```json
{
  "step_no": 1,
  "step_name": "输入用户名",
  "action_type": "INPUT",
  "page_url": "https://test.example.com/login",
  "locator": {
    "preferred": "getByLabel('用户名')",
    "strategy": "label",
    "selector_value": "用户名",
    "fallbacks": [
      "getByPlaceholder('请输入用户名')",
      "locator('#username')"
    ],
    "stability": "HIGH"
  },
  "input": {
    "value_masked": "***",
    "value_source": "account_ref.username"
  },
  "expected_effect": "用户名输入框被正确填充",
  "assertion_candidates": [
    "expect(page.getByLabel('用户名')).toHaveValue(...)"
  ],
  "attachments": [],
  "is_critical": true
}
```

每一步建议至少包含：
- `step_no`
- `step_name`
- `action_type`
- `page_url`
- `locator`
- `expected_effect`
- `is_critical`

如有输入类动作，再补：
- `input.value_masked`
- `input.value_source`

## 8. locator 结构建议

`locator` 是录制增强模式里最重要的稳定资产，建议结构如下：

```json
{
  "preferred": "getByRole('button', { name: '登录' })",
  "strategy": "role",
  "selector_value": "登录",
  "fallbacks": [
    "getByText('登录')",
    "locator('.login-btn')"
  ],
  "stability": "MEDIUM",
  "notes": "按钮文案稳定，但 class 可能变化"
}
```

字段建议：
- `preferred`：当前首选 locator
- `strategy`：`testid / role / label / text / css / xpath`
- `selector_value`：核心匹配值
- `fallbacks`：备用 locator
- `stability`：`HIGH / MEDIUM / LOW`
- `notes`：定位说明

## 9. 断言候选结构建议

断言候选建议和步骤分开存，便于 AI 组合：

```json
{
  "assertion_id": "A1",
  "step_no": 5,
  "assertion_type": "URL_CONTAINS",
  "target": "/user/list",
  "confidence": "HIGH",
  "recommended": true,
  "reason": "提交登录后页面跳转到用户列表页"
}
```

建议字段：
- `assertion_id`
- `step_no`
- `assertion_type`
- `target`
- `confidence`
- `recommended`
- `reason`

## 10. AI 重构约束建议

AI 使用 `step_model_json` 重构标准脚本时，必须遵循：

1. 不得改变 `steps` 的执行顺序
2. 不得删除 `is_critical = true` 的步骤
3. 不得擅自替换 `locator.preferred`，除非明确标注采用 fallback
4. 允许按标准框架抽方法、拆 `test.step`、补注释
5. 允许补充高置信度断言
6. 低置信度断言只允许作为建议输出
7. 若某一步无法稳定表达，必须输出风险提示，不得静默删除

建议给 AI 的系统约束可以概括为：
- “允许重构代码结构，不允许改变业务执行逻辑”

## 11. 标准脚本生成建议

从 `step_model_json` 到标准脚本，建议分两段：

### 11.1 规则层生成

适合规则生成的内容：
- 文件模板
- `test.describe`
- `test`
- `test.step`
- fixture 注入
- 注释骨架
- 公共 import

### 11.2 AI 增强层生成

适合 AI 生成的内容：
- 步骤命名优化
- 注释语义化
- 断言选择
- 风险提示文案
- 代码可读性整理

推荐策略：
- 结构靠规则
- 语义靠 AI

## 12. 最小可交付范围

如果要快速交付第一版，我建议把范围卡在这里：

- 任务支持 `RECORDING_ENHANCED / AI_DIRECT`
- 默认 `RECORDING_ENHANCED`
- 录制稿可落库
- `step_model_json` 可落库
- AI 能基于步骤模型生成标准脚本
- 标准脚本能回放验证
- 验证通过后可确认

首期可以不做：
- 录制稿与标准脚本可视化逐行 diff
- 复杂步骤自动抽 Page Object
- 多文件输出
- 高级断言自动编排
