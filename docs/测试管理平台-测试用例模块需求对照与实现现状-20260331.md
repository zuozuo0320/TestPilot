# 测试管理平台-测试用例模块需求对照与实现现状（2026-03-31）

## 1. 分析范围

- 需求依据：`TestPilot/docs/测试管理平台-需求文档-20260308.md` 中“模块三：测试用例管理”
- 分析方式：静态阅读当前前后端代码，不做代码修改、不做联机验证
- 分析时间点：2026-03-31

本次主要查看的代码范围如下：

- 后端：`internal/api/handler_testcase.go`、`internal/api/handler_module.go`、`internal/api/handler_attachment.go`、`internal/api/handler_xlsx.go`
- 后端：`internal/service/testcase_service.go`、`internal/service/module_service.go`、`internal/service/attachment_service.go`、`internal/service/xlsx_service.go`

## 2. 总体结论

- 当前代码已经具备“测试用例基础录入、列表查询、基础筛选、批量操作、附件管理、基础导入导出”的骨架能力。
- 当前代码还不能按照主需求文档对测试用例模块进行完整验收，核心缺口集中在“用例生命周期与版本规则”“目录树正式落库与规则约束”“导入导出细则”“角色权限边界”四块。
- 当前目录树实现并不是正式的“目录模块落库方案”，而是前端基于 `module_path` 字段临时派生出来的树；这块与需求文档差距最大。

## 3. 当前已实现功能

### 3.1 用例列表与基础操作

- 已实现分页展示。
- 已实现关键字搜索，支持按 `ID / title / tags` 查询。
- 已实现等级、评审结果、执行结果、标签筛选。
- 已实现创建人、更新人筛选。
- 已实现创建时间、更新时间范围筛选。
- 已实现按 `id / created_at / updated_at` 排序。
- 已实现空状态展示。
- 已实现用例新建、编辑、删除、复制。
- 已实现用例 ID 点击复制。
- 已实现批量删除、批量修改等级、批量移动。

### 3.2 编辑页与步骤编辑

- 已实现抽屉式编辑页。
- 已实现基础字段编辑：标题、等级、目录、评审结果、执行结果、标签、前置条件、步骤、备注。
- 已实现步骤新增、删除、拖拽排序。

### 3.3 附件、历史、关联、导入导出骨架

- 已实现附件上传、列表、下载、删除。
- 已实现单文件 10MB 限制。
- 已提供编辑历史查询接口。
- 已提供关联用例查询、新增、删除接口。
- 已提供 xlsx 导入、导出接口。

### 3.4 目录树与列表联动的现有能力

- 前端侧边栏已展示“全部用例 / 未规划用例 / 普通目录树”。
- 点击目录后，列表会按当前目录及其子目录范围过滤。
- 切换目录时会保留当前其他筛选条件。
- 切换目录时会重置到第 1 页。
- 切换目录后会清空当前勾选状态。
- 编辑用例修改目录后，列表会重新加载。
- 批量移动到“未规划用例”已支持，移动后列表会重新加载。

## 4. 部分实现或与需求不一致

### 4.1 目录树当前不是正式落库实现

- 前端没有实际调用 `src/api/module.ts` 中的目录接口。
- 当前目录树主要由 `TestCasePage.vue` 中的 `treeRows + module_path` 动态派生。
- 当前新建空目录只写入前端内存 `customModulePaths`，没有持久化；页面刷新后空目录会丢失。
- 后端虽然存在 `modules` 表和相关接口，但当前前端未接入，因此“目录正式存储、多人共享、刷新可见”并未完成。

### 4.2 目录创建、重命名、删除与需求规则不一致

- 目录重命名时，前端只会尝试更新当前页 `rows` 里的用例，不会覆盖项目下该目录及其子目录的全部用例。
- 目录删除时，前端不是“禁止删除非空目录”，而是尝试删除当前页中命中的用例，然后移除目录路径。
- 目录删除时对“非空”的判断基于当前页 `rows`，不是全量项目数据，存在误删/漏删风险。
- 后端 `module` 删除接口只校验“是否有子目录”，没有校验“目录下是否有用例”，不满足“仅空叶子目录可删”的硬规则。
- 后端 `module` 重命名也没有同步更新子目录路径和已归属用例。

### 4.3 目录统计口径与需求不一致

- 侧边栏普通目录后的数量，当前展示的是“当前目录及所有子目录累计数量”，不是“直属用例数”。
- “全部用例”节点当前没有显示总数。
- “未规划用例”节点有数量展示，但该逻辑来自前端派生，不是后端统一计算。

### 4.4 目录树功能只做了基础联动，未达到完整需求

- 未实现目录搜索。
- 未实现搜索命中后自动展开父级路径。
- 未实现“未找到匹配目录”提示。
- 未实现拖拽排序。
- 未实现同级排序持久化。
- “未规划用例”目前主要是前端 UI 层面的特殊节点，后端没有形成完整的硬规则保护。

### 4.5 新建用例默认目录与需求不一致

- 需求要求：若当前选中具体目录，新建用例默认带入当前目录；若选中“全部用例”，默认带入“未规划用例”。
- 当前 `openCreate()` 里固定写死为 `/未规划用例`，不会读取当前选中的目录。

### 4.6 列表字段与需求定义不一致

- 当前列表未实现“状态”字段。
- 当前列表未实现“版本号”字段。
- 当前列表展示了 `创建人 / 创建时间 / 操作`，但需求默认列要求的是“用例名称、状态、版本号、等级、所属模块、评审结果、执行结果、标签、更新人、更新时间”。
- 当前表头中的“状态”实际对应的是执行结果，不是用例生命周期状态。
- 当前“执行结果”只是直接展示 `exec_result` 字段，不是“执行记录中的最新一次执行结果”。

### 4.7 编辑页右侧信息大多还是占位内容

- “版本追踪”区块当前是静态占位数据，不是实际用例版本数据。
- “最近动态”区块当前是静态占位数据，不是实际编辑历史。
- “历史缺陷关联”当前是静态占位展示。
- 前端虽然会请求 `listCaseHistory`，但页面并没有按需求真实渲染“编辑历史/评审记录/执行记录”。
- “后置条件”当前只是占位，不是正式字段能力。

### 4.8 步骤编辑只做了部分

- 已实现新增、删除、拖拽排序。
- 未实现步骤复制。
- 未实现“在上方插入”。
- 未实现“在下方插入”。
- 未实现步骤类型切换。

### 4.9 导入导出仅为基础骨架，和需求细则差距较大

- 导出接口当前没有接收列表筛选参数，实际是全量导出，不满足“严格遵循当前筛选条件”。
- 导出文件名当前是 `testcases_项目ID_时间.xlsx`，不包含项目名称。
- 前端页面上的“导出报告”按钮当前没有绑定测试用例导出逻辑。
- 页面里虽然有隐藏的导入 input，但没有清晰的可见导入入口。
- 当前导出字段包含 `ID / 标题 / 等级 / 模块 / 评审结果 / 执行结果 / 标签 / 前置条件 / 步骤 / 备注 / priority`，与需求中的默认导出字段不一致。
- 当前导入会读取评审结果、执行结果等字段，这与“导入不允许写入状态/评审结果/执行结果/版本号/记录数据”的需求不一致。
- 当前导入结果只返回 `created / skipped`，没有返回总数、失败数、失败明细、重复提醒。
- 当前导入没有“目录为空时导入到当前选中目录”的能力，后端默认会落到 `/未分类`。
- 当前导入模板不是“步骤1/预期结果1、步骤2/预期结果2...”成对列展开模型。

### 4.10 “未规划用例”与后端默认值不一致

- 需求确认后的目录名称是“未规划用例”。
- 当前前端逻辑使用 `/未规划用例`。
- 当前后端新建/更新/导入默认目录仍是 `/未分类`。
- 这会导致前后端对“未绑定目录用例”的归属定义不一致。

### 4.11 权限边界未按需求落细

- 当前测试用例、目录、附件、xlsx 接口基本都只校验“是否有项目访问权限”。
- 当前没有按 `tester / reviewer / manager / admin / developer / readonly` 做测试用例模块的细粒度权限控制。
- 当前导入导出没有按“仅 `tester` / `manager` / `admin` 可用”做后端限制。
- 当前 `developer / readonly` 也没有在后端被限制为只读。

### 4.12 生命周期相关字段和规则没有真正建立

- 当前 `TestCase` 模型没有“状态”字段。
- 当前 `TestCase` 模型没有“版本号”字段。
- 当前创建和编辑时可以直接写 `review_result`、`exec_result`，并没有经过状态机流转。
- 当前没有“提审 / 评审 / 撤回 / 废弃 / 恢复”接口和规则。
- 当前没有“实质变更后版本 +1、已生效回草稿、结果重置”的业务逻辑。

### 4.13 编辑历史、评审记录、执行记录只存在局部骨架

- 当前有 `CaseHistory` 表和查询接口，但代码里没有真正写入历史记录。
- 当前没有评审记录模型和正式接口。
- 当前没有执行记录模型和正式接口。
- 因此“详情展示基础信息、当前状态、当前版本号、评审记录、执行记录、编辑历史”还未形成闭环。

### 4.14 关联用例仅有最小接口，前端未接入

- 后端已有关联用例最小 CRUD 接口。
- 前端页面当前没有关联用例展示和操作入口。
- 关联规则也未见更完整的校验约束。

### 4.15 更新人字段当前不会在编辑时刷新

- 用例更新逻辑没有把当前操作人写回 `updated_by`。
- 这意味着列表里的“更新人”并不能真实反映最后修改人。

### 4.16 名称唯一性规则比需求更严格

- 当前数据库对用例使用的是“同一项目内标题唯一”约束。
- 需求文档中已明确：同一项目、同一目录下名称重复允许导入，只返回重复提醒且不计入失败。
- 按现状，项目内任意目录的重名用例都会被数据库直接拦截。

## 5. 当前明确未实现的需求

### 5.1 P0 未完成

- 用例生命周期状态：`草稿 / 待评审 / 已生效 / 已废弃`
- 提审、评审、撤回、废弃、恢复等状态流转规则
- 按角色控制生命周期操作权限
- 仅草稿可删除；批量删除只允许草稿；勾选其他状态时返回不可删数量
- 已生效用例实质变更后回到草稿并重置评审结果、执行结果
- 目录删除仅允许空叶子目录，且服务端硬约束
- “未规划用例”为系统内置目录的服务端硬保护
- 新建用例默认带入当前选中目录

### 5.2 P1 未完成

- 版本号自动管理：新建为 V1，实质变更后自动 +1
- 列表默认字段按需求展示“状态、版本号、最新执行结果”
- 已废弃用例默认隐藏、支持按状态/开关显示
- 导出严格按当前筛选结果导出
- 导出字段、文件名、包含已废弃规则
- 导入模板固定字段与“步骤/预期结果成对列”规则
- 导入失败明细、重复提醒、部分成功结果详情
- 导入目录为空时落当前选中目录
- 导入禁止写入状态/评审结果/执行结果/版本号/记录数据
- 目录搜索与搜索联动
- 目录同级排序与持久化
- 目录直属用例数统计
- “全部用例”总数展示
- 编辑历史真实落库
- 评审记录
- 执行记录
- 详情页真实展示状态/版本/评审记录/执行记录/编辑历史
- 富文本前置条件、备注
- 富文本图片粘贴
- 步骤复制、上插、下插、类型切换
- 自动保存/草稿

### 5.3 P2 未完成

- 列表列配置
- 行内快捷编辑
- 步骤模板
- 关联用例的前端交互与完整业务规则

## 6. 自动化测试覆盖现状

- 已有后端测试主要覆盖：创建、默认值、分页列表、更新、删除、列表筛选和排序。
- 目前未见测试覆盖：目录树接口、批量删除规则、批量移动联动、附件接口、xlsx 导入导出规则、编辑历史、关联用例、生命周期与版本规则。
- 前端仅看到路径规范化的小型单测，未见针对测试用例页面主流程的系统性测试。

## 7. 结论

- 当前测试用例模块可以支持“基础录入和基础查询”，但还不能视为“已按需求完成”。
- 如果后续要进入正式开发或验收，建议先优先补齐三块：用例生命周期与版本模型、目录树正式落库与规则收口、导入导出规则闭环。
- 从差距大小看，目录树和生命周期是当前最需要先收敛设计与实现口径的部分。
