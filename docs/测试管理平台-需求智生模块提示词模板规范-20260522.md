## 1. 适用范围

本文档适用于：

- `ai_skills.prompt_template` 字段中所有提示词的编写
- Executor 端 `prompt_renderer` 模块的渲染逻辑
- Executor 端 `schema_validator` 模块的输出校验
- Admin 在系统管理中创建/编辑自定义 Skill 的编写规则

不适用于：

- 测试智编模块的 Playwright AI 重构（见 `测试管理平台-测试智编模块AI重构提示词规范`）
- 用例评审 v0.2 中 Layer 2 LLM 检查（Phase 2 接入时另立规范）

---

## 2. AI 角色定义

AI 在本模块的定位是 **"受控用例草稿生成器"**，不是自由创作者。

**AI 允许做的事**

- 基于给定需求文本拆解出可独立验证的测试场景
- 为每个场景产出标题、级别、前置、步骤、后置、备注、建议标签
- 在用户已有标签库中选取合适标签
- 自评置信度（0.00–1.00）
- 输出风险提示（写在备注字段或 `notes` 区块）

**AI 不允许做的事**

- 编造需求文本中不存在的业务概念
- 输出执行性脚本代码（本模块不做自动化脚本生成，那是测试智编模块的职责）
- 自由生成项目标签库以外的新标签（建议标签会被 Go 侧校验，未匹配项不会自动落库）
- 突破 `max_cases` 上限
- 输出 JSON 以外的格式（无前导/尾随文字、无 Markdown 代码块包裹）
- 修改用例 status / version / review_result 等系统字段

## 3. 事实来源优先级

LLM 在生成过程中，事实来源按以下优先级取用，**高优先级覆盖低优先级**：

1. `requirement_text` — 截断后的需求文档纯文本，**唯一事实源**
2. `extra_prompt` — 用户在任务创建时的补充上下文，**事实补充与侧重点指引**
3. `project_context` — 项目名、项目描述等业务背景，**仅用于命名风格和遣词**
4. `existing_tags` — 项目已有标签库，**仅用于标签选取**
5. `few_shot_examples`（v1.1 接入） — 优质用例样例，**仅用于格式参考**
6. Skill 模板本身的描述与默认指令，**最低优先级**

**关键原则**：

- `requirement_text` 是事实源。文档没说的内容不允许凭空补全（如"想必应该验证 X"）
- `extra_prompt` 可以指定侧重点（如"重点关注权限场景"），但不能与文档矛盾
- 标签建议必须从 `existing_tags` 中选取；建议新标签时只能写入 `notes`，不进 `tags_suggested`

## 4. 占位符契约

### 4.1 占位符清单

下表是 prompt_template 中允许使用的全部占位符。Go 与 Executor 两端实现必须严格一致。

| 占位符 | 类型 | 是否必填 | 来源 | 缺省渲染值 |
|---|---|---|---|---|
| `{{requirement_text}}` | string | ✓ | `requirement_docs.raw_content`（截断后） | — 缺失即报错 |
| `{{project_context}}` | string | ✗ | 项目名 + 描述拼接，最多 500 字 | 空串 |
| `{{existing_tags}}` | string | ✗ | 标签库列表，逗号分隔 | 空串 |
| `{{max_cases}}` | int | ✓ | 任务表 `max_cases` 字段 | — 缺失即报错 |
| `{{default_level}}` | string | ✗ | 任务表 `default_level` 字段 | `"P2"` |
| `{{extra_prompt}}` | string | ✗ | 任务表 `extra_prompt` 字段 | 空串 |
| `{{few_shot_examples}}` | string | ✗ | v1.1 接入；v1.0 始终为空串 | 空串 |
| `{{output_schema_hint}}` | string | ✗ | 根据 `output_schema` 字段从内置 hint 表读取 | 空串 |

### 4.2 占位符语法

- 使用双花括号语法：`{{name}}`
- 大小写敏感，全小写
- 占位符前后允许任意空白字符（包括换行）
- **不支持表达式、条件、循环**，仅纯文本替换
- 未识别的占位符（如拼错）**不替换**，保留原文 —— 这是 Skill 编辑时的合法性校验信号
- 占位符内文本不做 HTML / Markdown 转义；调用方负责前置清洗

### 4.3 渲染顺序

1. Executor 收到 `/requirement/generate-cases` 请求，从入参解析所有占位符值
2. 对 `requirement_text` 单独做一次预处理：
   - 长度校验（>50 000 字符直接报错，调用方应已截断）
   - 去除连续 3 个以上空行 → 缩为 2 个
   - 去除尾随空白
3. 按 4.1 表逐个 `str.replace`（一次性、不递归）
4. 校验渲染后文本中是否还有 `{{xxx}}` 残留 —— 若有则视为模板错误，立即拒绝
5. 调用 LLM

### 4.4 渲染示例

**Skill 模板**

```
你是测试工程师。根据需求生成最多 {{max_cases}} 条用例，
默认级别 {{default_level}}。

【项目上下文】{{project_context}}
【已有标签】{{existing_tags}}
【用户补充】{{extra_prompt}}
【需求】
{{requirement_text}}

{{few_shot_examples}}
```

**入参**

```json
{
  "requirement_text": "用户登录页面需支持账号密码登录...",
  "project_context": "AiSight Demo - 内部测试管理系统",
  "existing_tags": "登录,权限,边界值",
  "max_cases": 30,
  "default_level": "P2",
  "extra_prompt": "侧重权限场景",
  "few_shot_examples": ""
}
```

**渲染结果**（喂给 LLM 的最终 prompt）

```
你是测试工程师。根据需求生成最多 30 条用例，
默认级别 P2。

【项目上下文】AiSight Demo - 内部测试管理系统
【已有标签】登录,权限,边界值
【用户补充】侧重权限场景
【需求】
用户登录页面需支持账号密码登录...


```

末尾的双空行来自 `{{few_shot_examples}}` 空串渲染 —— **这是预期行为，不要在模板里给该占位符前后加额外空行**。

## 5. 输出 Schema：`standard_case_v1`

### 5.1 JSON Schema 定义

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "standard_case_v1",
  "type": "object",
  "required": ["cases"],
  "additionalProperties": false,
  "properties": {
    "cases": {
      "type": "array",
      "minItems": 1,
      "maxItems": 50,
      "items": {
        "type": "object",
        "required": ["seq_no", "title", "level", "steps"],
        "additionalProperties": false,
        "properties": {
          "seq_no": {
            "type": "integer",
            "minimum": 1,
            "description": "任务内序号，1-based，必须连续且唯一"
          },
          "title": {
            "type": "string",
            "minLength": 4,
            "maxLength": 120,
            "description": "用例标题。简明描述被测场景与预期结果"
          },
          "level": {
            "type": "string",
            "enum": ["P0", "P1", "P2", "P3"],
            "description": "用例级别"
          },
          "precondition": {
            "type": "string",
            "maxLength": 2000,
            "description": "前置条件，纯文本"
          },
          "steps": {
            "type": "array",
            "minItems": 1,
            "maxItems": 30,
            "items": {
              "type": "object",
              "required": ["action", "expected"],
              "additionalProperties": false,
              "properties": {
                "action": {
                  "type": "string",
                  "minLength": 2,
                  "maxLength": 500,
                  "description": "操作描述"
                },
                "expected": {
                  "type": "string",
                  "minLength": 2,
                  "maxLength": 500,
                  "description": "预期结果"
                }
              }
            }
          },
          "postcondition": {
            "type": "string",
            "maxLength": 2000,
            "description": "后置条件，纯文本。无需后置时填空串"
          },
          "remark": {
            "type": "string",
            "maxLength": 1000,
            "description": "备注。AI 风险提示、设计依据等可写在此处"
          },
          "tags_suggested": {
            "type": "array",
            "maxItems": 10,
            "items": { "type": "string", "maxLength": 50 },
            "description": "建议标签，必须从 existing_tags 中选取"
          },
          "ai_confidence": {
            "type": "number",
            "minimum": 0.0,
            "maximum": 1.0,
            "description": "LLM 自评置信度"
          },
          "notes": {
            "type": "string",
            "maxLength": 500,
            "description": "AI 给人工的备注（如建议新标签、依赖信息），不进用例正文"
          }
        }
      }
    },
    "summary": {
      "type": "object",
      "additionalProperties": false,
      "properties": {
        "coverage_areas": {
          "type": "array",
          "items": { "type": "string", "maxLength": 100 },
          "description": "本批用例覆盖的功能领域，便于审阅"
        },
        "uncovered_risks": {
          "type": "array",
          "items": { "type": "string", "maxLength": 200 },
          "description": "AI 识别但未生成用例的风险点（如文档不清晰、超出范围）"
        }
      }
    }
  }
}
```

### 5.2 字段到 testcases 表的映射

| Schema 字段 | testcases 表字段 | 转换规则 |
|---|---|---|
| `seq_no` | — | 仅产物预览池使用，不入 testcases |
| `title` | `title` | 直接写入 |
| `level` | `level` | 直接写入；不在枚举内回退到 `default_level` |
| `precondition` | `precondition` | 直接写入 |
| `steps[]` | `steps` (TEXT) | **JSON 序列化**为字符串存储：`json.Marshal(steps)`。前端用例编辑器按 JSON 数组解析；与人工创建的用例 `steps` 兼容（人工可能存 Markdown，前端组件已能识别两种格式） |
| `postcondition` | `postcondition` | 直接写入 |
| `remark` | `remark` | 直接写入 |
| `tags_suggested[]` | `tags` (legacy) + `test_case_tags` 关联表 | Go 侧采纳时校验：仅写入与项目 `tags.name` 匹配的标签；未匹配项追加到 `remark` 末尾"【AI 建议新标签】" |
| `ai_confidence` | — | 仅存于 `requirement_gen_results.ai_confidence`，不入 testcases |
| `notes` | — | 仅存于 `requirement_gen_results.raw_json`，不入 testcases |
| `summary.coverage_areas` | — | 任务详情页展示，不入 testcases |
| `summary.uncovered_risks` | — | 任务详情页"风险提示"区块展示 |

### 5.3 Schema 校验失败的处理

Executor 校验失败时**立即 FAILED**（业务层错误，不重试）：

| 失败类型 | `fail_reason` 前缀 | 示例 |
|---|---|---|
| 非合法 JSON | `[SCHEMA] invalid_json` | LLM 输出含前后说明文字、未闭合的引号等 |
| 缺少必填字段 | `[SCHEMA] missing_field` | `[SCHEMA] missing_field: cases[2].steps` |
| 字段类型错误 | `[SCHEMA] type_mismatch` | `[SCHEMA] type_mismatch: cases[0].level expected string got int` |
| 枚举越界 | `[SCHEMA] enum_violation` | `[SCHEMA] enum_violation: cases[1].level=P4` |
| 长度越界 | `[SCHEMA] length_exceeded` | `[SCHEMA] length_exceeded: cases[5].title 158 > 120` |
| seq_no 不连续 / 重复 | `[SCHEMA] seq_no_violation` | `[SCHEMA] seq_no_violation: duplicate 3` |
| 超过 max_cases | `[SCHEMA] cases_overflow` | `[SCHEMA] cases_overflow: 52 > 50` |

## 6. Prompt 通用结构

所有内置 Skill 与自定义 Skill 应遵循以下七段式结构：

```
【1. 角色定义】
说明 AI 在本任务中的角色与能力边界

【2. 任务目标】
清晰指出要做什么（生成用例）+ 数量上限

【3. 输出要求】
- 必须输出 JSON，符合 standard_case_v1
- 不允许 Markdown 代码块包裹
- seq_no 从 1 起连续递增

【4. 业务上下文】
{{project_context}}

【5. 约束】
- 标签必须来自：{{existing_tags}}
- 默认级别：{{default_level}}
- 用户补充：{{extra_prompt}}

【6. 事实源】
{{requirement_text}}

【7. 参考样例】（可选）
{{few_shot_examples}}
```

七段中前三段为静态指令，后四段使用占位符注入。**自定义 Skill 至少要包含第 1、2、3、6 段**，否则 LLM 输出质量难以保证。

## 7. 内置 Skill 全文

### 7.1 `functional_testcase` — 通用功能测试用例

```
你是一名资深测试工程师。请基于下方"需求文本"生成测试用例草稿，供人工审阅采纳。

【任务目标】
- 从需求中拆解出可独立验证的测试场景
- 覆盖三类场景：正向流程、异常分支、边界值
- 总数不超过 {{max_cases}} 条
- 用例默认级别 {{default_level}}；P0 仅用于会阻塞主流程的关键路径

【输出要求】
- 严格输出符合 standard_case_v1 的 JSON 对象（含 cases 数组与可选 summary 对象）
- 禁止任何 JSON 之外的文字（不要 Markdown 代码块包裹，不要自然语言说明）
- cases[].seq_no 从 1 起连续递增
- 每条用例的 steps 数组中，action 和 expected 必须成对出现
- title 简明，不超过 60 字；避免"测试登录"这种笼统标题，改为"使用合法账号密码登录后成功跳转工作台"

【业务上下文】
{{project_context}}

【约束】
- tags_suggested 仅能从以下标签中选取（不得新增）：{{existing_tags}}
- 若识别到值得新建的标签，写到该用例的 notes 字段，不要放到 tags_suggested
- 用户额外指引：{{extra_prompt}}

【生成原则】
1. 需求文本是唯一事实源。文档未明确描述的业务规则不允许凭空补全
2. 异常分支应覆盖：必填缺失、类型错误、长度超限、权限不足、并发冲突等
3. 边界值优先关注：数字范围两端、字符串长度上下限、时间临界点、空集合
4. 同一功能的多条用例之间，标题应能清晰区分被测变量
5. 若需求文本本身不足以支撑生成（信息过少、矛盾），在 summary.uncovered_risks 中说明

【需求文本】
{{requirement_text}}

{{few_shot_examples}}
```

### 7.2 `api_testcase` — 接口测试用例

```
你是一名 API 测试专家。请基于下方"接口描述"生成接口测试用例草稿，供人工审阅采纳。

【任务目标】
- 针对每个接口，覆盖：合法请求、必填缺失、字段类型错误、长度边界、未鉴权、越权访问、并发场景
- 总数不超过 {{max_cases}} 条
- 用例默认级别 {{default_level}}

【输出要求】
- 严格输出符合 standard_case_v1 的 JSON 对象
- 禁止任何 JSON 之外的文字
- cases[].seq_no 从 1 起连续递增
- 每条 step 的 action 字段必须包含以下结构化信息（用换行分隔）：
    Method: {GET/POST/PUT/DELETE/...}
    URL: {path 含路径参数}
    Headers: {关键 header，如 Authorization}
    Body: {请求体 JSON 或表单字段}
- 每条 step 的 expected 字段必须包含：
    Status: {期望 HTTP 状态码}
    Body: {关键字段断言，可用 JSONPath 或自然语言}
    Side-effect: {对系统状态的副作用，如"DB users 表新增一行"}

【业务上下文】
{{project_context}}

【约束】
- tags_suggested 仅能从以下标签中选取：{{existing_tags}}
- 用户额外指引：{{extra_prompt}}

【生成原则】
1. 接口描述是唯一事实源。未声明的 header / 字段 / 错误码不允许凭空补全
2. 必填字段每个都要单独有一条"缺失该字段"的异常用例
3. 状态码覆盖至少 200 / 4xx / 401 / 403 / 500（若文档支持）
4. 鉴权场景必测：未带 token、过期 token、其他用户 token
5. 若接口描述不足（缺少错误码定义、字段类型），在 summary.uncovered_risks 中说明

【接口描述】
{{requirement_text}}

{{few_shot_examples}}
```

## 8. LLM 输出示例（功能用例）

输入需求文本（节选）：

> 系统登录页要求用户输入邮箱和密码。邮箱必填，必须为合法格式；密码必填，长度 8–32 位。
> 连续 5 次错误后账号锁定 30 分钟。已锁定账号显示"账号已锁定，请稍后重试"。

**合规输出**（节选）：

```json
{
  "cases": [
    {
      "seq_no": 1,
      "title": "使用合法邮箱与密码登录后成功进入工作台",
      "level": "P0",
      "precondition": "已存在一个状态为正常的用户账号",
      "steps": [
        {
          "action": "在登录页输入合法邮箱 user@example.com 和密码 Passw0rd!",
          "expected": "输入框无校验报错"
        },
        {
          "action": "点击登录按钮",
          "expected": "页面跳转到工作台，URL 为 /workbench"
        }
      ],
      "postcondition": "用户处于已登录状态",
      "remark": "",
      "tags_suggested": ["登录"],
      "ai_confidence": 0.95
    },
    {
      "seq_no": 2,
      "title": "邮箱格式非法时登录被拒绝",
      "level": "P1",
      "precondition": "无",
      "steps": [
        {
          "action": "在邮箱输入框输入 invalid-email（缺少 @）",
          "expected": "输入框失焦后显示"邮箱格式不正确""
        },
        {
          "action": "点击登录按钮",
          "expected": "登录按钮置灰或点击无效，不发起请求"
        }
      ],
      "postcondition": "用户停留在登录页",
      "remark": "",
      "tags_suggested": ["登录","边界值"],
      "ai_confidence": 0.9
    },
    {
      "seq_no": 5,
      "title": "连续 5 次密码错误后账号被锁定 30 分钟",
      "level": "P0",
      "precondition": "存在一个正常状态账号 alice@example.com",
      "steps": [
        {
          "action": "输入正确邮箱、错误密码，点击登录，重复 5 次",
          "expected": "前 4 次提示"密码错误"，第 5 次提示"账号已锁定，请稍后重试""
```

## 9. 常见 LLM 失败模式与对策

| 失败模式 | 表现 | 对策 |
|---|---|---|
| **代码块包裹** | LLM 输出 \`\`\`json ... \`\`\` | Executor 在 schema 校验前做一次容错剥离：若文本以 \`\`\` 开头则剥除前后代码块标记。**这是宽容处理，不算 SCHEMA 违规** |
| **额外解释文字** | "好的，以下是用例：{...}" | 同上容错：尝试从首个 `{` 到末尾匹配的 `}` 截取 JSON。仍解析失败则报 `[SCHEMA] invalid_json` |
| **超量生成** | cases 数 > max_cases | Schema 校验阶段直接报 `[SCHEMA] cases_overflow` 失败，不做截断（避免遗漏关键用例） |
| **seq_no 缺失或跳号** | 缺 seq_no 字段或 1,2,4,5 | 报 `[SCHEMA] seq_no_violation` |
| **标签虚构** | tags_suggested 含项目库没有的标签 | Schema 不报错；Go 侧采纳时过滤未匹配标签 + 在用例 remark 追加"【AI 建议新标签：X、Y】" |
| **级别枚举错误** | level=P4 / 高 / Medium | 报 `[SCHEMA] enum_violation`，**不做静默转换** |
| **凭空补全业务** | 编造需求未提及的字段 | Schema 层无法检测，依赖 uncovered_risks 引导 + 人工评审兜底 |
| **重复内容** | 同一场景生成 3 条仅措辞不同的用例 | Schema 层无法检测；v1.1 可在 Executor 端加 title 相似度去重 |
| **中英文混杂** | 标题中英文夹杂 | Schema 层不报错；prompt 中已要求中文为主，依赖模型遵循度 |

## 10. 自定义 Skill 编写规范

Admin 在系统管理 → Skill 管理中创建/编辑自定义 Skill 时，遵循以下规则：

### 10.1 强制要求

- prompt_template 必须包含 `{{requirement_text}}` 占位符（否则 LLM 拿不到事实源）
- prompt_template 必须明确要求"输出 JSON 符合 standard_case_v1"
- prompt_template 必须明确"禁止 JSON 之外的文字"
- output_schema 必须从预设列表中选取（v1.0 仅有 `standard_case_v1`）
- 不允许使用本规范第 4.1 节未列出的占位符

### 10.2 强烈建议

- 包含七段式结构中的至少 1/2/3/6 段
- 在第 5 段引用 `{{existing_tags}}`，明确约束 AI 标签选取
- 在第 5 段引用 `{{default_level}}`，明确级别基线
- 在角色定义中明确"测试工程师/API 专家/性能专家"等具体角色
- 在生成原则中明确事实源的优先级

### 10.3 反模式（应避免）

- ❌ "请尽可能多地生成用例" — 应给具体上限
- ❌ "可以参考其他类似项目的做法" — 引导 LLM 凭空补全
- ❌ "如果信息不够请补全" — 应改为"在 summary.uncovered_risks 中说明"
- ❌ 在 prompt 中插入项目敏感信息（API Key、内部 URL、密码） — Skill 模板在数据库中明文存储
- ❌ 一次性要求 LLM 既生成用例又生成自动化脚本 — 拆分到测试智编模块

### 10.4 校验时机

- Skill 创建/编辑时，Go 侧 Service 层做**静态校验**：
  - 占位符白名单校验：所有 `{{...}}` 必须在 4.1 表内
  - 必填占位符校验：必须含 `{{requirement_text}}`
  - 关键词校验：必须包含 "JSON"、"standard_case_v1"、"max_cases" 中各至少一次
  - 长度校验：prompt_template ≤ 8000 字符
- 校验失败返回错误码 130205 "Skill 提示词不符合规范"

## 11. 提示词版本管理

v1.0 不引入 Skill 模板的版本表。修改 Skill 模板会**直接覆盖**当前内容，已经发起的任务**不受影响**（任务执行时使用的是发起时刻的 prompt 快照？—— 见下文）。

**当前 v1.0 策略**：Executor 在任务执行时**重新从 DB 读取 Skill 当前 prompt_template**。

**潜在问题**：管理员修改 Skill 期间，若有任务正在 PENDING → RUNNING 转换中，可能用到新模板。

**v1.0 规避方案**：

1. Skill 编辑时使用 `lock_version` 乐观锁（已在 SQL 草案中体现），避免两位 admin 同时改
2. 编辑前提示 admin："本项目当前有 N 个 RUNNING 任务，编辑后的模板仅影响此后新建任务"
3. 实际不再回填编辑前的旧任务，因为新模板可能更优

**v1.1 演进**：引入 `ai_skill_versions` 表，任务表记录 `skill_version_id` 快照，确保严格回放。

## 12. 联调测试要点

提示词模板规范的联调验收项：

- [ ] 8 个占位符全部在 prompt 中渲染正确，无 `{{}}` 残留
- [ ] 缺失必填占位符（`requirement_text` / `max_cases`）时 Executor 返回明确错误
- [ ] 未知占位符（如 `{{foo}}`）不被替换，且校验时被 Go 拒绝
- [ ] standard_case_v1 全部失败类型都能被 schema_validator 检出并返回正确 fail_reason 前缀
- [ ] LLM 输出包含代码块包裹时，Executor 能正确剥离后解析
- [ ] LLM 输出含前后说明文字时，Executor 能截取核心 JSON 解析
- [ ] 内置 functional_testcase 在标准需求文本下输出至少 10 条结构合法用例
- [ ] 内置 api_testcase 在标准接口文档下输出至少 10 条结构合法用例
- [ ] 标签建议中含虚构标签时，落库逻辑正确过滤并写入 remark
- [ ] level 越界（P4）时整批任务 FAILED，不做静默纠正
