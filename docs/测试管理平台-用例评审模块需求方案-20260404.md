## 0. 当前结论

截至 2026-04-06，用例评审模块已经不是“单条用例状态按钮增强版”，而是独立的评审计划模块，当前系统基线如下：

1. **评审统一通过 `case-review` 模块执行**，测试用例模块不再承担 legacy 直接提审、直接通过、直接驳回链路。
2. **评审计划驱动用例进入评审流程**，计划下包含评审项、评审人分配、评审记录、统计结果。
3. **评审结果会自动回写测试用例主表**，`test_cases.status` 与 `test_cases.review_result` 不再由 testcase 接口直接维护。
4. **用例列表页保留评审摘要与入口**，包括加入评审、查看当前评审、查看评审摘要，但不再直接提交评审结论。

## 1. 模块边界

### 1.1 用例评审模块负责

- 创建、查看、编辑、复制、关闭、删除评审计划
- 为评审计划关联/移除用例
- 为评审项分配评审人、批量改评审人、批量重新提审
- 单条评审、批量评审、评审记录留痕
- 计划级统计与计划状态自动重算
- 将评审结果回写到 `test_cases`

### 1.2 测试用例模块负责

- 测试用例的创建、编辑、删除、废弃、恢复
- 在列表和详情中展示评审摘要投影
- 提供“加入评审”“查看当前评审”等跳转入口
- 基于回写结果展示 `review_result`

### 1.3 已移除的旧链路

以下 testcase 维度 legacy 直评接口已移除，不再作为业务入口：

- `POST /projects/:projectID/testcase/:testcaseID/submit-review`
- `POST /projects/:projectID/testcase/:testcaseID/approve-review`
- `POST /projects/:projectID/testcase/:testcaseID/reject-review`

## 2. 核心对象

### 2.1 评审计划 `case_reviews`

评审计划是顶层业务实体，当前已实现的核心字段包括：

- `name`
- `module_id`
- `review_mode`
- `status`
- `default_reviewer_ids`
- `planned_start_at`
- `planned_end_at`
- `description`
- `case_total_count / pending_count / approved_count / rejected_count / needs_update_count / pass_rate`

### 2.2 评审项 `case_review_items`

每条被纳入计划的用例都会生成独立评审项，保存：

- 所属计划、所属项目、目标用例 ID
- 用例版本快照、标题快照、模块快照
- 评审项状态 `pending / reviewing / completed`
- 最终结果 `pending / approved / rejected / needs_update`
- 当前轮次 `current_round_no`
- 最新意见、最新评审时间

### 2.3 评审人分配 `case_review_item_reviewers`

评审人分配按“评审项”维度保存，记录：

- `review_item_id`
- `reviewer_id`
- 当前状态 `pending / reviewed`
- 最近一次提交结果
- 最近一次提交意见
- 最近一次提交时间

### 2.4 评审记录 `case_review_records`

评审记录为追加式留痕，当前已记录：

- 计划 ID、评审项 ID、用例 ID
- 评审人 ID、评审人姓名
- 轮次 `round_no`
- 评审结果 `approved / rejected / needs_update`
- 评审意见
- 提交后聚合结果 `aggregate_result_after_submit`
- 创建时间

### 2.5 用例主表评审投影

当前用例列表/详情依赖以下投影字段：

- `review_result`
- `in_review`
- `current_review_id`
- `current_review_name`
- `related_review_count`

## 5. 真实业务规则

### 5.1 枚举口径

#### 5.1.1 计划模式

- `single`
- `parallel`

#### 5.1.2 计划状态

- `not_started`
- `in_progress`
- `completed`
- `closed`

#### 5.1.3 评审项状态

- `pending`
- `reviewing`
- `completed`

#### 5.1.4 评审结果

- `pending`
- `approved`
- `rejected`
- `needs_update`

### 5.2 创建与更新计划

#### 5.2.1 创建计划

当前创建计划规则：

- `name` 必填
- `review_mode` 必填，取值为 `single / parallel`
- `default_reviewer_ids` 必填，且至少 1 人
- 可选传入 `testcase_ids`
- 可选传入 `auto_submit`

当创建时同时带入 `testcase_ids` 且 `auto_submit=true` 时，关联的测试用例主表会被回写为：

- `test_cases.status = pending`
- `test_cases.review_result = 待评审`

#### 5.2.2 更新计划

当前后端已支持更新：

- `name`
- `module_id`
- `review_mode`
- `default_reviewer_ids`
- `planned_start_at`
- `planned_end_at`
- `description`

约束：

- `closed` 计划不可编辑
- `completed` 计划不可修改 `review_mode`

### 5.3 删除、关闭、复制计划

#### 5.3.1 删除计划

当前删除逻辑不是“只有未开始才能删除”，真实规则为：

- `not_started` 或 `closed`：允许删除
- `in_progress / completed`：
  - 若已有评审记录，不允许删除
  - 若尚无评审记录，允许删除

#### 5.3.2 关闭计划

- `close` 为手动关闭动作
- 关闭后计划状态变为 `closed`
- `closed` 计划不可再修改、不可再关联或批处理评审项

#### 5.3.3 复制计划

当前复制计划支持：

- `include_cases`
- `reset_reviewers`

真实行为：

- 会复制计划级 `default_reviewer_ids`
- 若 `include_cases=true`，会复制评审项，但新计划中的评审项状态统一重置为初始状态
- 若 `reset_reviewers=false`，会复制评审人分配
- 不复制历史评审记录

### 5.4 关联与移除用例

#### 5.4.1 关联用例

关联用例规则如下：

- `closed / completed` 计划不允许再关联用例
- 可以为每条用例单独传 `reviewer_ids`
- 若某条用例未传 `reviewer_ids`，则回退使用计划的 `default_reviewer_ids`
- 若计划默认评审人和条目评审人都为空，则直接报错：
  - 错误码：`120107`
  - 错误名：`CodeReviewEmptyReviewer`
- 同一条测试用例如果已经存在于其他 `not_started / in_progress` 计划中，不允许重复关联
- 关联成功后会创建：
  - `case_review_items`
  - `case_review_item_reviewers`
- `auto_submit=true` 时同步把 testcase 主表回写为待评审

#### 5.4.2 移除用例

移除规则如下：

- 仅允许在计划未关闭、未完成时执行
- 执行前必须校验 `item_id` 全部属于当前 `review_id + project_id`
- 已存在评审记录的评审项不允许移除
- 删除评审项后需要同步删除其评审人分配
- 删除完成后自动重算计划统计

### 5.5 批量改评审人与重新提审

#### 5.5.1 批量改评审人

真实逻辑：

- 仅允许在计划未关闭、未完成时执行
- 执行前先做评审项归属校验
- 先删除旧评审人，再创建新评审人分配
- 若该评审项此前已经产生最终结果，则：
  - 评审项重置为 `pending`
  - `final_result` 重置为 `pending`
  - `current_round_no + 1`
  - reviewer 状态重置
  - testcase 主表回写为：
    - `status = pending`
    - `review_result = 待评审`
- 最后自动重算计划统计

#### 5.5.2 批量重新提审

真实逻辑：

- 仅允许在计划未关闭、未完成时执行
- 执行前先做评审项归属校验
- 每个评审项都会被重置为：
  - `review_status = pending`
  - `final_result = pending`
  - `current_round_no + 1`
- 当前轮评审人状态会被重置
- testcase 主表回写为：
  - `status = pending`
  - `review_result = 重新提审`
- 最后自动重算计划统计

### 5.6 单条评审与聚合规则

#### 5.6.1 单人评审 `single`

当前真实规则为：

- 单人评审采用“最后一次有效提交为准”
- 当前实现通过 `ListReviewersByItemID` 的排序保证确定性：
  - `ORDER BY reviewed_at DESC, id DESC`
- 聚合时取最近一条已提交评审人的 `latest_result`

#### 5.6.2 多人评审 `parallel`

当前真实规则为：

- 只要还有评审人未提交，评审项最终结果保持 `pending`，评审项状态保持 `reviewing`
- 所有人都提交后再汇总最终结果：
  - 有任一 `rejected` => 最终 `rejected`
  - 否则若全部 `approved` => 最终 `approved`
  - 否则若存在 `needs_update` => 最终 `needs_update`

### 5.7 测试用例主表回写规则

评审结果会统一回写 `test_cases.status` 与 `test_cases.review_result`：

| 聚合结果 | `test_cases.status` | `test_cases.review_result` |
|---|---|---|
| `approved` | `active` | `已通过` |
| `rejected` | `draft` | `已驳回` |
| `needs_update` | `draft` | `建议修改` |
| `pending` | `pending` | `待评审` |
| 批量重新提审 | `pending` | `重新提审` |

### 5.8 计划状态自动重算

当前计划统计与状态重算规则为：

- `total = 0` 时：`not_started`
- `pending_count = total` 时：`not_started`
- `approved + rejected + needs_update = total` 时：`completed`
- 其他情况：`in_progress`

同时会重算：

- `case_total_count`
- `pending_count`
- `approved_count`
- `rejected_count`
- `needs_update_count`
- `pass_rate`

## 6. 前端当前实现现状

### 6.1 已实现

- 评审列表、创建页、详情页三条完整路由
- 用例列表单条/批量加入评审
- 通过路由 query 从用例列表跳入评审模块：
  - `?create=1&testcaseIds=...`
  - `?reviewId=...`
- 列表页详情抽屉可继续关联用例
- 详情页支持查看评审记录、批量改评审人、批量重新提审

### 6.2 当前前端仍需注意的实现差异

以下差异是“后端已支持 / 前端未完全暴露”的真实状态：

1. `UpdateReviewPayload` 当前前端类型中尚未包含 `default_reviewer_ids`，且前端暂无完整的计划编辑 UI 接入。
2. 列表页详情抽屉中的快捷单条评审当前只暴露 `approved / rejected`。
3. 详情页批量评审当前前端只暴露 `approved / rejected`，后端接口本身已支持 `needs_update`。

## 7. 与用例列表的协作口径

用例列表页当前不再承担评审过程本身，只保留评审摘要与入口：

- `review_result`：展示当前评审结果投影
- `in_review`：标识当前是否处于活跃评审中
- `current_review_id / current_review_name`：定位当前活跃评审计划
- `related_review_count`：展示历史关联过的评审计划数量

“当前活跃评审”选择逻辑为：

1. 只在 `case_review_items.review_status in ('pending','reviewing')` 且计划状态为 `not_started / in_progress` 的集合中选择
2. 优先选择 `in_progress` 的计划
3. 再按 `cri.updated_at / cr.updated_at / cr.id` 倒序确定当前展示计划
