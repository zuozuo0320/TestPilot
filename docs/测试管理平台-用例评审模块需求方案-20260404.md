> 适用对象：产品、研发、测试

## 0. 本轮明确结论

本轮需求讨论先明确两条硬结论，后续设计和开发都以此为准：

1. Aisight 的“用例评审模块”整体方向，完全参考 MeterSphere 的用例评审模块，不再做“单条状态按钮增强版”。
2. 当某条用例在评审计划中被最终评审通过后，必须自动同步更新测试用例列表中的评审状态；该状态不是手工维护，而是评审结果的回写结果。

换句话说：

- 评审模块是独立模块；
- 评审通过会影响用例主表状态；
- 用例列表中的评审状态展示，必须和评审模块的最终结果保持一致。

## 2. MeterSphere 用例评审模块调研结论

## 2.1 模块定位

MeterSphere 的“用例评审”不是单条用例上的一个附属动作，而是一个独立业务模块。  
它的核心对象不是 `test_case.review_result`，而是“评审计划”。

其核心流程可以概括为：

1. 创建评审计划；
2. 关联一批测试用例；
3. 指定评审模式与评审人；
4. 评审人逐条评审；
5. 形成计划级进度、通过率和历史记录；
6. 将最终评审结果回写到用例层。

这意味着 MeterSphere 的“用例评审”本质是：

> 通过“评审计划”驱动一批测试用例进入评审流程，并把最终评审结果沉淀和回写。

## 2.2 核心对象

### 2.2.1 评审计划

评审计划是顶层业务对象，至少包含：

- 评审名称
- 评审模式
- 评审人
- 评审周期
- 关联用例集合
- 计划级状态
- 计划级通过率
- 创建人
- 所属模块/目录

### 2.2.2 评审计划中的关联用例

每条关联进评审计划的用例，都有独立评审上下文，至少包含：

- 用例基本信息
- 当前评审状态
- 当前评审人
- 最终评审结果
- 评审意见
- 是否重新提审

### 2.2.3 评审记录

评审是过程化操作，不是一次性覆盖。  
MeterSphere 会保留“谁在什么时候给了什么评审结果、填了什么意见”的记录，并允许后续再次提审。

## 2.3 页面结构

### 2.3.1 评审计划列表页

结合官方文档和用户截图，列表页具备以下能力：

- 视图切换：`全部 / 我评审的 / 我创建的`
- 搜索：按 `ID / 名称 / 标签` 或 `ID / 用例名称 / 所属模块` 搜索
- 条件筛选：状态、评审模式、评审人、模块等
- 列表字段至少包括：
  - ID
  - 评审名称
  - 用例数量
  - 评审状态
  - 通过率
  - 评审模式
  - 评审人
  - 创建人
  - 所属模块

### 2.3.2 评审详情页

评审详情页至少支持：

- 查看计划基本信息
- 查看关联用例
- 编辑评审计划
- 复制评审计划
- 删除评审计划
- 关联新用例
- 批量评审
- 批量重新提审
- 查看计划级统计

### 2.3.3 单条用例评审页/抽屉

单条用例评审至少支持：

- 查看用例详情
- 选择评审结果并提交
- 填写评审意见
- 切换下一条待评审用例
- 按评审状态筛选当前计划中的用例

## 2.4 关键业务规则

### 2.4.1 评审模式

MeterSphere 的规则很明确：

- `单人评审`：最后一次评审结果为最终评审结果
- `多人评审`：所有评审人都通过才算通过；任意评审人不通过则不通过

### 2.4.2 评审人

- 只有评审人才可执行评审操作
- 计划可配置默认评审人
- 关联用例时可以继承计划评审人，也可以按用例单独指定评审人

### 2.4.3 评审周期

- 评审周期是计划管理字段
- 超过计划时间仍允许继续评审，不是硬性阻断

### 2.4.4 通过率

- 通过率 = 已通过用例数 / 关联用例总数 × 100%

### 2.4.5 复制与重新提审

MeterSphere 支持：

- 复制评审计划，并重置新计划中用例的评审状态
- 对评审中的用例执行重新提审

这说明“评审”是可重复发起的业务活动，而不是静态一次性字段。

## 3. 当前 Aisight 现状判断

## 3.1 已有能力

当前系统不是完全没有评审，而是只有“单条用例状态流转能力”。

### 3.1.1 后端已有能力

当前后端已经具备：

- 用例生命周期字段：`draft / pending / active / discarded`
- 用例版本号字段：`V1 / V2 / ...`
- 单条用例提审：`submit-review`（已移除）
- 单条用例评审通过：`approve-review`（已移除）
- 单条用例评审驳回：`reject-review`（已移除）
- 已生效用例发生实质变更后：
  - 自动版本 +1
  - 状态重置为草稿
  - 评审结果重置
  - 执行结果重置

### 3.1.2 前端已有能力

当前前端 `TestCasePage.vue` 已经具备：

- 单条用例提交评审
- 单条用例评审通过
- 单条用例评审驳回
- 用例列表按评审结果筛选
- 用例行中展示评审结果

## 3.2 当前缺失的核心能力

### 3.2.1 缺少“评审计划”实体

当前系统没有：

- 评审计划表
- 评审计划-用例关联表
- 评审记录表
- 计划级统计字段

### 3.2.2 缺少独立评审模块页面

前端虽然已有“用例评审”导航入口，但当前仍是占位页，不是正式模块页面。

### 3.2.3 缺少计划级统计和批量能力

当前没有：

- 计划列表
- 我评审的 / 我创建的
- 计划级通过率
- 批量评审
- 批量改评审人
- 批量重新提审

### 3.2.4 缺少正式评审记录留痕

当前没有独立的评审意见与评审记录模型，因此无法完整回答：

- 谁评审了哪条用例
- 给出的评审意见是什么
- 是否经过多轮评审
- 不同评审计划之间如何追踪历史结果

## 3.3 结论

当前系统已经有“单条用例评审状态机”，但没有“MeterSphere 风格的用例评审模块”。  
如果要完全对齐 MeterSphere，下一步必须补的是：

> 评审计划 + 计划内关联用例 + 评审记录 + 最终结果回写

而不是继续单独增强 `test_cases.review_result` 字段。

## 4. 产品定位（本轮定稿）

Aisight 的“用例评审模块”定义为：

> 面向项目内测试用例的计划型评审系统，用于按版本、模块、里程碑或业务范围组织一批用例进行集中评审，并沉淀评审计划、评审过程、评审意见、通过率和最终结果回写。

本模块与现有“测试用例管理”的关系如下：

- 测试用例模块负责“用例本体管理”
- 用例评审模块负责“评审活动管理”
- 评审模块的最终结果，需要自动回写用例主表，并体现在用例列表中

评审计划中的单条用例必须支持：

  - `通过`
  - `不通过`

- 填写评审意见
- 提交评审
- 自动跳转下一条待评审用例

### 5.1.6 评审结果自动回写用例列表状态（硬规则）

这是本轮确认后的硬规则。

#### 规则一：进入评审计划时

- 草稿用例加入评审计划并正式提审后：
  - `test_cases.status = pending`
  - `test_cases.review_result = 待评审`
- 用例列表页中该条用例的评审状态，必须立即显示为 `待评审`

#### 规则二：评审通过时

- 当某条用例在评审计划中被判定为“最终通过”后：
  - `test_cases.status = active`
  - `test_cases.review_result = 已通过`
- 用例列表页中该条用例的评审状态，必须自动更新为 `已通过`
- 不允许通过前端手工额外操作去同步，必须由后端评审结果回写触发

#### 规则三：评审不通过/建议修改时

- 当某条用例在评审计划中最终结果为“不通过”或“建议修改”后：
  - `test_cases.status = draft`
  - `test_cases.review_result = 已驳回` 或 `建议修改`（最终文案可后续统一）
- 用例列表页中该条用例的评审状态必须同步更新

#### 规则四：重新提审时

- 对已修改后的用例再次提审：
  - `test_cases.status = pending`
  - `test_cases.review_result = 重新提审`
- 进入新一轮评审后，列表页评审状态必须自动展示 `重新提审` 或 `待评审`

#### 规则五：已生效用例发生实质变更时

- 已通过评审并生效的用例，如果核心内容发生实质变更：
  - 自动版本 +1
  - 状态重置为草稿
  - 评审结果重置为 `重新提审`
- 后续只能重新进入评审计划后，才能再次变成 `已通过`

### 5.1.7 用例列表中的评审状态字段定义

为避免后续实现歧义，明确如下：

- 测试用例列表页中的“评审状态/评审结果”字段，数据来源是 `test_cases.review_result`
- `test_cases.review_result` 的最终值来源于评审模块回写
- 列表页不允许把评审状态作为可任意编辑字段
- 评审状态是评审模块的结果投影，不是列表页独立状态

## 6. 关键业务规则

## 6.1 评审模式规则

### 6.1.1 单人评审

- 一条关联用例可以配置多个评审人
- 任意一个有权限的评审人都可以给出结果
- 以最后一次有效评审结果，作为该用例在本次计划中的最终结果

### 6.1.2 多人评审

- 所有指定评审人都必须完成评审
- 所有人都通过，才算最终通过
- 任一评审人不通过，则该用例最终不通过
- 若存在“建议修改”且没有所有人通过，则该用例不算最终通过

## 6.2 计划状态规则

计划级状态定义为：

- `未开始`：计划已创建，但没有任何关联用例进入正式评审
- `进行中`：存在待评审或已部分评审用例
- `已完成`：所有关联用例均已有最终结果
- `已关闭`：计划被人工关闭，不再允许新增评审动作

## 6.3 关联规则

### 6.3.1 P0 规则

- 同一条用例，同一时刻仅允许存在一个“进行中”的评审计划关联
- 避免同一用例被多个评审计划同时驱动，造成主表状态冲突

### 6.3.2 P1 可扩展规则

- 后续如业务需要，可支持不同版本的同一用例进入不同评审计划
- 但前提是每个版本快照必须可追溯

## 6.4 重新提审规则

- 已评审但需要修改的用例，修改完成后可重新提审
- 重新提审后：
  - 用例级评审状态回到 `待评审`
  - 原评审记录保留
  - 新一轮评审记录追加，不覆盖旧记录

## 6.5 通过率规则

- 通过率 = 当前计划中“最终通过”的关联用例数 / 当前计划关联用例总数 × 100%
- “建议修改”和“待评审”均不计入通过

## 11. 本轮已确认项

本轮确认，不再作为待讨论项：

1. 完全参考 MeterSphere 的用例评审模块模型；
2. 用例评审必须以“评审计划”为核心对象；
3. 用例评审通过后，必须自动同步更新用例列表的评审状态；
4. 用例列表评审状态字段必须来源于评审模块回写；
5. 评审模块应作为独立页面/模块实现，而不是继续依附在单条用例操作上。

## 13. 本轮结论

本轮需求已经从“讨论稿”推进到“需求基线”：

- 产品方向：完全参考 MeterSphere
- 业务模型：评审计划驱动
- 状态联动：评审结果自动回写用例主表
- 列表展示：用例列表评审状态与评审模块结果保持一致

3. 前端 Composable 重构（将 `CaseReviewPage.vue` 中的冗余逻辑提取为 `useCaseReviewList` 等）。
