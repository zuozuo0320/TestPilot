# 测试管理平台-用例评审模块 MySQL 表结构 SQL 草案

> 版本：v0.1（创建于 2026-04-04）
> 关联文档：`测试管理平台-用例评审模块需求方案-20260404.md`
> 文档定位：用于后端建表、Repository 开发、事务设计与 DDL 评审

## 1. 说明

本文档基于已经确认的需求基线，输出“用例评审模块”首期可落地的 MySQL DDL 草案。

本次设计遵循以下前提：

- 数据库：`MySQL 8.0+`
- 字符集：`utf8mb4`
- 存储引擎：`InnoDB`
- 时间字段：统一使用 `datetime(3)`
- 主键：统一使用 `BIGINT UNSIGNED`
- 跨域对象：优先走逻辑关联，不强制增加物理外键

设计原则：

- 评审模块以“评审计划”为核心，不直接把评审过程揉进 `test_cases` 主表。
- `test_cases.status` 与 `test_cases.review_result` 仍然保留，但它们是评审结果投影，不是人工维护字段。
- 多人评审必须支持“按评审人逐个留痕”，因此不能只在计划或关联表上保存 reviewer ID 列表。
- 所有“最终结果回写”必须在同一事务中完成，避免评审表与用例主表状态不一致。

1. `test_case_reviews`

## 3. 状态值建议

为避免后续字符串漂移，建议在后端统一维护状态常量。

### 3.1 评审计划状态

- `not_started`：未开始
- `in_progress`：进行中
- `completed`：已完成
- `closed`：已关闭

### 3.2 评审模式

- `single`：单人评审
- `parallel`：多人评审

### 3.3 评审项状态

- `pending`：待评审
- `reviewing`：评审中
- `completed`：已出最终结果

### 3.4 评审项最终结果

- `pending`：待评审
- `approved`：通过
- `rejected`：不通过
- `needs_update`：建议修改

### 3.5 评审人处理状态

- `pending`：尚未评审
- `reviewed`：已提交评审

### 3.6 `test_cases.review_result` 建议统一文案

首期建议统一为以下值：

- `未评审`
- `待评审`
- `重新提审`
- `已通过`
- `已驳回`
- `建议修改`

说明：

- 当前代码里 `review_result` 使用中文值，首期不建议为了评审模块单独把历史数据改成英文枚举。
- 但必须在 `internal/model` 中补统一常量，避免服务层继续散落硬编码字符串。

## 4. 表结构设计

## 4.1 `test_case_reviews`

评审计划主表。

```sql
CREATE TABLE IF NOT EXISTS `test_case_reviews` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  `project_id` BIGINT UNSIGNED NOT NULL COMMENT '所属项目ID',
  `name` VARCHAR(128) NOT NULL COMMENT '评审计划名称',
  `module_id` BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '所属模块ID，0表示未指定',
  `review_mode` VARCHAR(16) NOT NULL COMMENT '评审模式：single/parallel',
  `status` VARCHAR(16) NOT NULL DEFAULT 'not_started' COMMENT '计划状态：not_started/in_progress/completed/closed',
  `description` VARCHAR(500) DEFAULT NULL COMMENT '描述/备注',
  `planned_start_at` DATETIME(3) DEFAULT NULL COMMENT '计划开始时间',
  `planned_end_at` DATETIME(3) DEFAULT NULL COMMENT '计划结束时间',
  `case_total_count` INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '关联用例总数',
  `pending_count` INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '待评审数量',
  `approved_count` INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '已通过数量',
  `rejected_count` INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '不通过数量',
  `needs_update_count` INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '建议修改数量',
  `pass_rate` DECIMAL(5,2) NOT NULL DEFAULT 0.00 COMMENT '通过率，单位百分比',
  `created_by` BIGINT UNSIGNED NOT NULL COMMENT '创建人',
  `updated_by` BIGINT UNSIGNED NOT NULL COMMENT '更新人',
  `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
  `updated_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_case_reviews_project_status` (`project_id`, `status`),
  KEY `idx_case_reviews_project_creator` (`project_id`, `created_by`),
  KEY `idx_case_reviews_project_mode` (`project_id`, `review_mode`),
  KEY `idx_case_reviews_project_module` (`project_id`, `module_id`),
  KEY `idx_case_reviews_project_created` (`project_id`, `created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用例评审-评审计划主表';
```

字段说明：

- `case_total_count ~ needs_update_count` 为列表页和详情页统计冗余字段。
- `pass_rate` 统一由后端事务内重算，不允许前端传入。
- `module_id` 用于计划归属模块筛选，不等同于所有关联用例都必须属于该模块。

## 4.2 `test_case_review_items`

评审计划与测试用例的关联表，也是“计划内单条用例评审上下文”主表。

```sql
CREATE TABLE IF NOT EXISTS `test_case_review_items` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  `review_id` BIGINT UNSIGNED NOT NULL COMMENT '评审计划ID',
  `project_id` BIGINT UNSIGNED NOT NULL COMMENT '所属项目ID，冗余便于检索',
  `testcase_id` BIGINT UNSIGNED NOT NULL COMMENT '测试用例ID',
  `testcase_version` VARCHAR(20) NOT NULL COMMENT '进入评审时的用例版本号快照',
  `module_id` BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '用例所属模块ID快照',
  `title_snapshot` VARCHAR(200) NOT NULL COMMENT '用例标题快照',
  `review_status` VARCHAR(16) NOT NULL DEFAULT 'pending' COMMENT '评审项状态：pending/reviewing/completed',
  `final_result` VARCHAR(16) NOT NULL DEFAULT 'pending' COMMENT '最终结果：pending/approved/rejected/needs_update',
  `current_round_no` INT UNSIGNED NOT NULL DEFAULT 1 COMMENT '当前评审轮次，从1开始',
  `reviewed_at` DATETIME(3) DEFAULT NULL COMMENT '最终结果形成时间',
  `latest_comment` VARCHAR(1000) DEFAULT NULL COMMENT '最近一次评审意见摘要',
  `sort_order` INT NOT NULL DEFAULT 0 COMMENT '列表排序值',
  `created_by` BIGINT UNSIGNED NOT NULL COMMENT '关联创建人',
  `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
  `updated_by` BIGINT UNSIGNED NOT NULL COMMENT '更新人',
  `updated_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_review_case` (`review_id`, `testcase_id`),
  KEY `idx_review_items_review` (`review_id`, `review_status`, `final_result`),
  KEY `idx_review_items_case` (`project_id`, `testcase_id`),
  KEY `idx_review_items_module` (`project_id`, `module_id`),
  KEY `idx_review_items_updated` (`review_id`, `updated_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用例评审-计划关联用例表';
```

字段说明：

- `testcase_version`、`module_id`、`title_snapshot` 是评审进入时的快照字段，用于回溯和列表稳定展示。
- `current_round_no` 用于重新提审时递增，历史轮次由 `test_case_review_records` 保留。
- `latest_comment` 仅用于列表展示，完整意见以记录表为准。

## 4.3 `test_case_review_item_reviewers`

评审项与评审人的当前分配表，用于“我评审的”查询、多人评审聚合和批量改评审人。

```sql
CREATE TABLE IF NOT EXISTS `test_case_review_item_reviewers` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  `review_id` BIGINT UNSIGNED NOT NULL COMMENT '评审计划ID',
  `review_item_id` BIGINT UNSIGNED NOT NULL COMMENT '评审项ID',
  `project_id` BIGINT UNSIGNED NOT NULL COMMENT '所属项目ID，冗余便于检索',
  `reviewer_id` BIGINT UNSIGNED NOT NULL COMMENT '评审人ID',
  `review_status` VARCHAR(16) NOT NULL DEFAULT 'pending' COMMENT '评审人处理状态：pending/reviewed',
  `latest_result` VARCHAR(16) DEFAULT NULL COMMENT '该评审人在当前轮次的最近结果',
  `latest_comment` VARCHAR(1000) DEFAULT NULL COMMENT '该评审人在当前轮次的最近意见',
  `reviewed_at` DATETIME(3) DEFAULT NULL COMMENT '该评审人在当前轮次最近一次提交时间',
  `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
  `updated_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_item_reviewer` (`review_item_id`, `reviewer_id`),
  KEY `idx_item_reviewers_reviewer` (`project_id`, `reviewer_id`, `review_status`),
  KEY `idx_item_reviewers_review` (`review_id`, `reviewer_id`),
  KEY `idx_item_reviewers_item` (`review_item_id`, `review_status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用例评审-评审项当前评审人表';
```

字段说明：

- 该表只维护“当前轮次当前分配状态”，不承担全量历史存档职责。
- 重新提审时，不新建新行，直接重置 `review_status/latest_result/latest_comment/reviewed_at` 即可。
- 如果业务后续需要保留“历史分配关系”，可在 P1 新增 reviewer assignment history 表。

## 4.4 `test_case_review_records`

评审记录表，保存每一次评审动作的不可变留痕。

```sql
CREATE TABLE IF NOT EXISTS `test_case_review_records` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
  `review_id` BIGINT UNSIGNED NOT NULL COMMENT '评审计划ID',
  `review_item_id` BIGINT UNSIGNED NOT NULL COMMENT '评审项ID',
  `project_id` BIGINT UNSIGNED NOT NULL COMMENT '所属项目ID，冗余便于检索',
  `testcase_id` BIGINT UNSIGNED NOT NULL COMMENT '测试用例ID',
  `reviewer_id` BIGINT UNSIGNED NOT NULL COMMENT '评审人ID',
  `round_no` INT UNSIGNED NOT NULL COMMENT '评审轮次',
  `result` VARCHAR(16) NOT NULL COMMENT '本次评审结果：approved/rejected/needs_update',
  `comment` VARCHAR(2000) DEFAULT NULL COMMENT '评审意见',
  `aggregate_result_after_submit` VARCHAR(16) NOT NULL COMMENT '本次提交后聚合出的最终结果快照',
  `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '提交时间',
  PRIMARY KEY (`id`),
  KEY `idx_review_records_item_round` (`review_item_id`, `round_no`, `created_at`),
  KEY `idx_review_records_reviewer` (`project_id`, `reviewer_id`, `created_at`),
  KEY `idx_review_records_case` (`project_id`, `testcase_id`, `created_at`),
  KEY `idx_review_records_review` (`review_id`, `created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用例评审-评审记录表';
```

字段说明：

- `aggregate_result_after_submit` 用于追溯“这一条评审意见提交后，系统把该用例聚合成什么状态”。
- 该表只新增不更新，作为审计与历史展示来源。

## 5. 与现有 `test_cases` 主表的协同

本次不新增新的“主表评审状态表”，仍然继续使用现有 `test_cases` 字段：

- `status`
- `version`
- `review_result`
- `exec_result`

但要调整现有规则：

### 5.1 正式提审进入评审计划

回写主表：

```text
status = pending
review_result = 待评审
```

### 5.2 评审最终通过

回写主表：

```text
status = active
review_result = 已通过
```

### 5.3 评审最终不通过

回写主表：

```text
status = draft
review_result = 已驳回
```

### 5.4 评审结果为建议修改

回写主表：

```text
status = draft
review_result = 建议修改
```

### 5.5 重新提审

回写主表：

```text
status = pending
review_result = 重新提审
```

### 5.6 已生效用例发生实质变更

当前 `internal/service/testcase_service.go` 中，已生效用例发生实质变更后会把 `review_result` 重置成“未评审”。

为了与本轮评审需求保持一致，建议改为：

```text
version = 原版本 + 1
status = draft
review_result = 重新提审
exec_result = 未执行
```

这样业务语义才是“变更后必须重新进入评审”，而不是“好像从未评审过”。

## 6. 核心事务设计

## 6.1 创建评审计划事务

事务内动作：

1. 写入 `test_case_reviews`
2. 批量写入 `test_case_review_items`
3. 批量写入 `test_case_review_item_reviewers`
4. 如果选择“创建即提审”，批量回写 `test_cases.status/review_result`
5. 重算计划统计字段

约束：

- 同一条用例，同一时刻只能存在一个 `not_started/in_progress` 的评审计划关联。
- 发现冲突时，整单回滚。

## 6.2 单条评审提交事务

事务内动作：

1. 锁定 `test_case_review_items` 当前行
2. 校验操作者是否在 `test_case_review_item_reviewers` 中
3. 写入 `test_case_review_records`
4. 更新当前 reviewer 在 `test_case_review_item_reviewers` 的最近结果
5. 重新聚合该 `review_item` 的 `review_status/final_result`
6. 如已形成最终结果，则回写 `test_cases`
7. 重算 `test_case_reviews` 统计字段

## 6.3 重新提审事务

事务内动作：

1. 锁定 `test_case_review_items`
2. `current_round_no = current_round_no + 1`
3. 重置 `review_status = pending`
4. 重置 `final_result = pending`
5. 清空 `reviewed_at/latest_comment`
6. 重置当前 reviewer 表状态
7. 回写 `test_cases.status = pending`
8. 回写 `test_cases.review_result = 重新提审`
9. 重算计划统计字段

## 7. 聚合规则建议

## 7.1 单人评审模式 `single`

规则：

- 允许配置多个评审人，但任意一个评审人都可以提交结果。
- 当前轮次最后一次有效提交，作为该评审项最终结果。

聚合方式：

1. 查询该评审项当前轮次最后一条评审记录
2. 将其 `result` 作为 `final_result`
3. 将 `review_status` 置为 `completed`

## 7.2 多人评审模式 `parallel`

规则：

- 所有指定评审人都必须完成评审。
- 任一评审人为 `rejected`，则最终结果为 `rejected`。
- 全部评审人都为 `approved`，则最终结果为 `approved`。
- 不存在 `rejected`，但存在 `needs_update`，且所有评审人都已提交，则最终结果为 `needs_update`。
- 只要还有任一评审人未提交，则结果保持 `pending` 或 `reviewing`，不回写主表最终状态。

建议实现：

```text
若 reviewer.pending_count > 0:
  item.review_status = reviewing 或 pending
  item.final_result = pending
若 reviewer.pending_count == 0 且存在 rejected:
  item.review_status = completed
  item.final_result = rejected
若 reviewer.pending_count == 0 且全部 approved:
  item.review_status = completed
  item.final_result = approved
若 reviewer.pending_count == 0 且无 rejected 且存在 needs_update:
  item.review_status = completed
  item.final_result = needs_update
```

## 8. 查询与索引建议

## 8.1 评审计划列表

主要查询维度：

- `project_id`
- `status`
- `review_mode`
- `created_by`
- `module_id`
- `created_at`

## 8.2 我评审的

典型查询路径：

1. 先用 `test_case_review_item_reviewers(project_id, reviewer_id, review_status)` 找到相关 item
2. 再回表取 `review_id`
3. 再查 `test_case_reviews`

因此 reviewer 表上的 `idx_item_reviewers_reviewer` 必须保留。

## 8.3 用例是否已在进行中评审

典型约束查询：

- 按 `project_id + testcase_id`
- 关联 `review_id` 对应计划状态为 `not_started/in_progress`

首期建议通过 `test_case_review_items` + `test_case_reviews` 联查完成，不额外增加主表冗余字段。

## 11. 本轮结论

本表结构草案明确了 3 件事：

1. 必须以评审计划为中心建模；
2. 必须支持“评审人级别”的当前状态与历史留痕；
3. 必须通过事务把评审最终结果自动回写到 `test_cases` 主表。

如果继续推进实现，下一步建议直接进入：

- GORM Model 草案
- Repository 接口定义
- Service 事务编排与聚合伪代码
