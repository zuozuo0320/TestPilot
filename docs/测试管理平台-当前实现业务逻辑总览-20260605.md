## 3. Go 后端当前实现

### 3.1 启动流程

`cmd/server/main.go` 当前负责：

1. 加载并校验配置。
2. 处理子命令：`cleanup-demo`、`seed-bootstrap`、`seed-demo`、`migrate`、`migrate-sql`、`seed`。
3. 连接 MySQL，执行 `AutoMigrate` 与增量 SQL 迁移。
4. 按配置执行基础种子数据与演示数据初始化。
5. 初始化 Redis；Redis 不可用时继续运行，相关队列能力降级。
6. 初始化 Asynq 需求智生任务队列与 worker。
7. 构建 Repository → Service → API 三层依赖。
8. 注册 Gin 路由并启动 HTTP 服务。
9. 收到退出信号后优雅关闭 HTTP server 与 worker/client。

### 3.2 后端分层

| 层 | 目录 | 职责 |
|---|---|---|
| API/Handler | `internal/api/` | Gin 路由、请求解析、鉴权中间件、统一响应、Executor 回调 |
| Service | `internal/service/` | 业务规则、事务编排、外部 Executor 调用、领域状态流转 |
| Repository | `internal/repository/` | GORM 数据访问、分页查询、事务内 CRUD |
| Model | `internal/model/` | 数据模型、枚举、AutoMigrate |
| Config | `internal/config/` | 环境变量和配置校验 |
| Migration | `internal/migration/` | AutoMigrate 无法覆盖的增量 SQL |
| Queue | `internal/queue/` | Asynq 客户端和 worker server |
| Store | `internal/store/` | MySQL、Redis 连接 |
| Seed | `internal/seed/` | 基础账号/角色/项目和演示数据初始化 |

## 6. 关键业务流

### 6.1 测试用例生命周期

1. 创建用例时默认 `draft`、版本 `V1`、默认优先级 `medium`、默认级别 `P1`、默认执行结果 `未执行`。
2. 草稿用例可编辑、删除、批量删除、移动、改级、打标。
3. 进入评审后用例状态可变为 `pending`。
4. `pending` 用例被查询或更新前会基于评审人最新结果计算有效评审状态。
5. 被评审通过后可转为 `active`；被驳回或需修改后回到 `draft` 并记录评审结果。
6. `active` 用例发生实质变更时保存历史快照，版本号递增，回到 `draft`，评审结果重置为未评审，执行结果重置为未执行。
7. `active` 用例可废弃为 `discarded`；废弃用例可恢复为 `draft`。

### 6.2 用例评审与 AI 门禁

1. 评审计划维护评审项、评审人、附件和评审记录。
2. 评审提交由 `CaseReviewSubmitService` 处理主评/影子评审人提交、批量评审、改派、重提等流程。
3. AI 门禁由 `CaseReviewRuleService` 同步执行 Layer 1 规则引擎。
4. 单项 rerun 或计划级 run-all 会更新 `case_review_items.ai_gate_status`。
5. 门禁失败时，critical/major finding 会生成 AI 来源 Action Item；minor 只返回提示，不生成缺陷。
6. rerun 会删除旧的 open AI 门禁缺陷，但保留 disputed/resolved 历史，避免重复生成已处理项。
7. Action Item 支持 open → resolved、open → disputed、非 open → reopen。
8. 重提前会检查 open+critical Action Item，存在时禁止重提。

### 6.3 测试智编

1. 前端创建 AI 脚本任务并维护关联用例。
2. 任务可走 AI 直生、录制增强、脚本编辑、确认/废弃、导出、克隆、删除等操作。
3. Executor 支持 browser-use 探索、Playwright codegen 录制、LLM 重构和 Playwright 验证。
4. 当前实现支持多项目 Playwright 工作区：`executor/pw_projects/projects/<project_key>/`。
5. 项目工作区自动补齐 `pages/`、`pages/shared/`、`tests/`、`fixtures/`、`registry/`、`auth_states/`。
6. `fixtures/base.fixture.ts` 根据 `registry/page-registry.json` 自动生成，不建议手工维护。

### 6.4 需求智生

1. 需求文档可通过上传或粘贴创建。
2. 上传文档支持 `docx/pdf/md/txt`，大小上限 10MB，创建后为 `not_parsed`。
3. 粘贴文本直接入库并标记 `parsed`，正文超过 50000 字符会截断并记录原始长度。
4. 文档解析由后端派发给 Executor，Executor 异步解析并通过 `/internal/requirement-docs/:docID/parse-callback` 回调。
5. 解析状态使用 CAS 控制：`not_parsed/parse_failed → parsing → parsed/parse_failed`。
6. 生成任务可经 Redis/Asynq 入队；Redis 不可用时降级为本地执行。
7. Executor 生成完成后通过任务 callback/heartbeat 回写结果和进度。
8. 生成结果可采纳为测试用例，也可丢弃。
9. AI Skill 用于影响生成策略，支持 CRUD、启停和智能路由。

### 6.5 标签管理

1. 每个项目最多 100 个标签。
2. 标签名去空格后必须 2-50 字符。
3. 颜色必须是 `#RRGGBB`，未传时默认 `#3B82F6`。
4. 同项目标签名唯一。
5. 每个用例最多绑定 10 个标签。
6. 删除标签会在事务中先解除所有用例关联，再删除标签。

### 6.6 AI 模型配置

1. 后端维护多条模型配置。
2. 任意时刻只能有一个启用模型。
3. 启用模型时先清除其他配置的启用状态，再将目标配置设为 active。
4. 启用后会同步 `api_key`、`base_url`、`model`、`reasoning_effort` 到 Executor。
5. 连接测试和模型列表拉取由后端代理调用 Executor。
6. 删除当前启用模型会被拒绝。

## 7. 更新历史文档时的对齐原则

1. 入口点以本文件第 1 节为准。
2. 后端接口以 `internal/api/router.go` 为准。
3. 前端页面和权限以 `src/router/index.ts` 为准。
4. 前端依赖版本以 `TestFront/package.json` 为准。
5. Executor endpoint 以 `executor/main.py` 中 `@app.*` 路由为准。
6. 业务状态流转以 `internal/service/` 的实现为准。
7. 如果历史文档出现“计划中、待创建、待接入”但当前代码已实现，应改为当前实现描述。
8. 如果历史文档描述了未落地能力，应标注为历史设计或移出当前实现文档。
