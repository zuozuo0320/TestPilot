## 2. 角色范围定义

本模块涉及以下角色：
- `admin`
- `manager`
- `tester`
- `reviewer`
- `developer`
- `readonly`

角色理解：
- `admin`：平台级最高管理角色
- `manager`：项目级管理角色
- `tester`：测试执行与脚本维护角色
- `reviewer`：脚本审核确认角色
- `developer`：默认只读查看角色
- `readonly`：纯查看角色

## 4. 已明确的权限口径

### 4.1 查看范围

已明确：
- `admin` 可查看所有任务
- 其他角色仅查看本项目范围内任务
- `readonly` 仅可查看，不可执行任何写操作

### 4.2 创建与执行

已明确：
- `admin / manager / tester` 可创建生成任务
- `admin / manager / tester` 可执行生成任务
- `reviewer / developer / readonly` 不可创建和执行生成任务

### 4.3 验证与编辑

已明确：
- `admin / manager / tester` 可手动触发回放验证
- `admin / manager / tester` 可对脚本草稿进行有限编辑
- `reviewer / developer / readonly` 不可执行上述操作

### 4.4 确认

已明确：
- `admin / manager / reviewer` 可执行最终确认
- `tester` 仅可提交待确认，不可执行最终确认
- `developer / readonly` 不可执行最终确认

### 4.5 废弃

已明确：
- `admin / manager` 可废弃任务与脚本版本
- `tester / reviewer / developer / readonly` 不可废弃

## 5. 导出权限定稿

已确认：
- `admin / manager / tester / reviewer` 可导出脚本
- `developer / readonly` 不允许导出脚本
- `developer` 保持严格只读权限，不提供导出能力

## 6. 前端实现口径

前端建议按以下原则实现：
- 按“角色 + 项目范围 + 当前状态”综合决定按钮显隐
- 任务详情接口返回按钮权限标识：
  - `can_execute`
  - `can_validate`
  - `can_edit`
  - `can_confirm`
  - `can_export`
  - `can_discard`
- 前端不可仅凭角色自行硬编码全部权限，应以后端返回的可操作标识为准

## 7. 后端实现口径

后端必须做到：
- 所有写操作接口均进行权限校验
- 前端即使隐藏了按钮，后端仍必须拦截未授权请求
- 项目范围权限必须校验，不能只校验角色
- 对 `confirm`、`discard`、`validate` 等敏感操作返回明确错误码或 403

建议重点校验接口：
- `POST /api/ai-script/tasks`
- `POST /api/ai-script/tasks/{taskId}/execute`
- `POST /api/ai-script/scripts/{scriptId}/edit`
- `POST /api/ai-script/scripts/{scriptId}/validate`
- `POST /api/ai-script/scripts/{scriptId}/confirm`
- `POST /api/ai-script/scripts/{scriptId}/discard`
- `GET /api/ai-script/scripts/{scriptId}/export`

## 8. 测试验收口径

测试至少应校验：
- `tester` 无法确认脚本
- `reviewer` 无法创建和执行生成任务
- `readonly` 无法创建、编辑、验证、确认、废弃、导出
- `manager` 只能操作本项目范围数据
- `admin` 可查看所有任务并执行全部管理动作
- `developer` 不允许导出脚本，前后端均按严格只读权限校验
