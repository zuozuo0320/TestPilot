# 测试管理平台-项目列表质量状态与测试进度真实化改造方案

> 版本：v0.2（更新于 2026-04-06）
> 项目：Aisight（后端 TestPilot / 前端 TestFront）
> 适用对象：产品、前端、后端、测试、联调
> 关联文档：
> - `测试管理平台-需求文档-20260308.md`
> - `测试管理平台-后端开发规范-20260405.md`
> - `TestFront/docs/前端架构开发规范-20260405.md`
> 文档定位：将项目列表“质量状态 / 测试进度”从前端 mock 展示升级为后端真实计算口径，并明确接口、数据来源与前端展示规则

---

## 0. 本轮结论

本轮确认项目列表中的“质量状态”和“测试进度”按以下口径收敛：

1. 项目列表页不允许继续使用前端 mock 数组渲染质量状态和测试进度。
2. `/projects` 列表接口需直接返回项目质量状态与测试进度摘要字段。
3. 测试进度建议按“已执行用例数 / 用例总数”计算，而不是按通过率展示。
4. 质量状态建议基于项目最新执行结果与通过率计算，保证同一项目在排序、筛选、分页后状态稳定。
5. 前端项目列表页只负责展示，不再自行推导或补位质量状态。

一句话定义：

- 测试进度：项目执行覆盖进度；
- 质量状态：项目当前执行质量状态。

---

## 1. 当前现状与问题

当前项目列表页存在以下问题：

1. “质量状态”来自前端固定数组，显示值为 `Stable / Degraded / Failing` 的轮播占位数据。
2. “测试进度”来自前端固定百分比数组，并非真实项目执行进度。
3. 页面按列表行号 `idx` 取状态和百分比，导致同一项目在排序、筛选、翻页后，质量状态和进度会发生变化。
4. 后端单项目概览接口虽然已有 `case_stats`、`latest_run`、`quality_gate`，但项目列表页未接入该真实能力。

因此，当前项目列表中的这两列本质上仍是“视觉占位”，不具备业务可信度。

---

## 2. 改造目标

本次改造目标如下：

1. 项目列表中的质量状态与测试进度全部真实化。
2. 消除前端按行号生成状态的随机性。
3. 将项目概览的执行质量能力下沉到项目列表接口，避免前端逐项目调用详情接口形成 N+1 请求。
4. 为后续项目筛选、排序、告警、看板复用提供统一数据口径。

---

## 3. 核心业务口径

## 3.1 测试进度

测试进度建议定义为：

`测试进度 = 已执行用例数 / 用例总数 × 100%`

字段说明：

- 用例总数：当前项目下全部用例数量
- 已执行用例数：`exec_result != 未执行` 的用例数量

边界规则：

1. 若项目下没有用例，则测试进度为 `0`
2. 进度保留整数或 1 位小数，由前端统一格式化展示
3. 进度用于表达“覆盖到什么程度”，不表达“执行质量好坏”

---

## 3.2 质量状态

质量状态建议优先基于项目最新一次执行质量计算；若暂无 run 数据，则回退基于用例主表当前执行结果计算。

推荐状态枚举：

- `stable`
- `degraded`
- `failing`
- `unknown`

推荐判定规则：

1. `unknown`
   - 项目没有用例，或
   - 项目没有任何执行数据，且用例主表也没有已执行结果
2. `stable`
   - 存在最新执行数据，且最新执行通过率 `>= 95%`，或
   - 无最新执行数据，但用例主表执行通过率 `>= 95%`
3. `degraded`
   - 存在最新执行数据，且最新执行通过率 `>= 80%` 且 `< 95%`，或
   - 无最新执行数据，但用例主表执行通过率 `>= 80%` 且 `< 95%`
4. `failing`
   - 存在最新执行数据，且最新执行通过率 `< 80%`，或
   - 无最新执行数据，但用例主表执行通过率 `< 80%`

说明：

1. 测试进度和质量状态是两个不同维度：
   - 进度回答“测了多少”
   - 质量回答“测得怎么样”
2. 若产品希望沿用现有三色视觉，也可在前端临时将 `unknown` 映射为灰色或弱提示态。
3. 建议采用“最新 run 优先，用例执行结果兜底”的策略，避免项目列表在历史已有执行结论但尚未沉淀 run 记录时长期显示 `unknown`。

---

## 3.3 与现有概览能力的关系

当前后端 `OverviewService` 已具备以下真实数据：

- `case_stats.total`
- `case_stats.executed`
- `case_stats.pass_rate`
- `latest_run.pass_rate`
- `quality_gate.status`

本次建议不是让前端循环调用 `/projects/{id}/overview`，而是：

1. 复用其中的计算口径
2. 将结果聚合后下沉到 `/projects` 列表接口
3. 由后端在项目列表查询中一次性返回摘要字段

---

## 4. 接口改造建议

## 4.1 `/projects` 返回字段

建议为项目列表每个项目增加以下字段：

- `quality_status`
- `quality_reason`
- `test_progress`
- `testcase_total_count`
- `executed_testcase_count`
- `latest_run_pass_rate`

返回示例：

```json
{
  "id": 1,
  "name": "AiSight Demo",
  "owner_id": 12,
  "owner_name": "李莎莎",
  "quality_status": "degraded",
  "quality_reason": "latest_run_pass_rate_between_80_95",
  "test_progress": 45,
  "testcase_total_count": 20,
  "executed_testcase_count": 9,
  "latest_run_pass_rate": 88.9
}
```

---

## 4.2 字段定义建议

### `quality_status`

项目当前质量状态，枚举建议：

- `stable`
- `degraded`
- `failing`
- `unknown`

### `quality_reason`

状态原因，建议保留为后端可读枚举，便于联调和后续告警：

- `no_testcases`
- `no_execution_data`
- `latest_run_pass_rate_ge_95`
- `latest_run_pass_rate_between_80_95`
- `latest_run_pass_rate_lt_80`
- `case_exec_pass_rate_ge_95`
- `case_exec_pass_rate_between_80_95`
- `case_exec_pass_rate_lt_80`

### `test_progress`

项目测试进度，建议取值范围：

- `0 ~ 100`

### `testcase_total_count`

项目用例总数。

### `executed_testcase_count`

项目已执行用例数。

### `latest_run_pass_rate`

项目最近一次执行的通过率，无执行数据时可返回 `null` 或 `0`，建议与 `quality_reason` 搭配解释。

---

## 5. 后端实现建议

## 5.1 避免 N+1

不建议前端项目列表页逐条请求 `/projects/{projectID}/overview`，原因如下：

1. 项目列表页会形成明显 N+1 请求
2. 列表翻页或搜索时会重复请求大量项目详情
3. 不利于排序和筛选扩展

因此建议后端直接扩展 `/projects` 聚合查询。

---

## 5.2 数据来源建议

推荐聚合来源如下：

1. 用例总数：`test_cases`
2. 已执行用例数：`test_cases.exec_result != '未执行'`
3. 最新执行通过率：`run_results / runs` 聚合出的最新 run 摘要
4. 质量状态：由 service 层统一根据聚合结果判定

---

## 5.3 分层职责建议

### Repository 层

负责：

- 聚合项目列表所需的统计字段
- 返回原始数值，不直接承载复杂业务判断

### Service 层

负责：

- 根据聚合结果计算 `quality_status`
- 计算 `quality_reason`
- 计算 `test_progress`
- 向 handler 返回最终列表结果

### Handler 层

负责：

- 保持 `/projects` 响应结构稳定
- 不新增业务判断

---

## 6. 前端展示建议

## 6.1 质量状态展示

建议前端按 `quality_status` 映射样式：

- `stable` → 绿色徽标
- `degraded` → 黄色徽标
- `failing` → 红色徽标
- `unknown` → 灰色徽标

展示文案建议：

- `stable` → `Stable`
- `degraded` → `Degraded`
- `failing` → `Failing`
- `unknown` → `Unknown`

## 6.2 测试进度展示

建议前端直接展示：

- 百分比数值：`test_progress`
- 进度条宽度：`test_progress%`

若需要增强可读性，可在 hover 或 tooltip 中补充：

- 已执行：`executed_testcase_count`
- 总数：`testcase_total_count`

---

## 7. 验收标准

满足以下条件可视为本需求落地完成：

1. 项目列表页不再出现 mock 质量状态和 mock 进度数组。
2. 同一项目在筛选、排序、分页后，质量状态与测试进度保持一致。
3. `/projects` 接口直接返回真实质量状态与测试进度字段。
4. 新建无执行数据项目时，列表能正确展示 `unknown` 或产品确认的空态映射。
5. 项目列表页无需逐条请求详情接口即可渲染状态。

---

## 8. 实施顺序建议

建议按以下顺序推进：

1. 确认项目列表质量状态与测试进度业务口径
2. 扩展 `/projects` 返回字段
3. 在后端 service 中下沉计算逻辑
4. 前端移除 mock 数组并改为真实字段渲染
5. 增补项目列表相关 API 测试与前端联调验证

---

## 9. 当前结论

自本方案生效起，项目列表“质量状态 / 测试进度”改造应遵循以下基线：

1. 质量状态和测试进度必须是真实业务数据
2. `/projects` 是项目列表展示的唯一接口入口
3. 前端不得再按 `idx` 或固定数组生成状态
4. 测试进度与质量状态必须分开定义，避免语义混淆
