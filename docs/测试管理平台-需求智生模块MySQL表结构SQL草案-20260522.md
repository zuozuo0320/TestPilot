# 需求智生模块 MySQL 表结构 SQL 草案

> 版本：v1.0 · 日期：2026-05-22 · 状态：可开发版
>
> 配套文档：
> - [需求智生模块需求方案 v1.1](./测试管理平台-需求智生模块需求方案-20260522.md)
> - [需求智生模块技术架构与集成方案](./测试管理平台-需求智生模块技术架构与集成方案-20260522.md)
>
> 适用：MySQL 8.4 / utf8mb4 / utf8mb4_unicode_ci
>
> 并发设计目标：单系统 10 并发生成任务，多人同时操作不冲突
> - **乐观锁**：所有可被并发修改的实体加 `lock_version` 字段
> - **CAS 状态机**：所有状态变更走 `UPDATE ... WHERE status=?`
> - **唯一约束**：防双写采纳、防 Skill key 冲突
> - **行锁友好的索引**：所有 `FOR UPDATE` 查询路径都有专用索引

## 二、新增表 DDL

### 2.1 `requirement_docs` 需求文档表

```sql
CREATE TABLE IF NOT EXISTS `requirement_docs` (
  `id`                  BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `project_id`          BIGINT UNSIGNED NOT NULL                COMMENT '所属项目',
  `title`               VARCHAR(200)    NOT NULL                COMMENT '文档标题',
  `source_type`         VARCHAR(20)     NOT NULL                COMMENT 'upload_file / paste_text',
  `file_format`         VARCHAR(20)     NOT NULL                COMMENT 'docx / pdf / md / txt / text',
  `file_path`           VARCHAR(500)    NOT NULL DEFAULT ''     COMMENT '原始文件相对路径（粘贴文本为空）',
  `file_size`           BIGINT          NOT NULL DEFAULT 0      COMMENT '文件大小（字节）',
  `raw_content`         LONGTEXT        NULL                    COMMENT '解析后纯文本（含粘贴文本原文）',
  `word_count`          INT             NOT NULL DEFAULT 0      COMMENT '解析后字数（截断后）',
  `original_word_count` INT             NOT NULL DEFAULT 0      COMMENT '截断前原始字数',
  `truncated`           TINYINT(1)      NOT NULL DEFAULT 0      COMMENT '是否因超长被截断',
  `parse_status`        VARCHAR(20)     NOT NULL DEFAULT 'not_parsed'
                                                                COMMENT 'not_parsed/parsing/parsed/parse_failed',
  `parse_error`         VARCHAR(1000)   NOT NULL DEFAULT ''     COMMENT '解析失败原因',
  `parse_started_at`    DATETIME        NULL                    COMMENT '上次开始解析时间（用于 stuck 检测）',
  `remark`              VARCHAR(500)    NOT NULL DEFAULT ''     COMMENT '用户备注',
  `lock_version`        INT             NOT NULL DEFAULT 0      COMMENT '乐观锁版本号',
  `created_by`          BIGINT UNSIGNED NOT NULL                COMMENT '上传人',
  `created_at`          DATETIME        NOT NULL,
  `updated_at`          DATETIME        NOT NULL,
  `deleted_at`          DATETIME        NULL                    COMMENT '软删时间',
  PRIMARY KEY (`id`),
  KEY `idx_rd_project_created` (`project_id`, `created_at` DESC),
  KEY `idx_rd_project_status`  (`project_id`, `parse_status`),
  KEY `idx_rd_creator`         (`created_by`),
  KEY `idx_rd_parse_stuck`     (`parse_status`, `parse_started_at`) COMMENT '用于扫描 stuck=parsing 的文档',
  KEY `idx_rd_deleted`         (`deleted_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='需求文档表';
```

**索引说明**：
- `idx_rd_project_created`：列表查询主索引，按项目过滤 + 时间倒序
- `idx_rd_parse_stuck`：兜底 worker 扫描"卡在 parsing 状态超过 N 分钟"的文档

### 2.2 `requirement_gen_tasks` 生成任务表

```sql
CREATE TABLE IF NOT EXISTS `requirement_gen_tasks` (
  `id`                  BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `project_id`          BIGINT UNSIGNED NOT NULL                COMMENT '所属项目',
  `requirement_doc_id`  BIGINT UNSIGNED NOT NULL                COMMENT '源需求文档',
  `skill_id`            BIGINT UNSIGNED NOT NULL                COMMENT '使用的 Skill',
  `ai_model_config_id`  BIGINT UNSIGNED NOT NULL                COMMENT '任务发起时刻激活的模型 ID（快照）',
  `ai_model_snapshot`   VARCHAR(200)    NOT NULL DEFAULT ''     COMMENT 'provider/model_id 文本快照（审计）',
  `task_name`           VARCHAR(200)    NOT NULL                COMMENT '任务名',
  `target_module_id`    BIGINT UNSIGNED NOT NULL DEFAULT 0      COMMENT '预设落库模块（0 表示未分类）',
  `default_level`       VARCHAR(10)     NOT NULL DEFAULT 'P2'   COMMENT '默认用例级别',
  `max_cases`           INT             NOT NULL DEFAULT 30     COMMENT '产物上限',
  `extra_prompt`        TEXT            NULL                    COMMENT '用户补充上下文',
  `parent_task_id`      BIGINT UNSIGNED NOT NULL DEFAULT 0      COMMENT 'rerun 来源任务 ID，0 表示原生任务',

  `status`              VARCHAR(20)     NOT NULL DEFAULT 'PENDING'
                                                                COMMENT 'PENDING/RUNNING/SUCCESS/FAILED/PARTIAL_ADOPTED/FULLY_CLOSED',
  `fail_reason`         VARCHAR(1000)   NOT NULL DEFAULT ''     COMMENT '失败原因（带 [类别] 前缀）',

  `generated_count`     INT             NOT NULL DEFAULT 0      COMMENT '已生成产物数',
  `adopted_count`       INT             NOT NULL DEFAULT 0      COMMENT '已采纳数',
  `discarded_count`     INT             NOT NULL DEFAULT 0      COMMENT '已丢弃数',

  `prompt_tokens`       INT             NOT NULL DEFAULT 0      COMMENT 'LLM prompt tokens 用量',
  `completion_tokens`   INT             NOT NULL DEFAULT 0      COMMENT 'LLM completion tokens 用量',
  `duration_ms`         BIGINT          NOT NULL DEFAULT 0      COMMENT '执行总耗时（毫秒）',

  `started_at`          DATETIME        NULL                    COMMENT '进入 RUNNING 时间',
  `finished_at`         DATETIME        NULL                    COMMENT '终态时间（SUCCESS/FAILED）',
  `last_heartbeat_at`   DATETIME        NULL                    COMMENT 'executor 最近心跳，用于超时回收',

  `executor_node_id`    VARCHAR(64)     NOT NULL DEFAULT ''     COMMENT 'executor 节点标识，便于多 executor 部署调试',
  `request_id`          VARCHAR(64)     NOT NULL DEFAULT ''     COMMENT '创建任务时的请求 ID，链路追踪',

  `lock_version`        INT             NOT NULL DEFAULT 0      COMMENT '乐观锁版本号',
  `created_by`          BIGINT UNSIGNED NOT NULL,
  `created_at`          DATETIME        NOT NULL,
  `updated_at`          DATETIME        NOT NULL,

  PRIMARY KEY (`id`),
  KEY `idx_rgt_project_status`     (`project_id`, `status`)             COMMENT '项目级配额计数 / 列表',
  KEY `idx_rgt_project_created`    (`project_id`, `created_at` DESC)    COMMENT '列表查询',
  KEY `idx_rgt_status_heartbeat`   (`status`, `last_heartbeat_at`)      COMMENT '超时回收 worker 扫描',
  KEY `idx_rgt_doc`                (`requirement_doc_id`)               COMMENT '文档维度查询',
  KEY `idx_rgt_skill`              (`skill_id`)                          COMMENT 'Skill 维度统计',
  KEY `idx_rgt_creator`            (`created_by`)                        COMMENT '我的任务',
  KEY `idx_rgt_parent`             (`parent_task_id`)                    COMMENT 'rerun 链路'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='需求智生-生成任务表';
```

**关键索引选择说明**：
- `idx_rgt_project_status`：配额降级到 DB 时走这条索引：`SELECT COUNT(*) WHERE project_id=? AND status='RUNNING'`
- `idx_rgt_status_heartbeat`：超时回收 worker 扫描，`WHERE status='RUNNING' AND last_heartbeat_at < NOW() - 5min`
- `idx_rgt_project_created`：列表查询主索引

**为什么不在 status 上单独建索引**：单字段 status 选择性极低（只有 6 个值），组合索引更高效。

### 2.3 `requirement_gen_results` AI 产物表

```sql
CREATE TABLE IF NOT EXISTS `requirement_gen_results` (
  `id`               BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `task_id`          BIGINT UNSIGNED NOT NULL                  COMMENT '所属任务',
  `project_id`       BIGINT UNSIGNED NOT NULL                  COMMENT '冗余项目 ID，便于权限校验',
  `seq_no`           INT             NOT NULL                  COMMENT '任务内序号（1-based）',

  `title`            VARCHAR(200)    NOT NULL                  COMMENT '用例标题',
  `level`            VARCHAR(10)     NOT NULL DEFAULT 'P2'     COMMENT '用例级别',
  `precondition`     TEXT            NULL                      COMMENT '前置条件',
  `steps`            LONGTEXT        NULL                      COMMENT '步骤（JSON 或 Markdown）',
  `postcondition`    TEXT            NULL                      COMMENT '后置条件',
  `remark`           TEXT            NULL                      COMMENT '备注',
  `tags_suggested`   VARCHAR(500)    NOT NULL DEFAULT ''       COMMENT 'AI 建议标签，逗号分隔',
  `ai_confidence`    DECIMAL(3,2)    NULL                      COMMENT 'LLM 自评置信度 0.00-1.00',
  `raw_json`         LONGTEXT        NULL                      COMMENT 'LLM 原始输出片段（审计）',

  `adopted`          TINYINT(1)      NOT NULL DEFAULT 0        COMMENT '是否已入库',
  `adopted_case_id`  BIGINT UNSIGNED NOT NULL DEFAULT 0        COMMENT '入库后的用例 ID',
  `adopted_at`       DATETIME        NULL,
  `adopted_by`       BIGINT UNSIGNED NOT NULL DEFAULT 0,

  `discarded`        TINYINT(1)      NOT NULL DEFAULT 0        COMMENT '是否已丢弃',
  `discarded_at`     DATETIME        NULL,
  `discarded_by`     BIGINT UNSIGNED NOT NULL DEFAULT 0,

  `lock_version`     INT             NOT NULL DEFAULT 0        COMMENT '乐观锁版本号（产物编辑）',
  `created_at`       DATETIME        NOT NULL,

  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_task_seq`       (`task_id`, `seq_no`)         COMMENT '任务内序号唯一',
  KEY `idx_rgr_task_adopted`     (`task_id`, `adopted`)         COMMENT '按采纳状态过滤',
  KEY `idx_rgr_task_discarded`   (`task_id`, `discarded`)       COMMENT '按丢弃状态过滤',
  KEY `idx_rgr_adopted_case`     (`adopted_case_id`)            COMMENT '用例反查产物'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='需求智生-AI 产物表（预览池）';
```

**关键设计**：
- 采纳/丢弃分两个 TINYINT 字段而不是合并 status：未来允许"丢弃后又重新可见"等扩展，且过滤索引更高效
- `uk_task_seq` 防止 LLM 输出重复 seq_no 时写入冲突
- 没有 `updated_at` —— 产物本身不允许变更，只允许采纳/丢弃；编辑通过 `lock_version` CAS 保护（场景 E）

### 2.4 `ai_skills` Skill 模板表

```sql
CREATE TABLE IF NOT EXISTS `ai_skills` (
  `id`              BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `project_id`      BIGINT UNSIGNED NOT NULL DEFAULT 0          COMMENT '0=系统内置，>0=项目副本',
  `skill_key`       VARCHAR(50)     NOT NULL                    COMMENT '稳定标识（项目内唯一）',
  `name`            VARCHAR(100)    NOT NULL                    COMMENT '显示名',
  `scope`           VARCHAR(20)     NOT NULL                    COMMENT 'functional/api/compat/security/custom',
  `description`     VARCHAR(500)    NOT NULL DEFAULT ''         COMMENT '适用场景描述',
  `prompt_template` LONGTEXT        NOT NULL                    COMMENT '提示词模板（含 {{requirement_text}} {{project_context}} {{few_shot_examples}} 占位符）',
  `output_schema`   VARCHAR(50)     NOT NULL DEFAULT 'standard_case_v1'
                                                                COMMENT 'JSON Schema 名（预设）',
  `is_system`       TINYINT(1)      NOT NULL DEFAULT 0          COMMENT '系统内置不可删',
  `is_active`       TINYINT(1)      NOT NULL DEFAULT 1          COMMENT '是否可用',
  `sort_order`      INT             NOT NULL DEFAULT 0,

  `lock_version`    INT             NOT NULL DEFAULT 0          COMMENT '乐观锁版本号',
  `created_by`      BIGINT UNSIGNED NOT NULL DEFAULT 0,
  `created_at`      DATETIME        NOT NULL,
  `updated_at`      DATETIME        NOT NULL,
  `deleted_at`      DATETIME        NULL                        COMMENT '软删，ID 保留不复用',

  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_skill_project_key` (`project_id`, `skill_key`) COMMENT '同项目内 key 唯一；系统内置 project_id=0',
  KEY `idx_skill_active`            (`is_active`),
  KEY `idx_skill_deleted`           (`deleted_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='需求智生-Skill 模板表';
```

**关键设计**：
- `uk_skill_project_key` 同时覆盖系统内置（`project_id=0`）和项目副本，**项目副本必须用与系统内置相同的 `skill_key` 才能被前端识别为"覆写"**，Service 层据此查找
- 软删 + 唯一索引的兼容：MySQL 唯一索引允许多个 NULL，但 `deleted_at` 为 DATETIME，软删时设具体时间，DB 层不会冲突 —— 不过为避免歧义，Service 层在创建时显式校验是否已有同 key 活跃记录

## 三、`testcases` 表增量字段

```sql
ALTER TABLE `testcases`
  ADD COLUMN `source_ai_generated`        TINYINT(1)      NOT NULL DEFAULT 0
                                          COMMENT '是否 AI 生成',
  ADD COLUMN `source_requirement_doc_id`  BIGINT UNSIGNED NOT NULL DEFAULT 0
                                          COMMENT '来源需求文档 ID',
  ADD COLUMN `source_gen_task_id`         BIGINT UNSIGNED NOT NULL DEFAULT 0
                                          COMMENT '来源生成任务 ID',
  ADD COLUMN `source_skill_id`            BIGINT UNSIGNED NOT NULL DEFAULT 0
                                          COMMENT '使用的 Skill ID',
  ADD KEY `idx_tc_source_doc`  (`source_requirement_doc_id`)
                               COMMENT '需求文档反查用例',
  ADD KEY `idx_tc_source_task` (`source_gen_task_id`)
                               COMMENT '生成任务反查用例';
```

**为什么用 DEFAULT 0 而不是 NULL**：
- 兼容性：现有 testcases 数据自动填 0，不需要 backfill
- 索引性能：MySQL 对 NULL 的索引效率不如非空整数
- 业务含义清晰：0 = 不是 AI 来源；> 0 = 对应外键

## 五、与现有 `ai_model_configs` 的关系

不修改既有表。任务表 `ai_model_config_id` 仅为软引用（不加外键约束），删除模型时通过应用层校验：

```sql
-- 创建任务时校验激活模型是否存在（应用层 SQL）
SELECT id, provider, model_id, is_active
  FROM ai_model_configs
 WHERE is_active = 1
   AND deleted_at IS NULL
 LIMIT 1;
```

无激活模型时返回错误码 130105。

## 六、关键并发 SQL 模式

本节列出 Service 层在并发场景下必须遵循的 SQL 模式。

### 6.1 CAS 状态推进

```sql
-- PENDING → RUNNING
UPDATE requirement_gen_tasks
   SET status = 'RUNNING',
       started_at = NOW(),
       last_heartbeat_at = NOW(),
       updated_at = NOW(),
       lock_version = lock_version + 1
 WHERE id = ?
   AND status = 'PENDING';
-- affected_rows = 0 视为已被其他节点启动

-- not_parsed/parse_failed → parsing
UPDATE requirement_docs
   SET parse_status = 'parsing',
       parse_started_at = NOW(),
       updated_at = NOW(),
       lock_version = lock_version + 1
 WHERE id = ?
   AND parse_status IN ('not_parsed', 'parse_failed');
```

### 6.2 采纳防双写

```sql
START TRANSACTION;

  SELECT id, adopted, adopted_case_id
    FROM requirement_gen_results
   WHERE id = ?
   FOR UPDATE;
  -- 应用层判断 adopted=1 → 回滚返回 130301

  INSERT INTO testcases
    (project_id, title, status, version, level, precondition, steps, postcondition,
     remark, priority, module_id, module_path,
     source_ai_generated, source_requirement_doc_id, source_gen_task_id, source_skill_id,
     created_by, updated_by, created_at, updated_at)
  VALUES
    (?, ?, 'draft', 'V1', ?, ?, ?, ?, ?, ?, ?, ?,
     1, ?, ?, ?, ?, ?, NOW(), NOW());
  SET @case_id = LAST_INSERT_ID();

  UPDATE requirement_gen_results
     SET adopted = 1,
         adopted_case_id = @case_id,
         adopted_at = NOW(),
         adopted_by = ?,
         lock_version = lock_version + 1
   WHERE id = ?
     AND adopted = 0
     AND discarded = 0;
  -- affected_rows = 0 → 同样回滚（极端情况下被并发采纳/丢弃）

COMMIT;
```

### 6.3 任务终态推进（带计数同步）

```sql
-- 全部产物被处理完后（adopted + discarded = generated_count）
UPDATE requirement_gen_tasks
   SET status = CASE
                  WHEN adopted_count > 0 THEN 'PARTIAL_ADOPTED'
                  ELSE 'FULLY_CLOSED'
                END,
       updated_at = NOW(),
       lock_version = lock_version + 1
 WHERE id = ?
   AND status = 'SUCCESS'
   AND (adopted_count + discarded_count) >= generated_count;
```

### 6.4 配额回退（Redis 不可用时的 DB 降级）

```sql
-- 项目级配额预检（事务内）
SELECT COUNT(*)
  FROM requirement_gen_tasks
 WHERE project_id = ?
   AND status = 'RUNNING'
 FOR UPDATE;
-- 应用层判断 < REQGEN_PROJECT_QUOTA 才允许创建
-- 然后在同一事务内 INSERT 新任务
```

**注意**：DB 配额降级会触发 InnoDB 间隙锁，可能造成短期阻塞，仅作为 Redis 故障时的降级路径。

### 6.5 超时回收扫描

```sql
-- worker 每 30s 扫描一次
SELECT id, project_id, ai_model_config_id
  FROM requirement_gen_tasks
 WHERE status = 'RUNNING'
   AND last_heartbeat_at < NOW() - INTERVAL 5 MINUTE
 ORDER BY last_heartbeat_at ASC
 LIMIT 50;
-- 走 idx_rgt_status_heartbeat
```

逐个标记失败：

```sql
UPDATE requirement_gen_tasks
   SET status = 'FAILED',
       fail_reason = '[TIMEOUT] executor heartbeat lost',
       finished_at = NOW(),
       updated_at = NOW(),
       lock_version = lock_version + 1
 WHERE id = ?
   AND status = 'RUNNING';
```

### 6.6 文档解析 stuck 兜底扫描

```sql
SELECT id
  FROM requirement_docs
 WHERE parse_status = 'parsing'
   AND parse_started_at < NOW() - INTERVAL 10 MINUTE
 LIMIT 50;
-- 走 idx_rd_parse_stuck
```

逐个 CAS 转 parse_failed：

```sql
UPDATE requirement_docs
   SET parse_status = 'parse_failed',
       parse_error = '[STUCK] parsing timeout',
       updated_at = NOW()
 WHERE id = ?
   AND parse_status = 'parsing';
```

### 6.7 Skill 乐观锁更新

```sql
UPDATE ai_skills
   SET name = ?,
       description = ?,
       prompt_template = ?,
       output_schema = ?,
       updated_at = NOW(),
       lock_version = lock_version + 1
 WHERE id = ?
   AND lock_version = ?
   AND deleted_at IS NULL;
-- affected_rows = 0 → 返回 130204 "已被他人修改，请刷新重试"
```

## 七、索引策略复核

| 表 | 索引 | 覆盖的高频查询 |
|---|---|---|

## 八、估算与容量规划

### 8.1 单表行数估算（按 1 年使用）

| 表 | 单项目月增 | 全平台年增（10 项目） | 大小级别 |
|---|---|---|---|
| requirement_docs | 20 | 2 400 行 | 小 |
| requirement_gen_tasks | 50 | 6 000 行 | 小 |
| requirement_gen_results | 50 × 25 = 1 250 | 150 000 行 | 中 |
| ai_skills | 5 | 600 行 | 极小 |

主要存储压力来自 `requirement_gen_results.raw_json`（每行 ~5KB）+ `requirement_docs.raw_content`（每文档 ~50KB）。一年 150 000 × 5KB + 2 400 × 50KB ≈ 870MB，可控。

### 8.2 大字段分离（可选演进）

如果未来 raw_content / raw_json 体积变大，可将 LONGTEXT 字段独立到附属表（如 `requirement_docs_content`），主表只留元数据。**v1.0 不做**。

## 十、与 GORM AutoMigrate 的协同

后端代码中需要在 `model.AutoMigrate` 添加 4 个新模型：

```go
return db.AutoMigrate(
    // ... 既有模型
    &RequirementDoc{},
    &RequirementGenTask{},
    &RequirementGenResult{},
    &AISkill{},
)
```

**但 AutoMigrate 不会处理**：
- 复合唯一索引（如 `uk_task_seq`）
- 软删唯一索引行为
- DEFAULT 表达式（如 `lock_version DEFAULT 0`）

因此**必须**在 `internal/migration/sql/20260522_create_requirement_gen.sql` 中显式声明，由 `migrate-sql` 子命令执行（见现有 `internal/migration` 模式）。
